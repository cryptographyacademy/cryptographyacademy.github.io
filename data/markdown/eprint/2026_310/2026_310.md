# Bolt: Faster SNARKs from Sketched Codes

Kobi Gurkan<sup>∗</sup> Andrija Novakovic† Ron D. Rothblum‡ February 18, 2026

#### Abstract

We introduce Bolt, a new Multilinear Polynomial Commitment Scheme (MLPCS) designed for high-performance SNARKs over binary fields. Bolt is geared towards SNARKs for large computations, in which prover speed is paramount but one can afford slightly larger proofs. The construction is based on the code-switching paradigm; our core technical contribution is a new "proof-system friendly" error-correcting code with extremely efficient encoding both asymptotically and concretely.

Theoretically, Bolt achieves a commitment time of approximately (3 + ε) · N field additions plus a Merkle Tree hash computation of size (1 +ε)· N field elements, where N is the size of the multilinear polynomial and ε > 0 is arbitrarily small. The prior state-of-the-art, Blaze (Brehm et al., Eurocrypt 2025) used 8N additions and a 4N size Merkle hash.

Concretely, our implementation demonstrates that these asymptotic gains translate into substantial real-world speedups. Our benchmarks show that for N = 2<sup>30</sup> over a 32-bit field, Bolt achieves a commitment time roughly 3× faster than Reed-Solomon based schemes, albeit with a moderately larger proof. Bolt also offers better commitment time and proof size than recent linear-time schemes. For example, its commitment time is about 1.34× faster than Brakedown (Golovnev et al., Crypto 2023) and with a 2× shorter proof.

## <span id="page-2-4"></span><span id="page-2-0"></span>1 Introduction

Succinct Non-Interactive Arguments of Knowledge (SNARKs) have evolved from deep within the theory of cryptography and complexity-theory into critical infrastructure for verifiable computation and blockchain scalability. As these systems are deployed to prove increasingly complex statements, the computational cost of the prover has emerged as a primary bottleneck.[1](#page-2-2)

Polynomial commitment schemes (PCS) are a dominant cost driver for efficient SNARKs. A PCS allows the prover to provide a succinct commitment to a large polynomial P (which, in a SNARK, typically encodes a computation trace) so that later it can prove statements of the form "P(x) = y". We focus on multilinear commitments in which P is a multilinear polynomial (i.e., a multivariate polynomial with degree at most 1 in each variable).

Existing multilinear polynomial commitments schemes (MLPCSs) generally fall into two categories: those based purely on hash functions (usually analyzed in the random oracle model), and those based on more structured cryptography such as hardness of discrete log or of lattice problems.

In this work we restrict our attention to hash-based scheme. Within this family there are two main approaches: those based on Reed-Solomon (RS) codes (e.g., [\[AHIV23,](#page-24-0) [BBHR18,](#page-25-0)[ZCF24,](#page-28-0) [ACFY24,](#page-24-1) [ACFY25,](#page-24-2) [NA25\]](#page-27-0)) and those based on linear-time encodable codes [\[BCG](#page-25-1)+17, [BCG20,](#page-25-2) [GLS](#page-27-1)+23[,RR25,](#page-28-1)[BCF](#page-25-3)+25[,NST25,](#page-27-2)[BMMS25a,](#page-26-0)[ARR25,](#page-24-3)[BMMS25b,](#page-26-1)[BFRW25\]](#page-25-4). RS-based schemes offer more compact proofs but suffer from the concrete overhead of Fast Fourier Transforms (FFTs), which become the bottleneck for large instances. Conversely, the newest linear-time schemes [\[BMMS25b,](#page-26-1)[BFRW25\]](#page-25-4) offer essentially optimal asymptotics but tend to have concretely larger proofs.

In this work we ask: to what extreme can we take the linear-time encoding approach? As linear-time proving becomes commonplace, we are no longer content with asymptotic bounds alone. Rather, we aim to minimize the computational rate of provers, that is, the constant c required for a PCS or SNARK to execute in time c · N, for inputs of size N. To do so we are willing to have a moderately larger, but still reasonable, proof-size.

## <span id="page-2-1"></span>1.1 Our Contribution

We construct a new Multilinear Polynomial Commitment Scheme (MLPCS), called Bolt, designed specifically for high-performance SNARKs over binary fields.[2](#page-2-3) Bolt is geared towards SNARKs for large computations, in which prover speed is paramount but one can afford slightly larger proofs. We believe that Bolt is particularly attractive for client-side proving in which the FFTs tend to be a major bottleneck, and in which trading off proof-size for better prover latency is usually desirable (let alone if the proof-size can be further reduced via recursion).

Bolt follows the code-switching paradigm [\[RR24\]](#page-27-3), specifically in a similar way to how it was used in [\[BCF](#page-25-3)+25]. Our core technical contribution is the construction of a new error-correcting code the Sketched Code — that is extremely efficient both asymptotically and concretely. Beyond the extremely fast encoding, we show that the sketched code has several other proof-system friendly features on which we elaborate in [Section 1.1.1](#page-4-1) below.

<span id="page-2-3"></span><span id="page-2-2"></span><sup>1</sup>See, e.g., <https://zkevm.ethereum.foundation>.

<sup>2</sup>The use of binary fields for concretely efficient SNARKs was pioneered by Diamond and Posen [\[DP25,](#page-26-2)[DP24a\]](#page-26-3). Our construction can be extended to prime fields in a straightforward manner, but we choose to focus on binary fields for simplicity and since it allows our scheme to be fully dominated by additions (i.e., XORs).

{3}------------------------------------------------

<span id="page-3-3"></span>The Asymptotic Angle. Asymptotically, Bolt achieves a commitment time of (3 + ϵ) · N field additions plus an additional Merkle tree computation on (1 + ϵ) · N field elements, where N is the size of the multilinear polynomial and ϵ > 0 is arbitrarily small. This improves significantly over the prior state-of-the-art, Blaze [\[BCF](#page-25-3)+25], which required 8N additions.[3](#page-3-1)

We summarize our main asymptotic results in the following theorem:

<span id="page-3-0"></span>Theorem 1.1 (Informally Stated). For every ϵ > 0, there exists a multilinear polynomial commitment scheme in the random oracle model with the following parameters:

- Soundness Error: 2 −λ .
- Commitment Time: (3 + ε) · N field additions and (1 + ε) · N Merkle hash computation.
- Evaluation Proof: (2 + o(1)) · N field operations.
- Proof Size: O(λ · log(N) · log log(N)).
- Verifier Time: O(λ · log(N) · log log(N)).

The most notable aspect of [Theorem 1.1](#page-3-0) is the roughly 3N commitment-time, improving over the 8N achieved in [\[BCF](#page-25-3)+25]. The other parameters are similar to those achieved by state-of-theart IOPs [\[ACFY25,](#page-24-2)[BMMS25a,](#page-26-0)[BMMS25b,](#page-26-1)[BFRW25\]](#page-25-4) (actually, the proof-size and verification time are off, by an O(log log N) factor from [\[BMMS25b,](#page-26-1) [BFRW25\]](#page-25-4), but could be improved by utilizing similar techniques, which we avoid for sake of simplicity).

Concrete Performance. The big-O notation in the proof-size in [Theorem 1.1](#page-3-0) hides large constants and so in the concrete instantiation we choose different parameters, to allow for reasonably sized proofs.

Our implementation demonstrates that the asymptotic gains translate into substantial real-world speedups. Our benchmarks show that for N = 2<sup>30</sup> over a 32-bit field, Bolt achieves a commitment time roughly 3× faster than Ligerito [\[NA25\]](#page-27-0), a recent Reed-Solomon based scheme (which has comparable prover cost to that of WHIR [\[ACFY25\]](#page-24-2)). The proof-size is about 19MB, which is considerably larger than in those schemes and so we only recommend using Bolt in contexts where proof-size is not a major issue, or if it is plausible to use recursion to compress the proof.

Bolt is faster than previous linear-time schemes and also offers additional advantages over them. Bolt's commitment is about 1.34× faster than Brakedown's [\[GLS](#page-27-1)+23] commitment and 1.41× faster than that of Blaze, with a proof that is about 2× smaller than Brakedown's and with an evaluation proof that is about 5× faster than Blaze's evaluation proof.[4](#page-3-2)

We refer the reader to [Section 8](#page-23-0) for the specifics of our benchmark methodology and additional details.

<span id="page-3-1"></span><sup>3</sup>Blaze also uses a Merkle hash computation of size 4N, but this can be alleviated via code puncturing (see [\[BCF](#page-25-3)<sup>+</sup>25, Section 5.4]). In contrast, the amount of additions seems inherent.

<span id="page-3-2"></span><sup>4</sup>Bolt has both an amortized and non-amortized variant. The non-amortized variant utilizes the Mulperm protocol [\[BCD25\]](#page-25-5) for which an implementation is not yet available. ### <span id="page-4-2"></span><span id="page-4-1"></span>1.1.1 Sketched Codes

As alluded to above, our core technical contribution is the design of a new efficiently encodable error correcting code. The simple approach, inspired by classical results in coding theory [\[GDP73,](#page-26-4)[Spi96\]](#page-28-2), allows one to compose error-correcting codes that have different properties together to obtain a new code with appealing properties — first and foremost, extremely fast encoding.

From an asymptotic perspective, the code, which is defined over the binary alphabet, can be encoded by a circuit of size (3 + ε) · k for messages of size k. It has a small but positive constant relative distance δ > 0. More accurately, similarly to the codes that underlie [\[GLS](#page-27-1)+23, [BCF](#page-25-3)+25] the code is actually a collection of codes. The guarantee is that a random code from the family has distance δ with high probability. We emphasize that the code just needs to be sampled once and can then be used for eternity. The code is guaranteed to have "good" minimal distance, unless some extremely unlikely event happened during the setup phase.

<span id="page-4-0"></span>Theorem 1.2. For any desired rate parameter r > 0 and ε > 0, there exists δ = δ(r, ε) > 0 and a family of error-correcting codes {C : {0, 1} <sup>k</sup> → {0, 1} <sup>n</sup>} of rate r, that are computable by circuits of size (3 + ε) · k, such that a random code from the family has distance δ with all but O(1/n) probability.

Similarly to Blaze [\[BCF](#page-25-3)+25, Section 1.2.2] the error probability can be reduced to any inverse polynomial error by invoking a more complicated sampling procedure, which runs in time inversely proportional to the desired error. See [Remark 6.2](#page-15-2) for details.

Beyond the efficient encoding, the sketched code that we construct has key properties, that are extremely useful in the construction of efficient proofs:

- 1. We construct an efficient proximity test for membership in the sketched code. As we show later (in [Section 1.2\)](#page-5-0), via this proximity test drastically improves the efficiency of the evaluation proof of our PCS.
- 2. The code natively has a high rate, supporting rates arbitrary close to 1. Such high rate codes are useful for proof-systems as they can (1) drastically reduce the amount of hashing required (recall that in hash based proofs the prover typically needs to generate a Merkle hash of the codewords), and (2) reduce the memory footprint of encoding. While the Reed-Solomon code in principle supports rates arbitrary close to 1, for rates above 1/2 it becomes unfriendly to algorithms that rely on the underlying group structure (e.g., FFT and proximity tests such as FRI and WHIR). Also note that the RAA ensemble considered in [\[BCF](#page-25-3)+25] has a rate of 1/4.

It is worth pointing out that any code can be made to have a high rate via code puncturing, but this approach seems quite wasteful and may be incompatible with the accompanying proof-system.

3. Lastly, the sketched code is natively systematic. This means that the encoding of a message m always starts with the message m itself.

Having a systematic encoding is extremely compelling in a SNARK application. The prover in a SNARK typically needs to encode the computation trace and then perform a Merkle hash of the codeword. When using a systematic code, since Merkle hashing can be done in a streaming manner, the prover can start hashing in parallel to processing the computation trace.

{5}------------------------------------------------

<span id="page-5-4"></span>While it is possible to transform any code (in particular the RAA or Reed-Solomon code) to be systematic, this usually introduces significant overhead.

#### <span id="page-5-0"></span>1.2 Technical Overview

Our goal is to construct a multilinear polynomial commitment scheme (MLPCS) over a finite field  $\mathbb{F}$ . We do so by constructing a similar scheme in the interactive oracle proof model, aka an interactive oracle PCS (IOPCS) [DP24b]. The IOPCS can be compiled into a cryptographic MLPCS and/or a SNARK by composing it with a vector commitment (e.g., via Merkle hashing, see [CY24] for details). An IOPCS consists of two protocols: a commitment protocol and an evaluation protocol. In the commitment protocol we are given as input a description of a vector  $x \in \mathbb{F}^k$ , which we associate with a function  $x:\{0,1\}^{\log k} \to \mathbb{F}$  in the natural way, and the output is a "commitment oracle".<sup>5</sup> In the evaluation phase, the prover and verifier are given as input  $z \in \mathbb{F}^{\log k}$  and  $v \in \mathbb{F}$  and the goal is to convince the verifier that  $\hat{x}(z) = v$ , where throughout this work  $\hat{x}$  denotes the multilinear extension of x (i.e., the unique multilinear polynomial that agrees with x on  $\{0,1\}^{\log k}$ , see Section 2.1). The evaluation phase may consists of both oracle messages and non-oracle (i.e., direct and explicit) messages.

Our construction of an IOPCS is based on the code-switching technique [RR24] as used by Brehm et al. [BCF<sup>+</sup>25]. We start by recalling their approach.

#### <span id="page-5-3"></span>1.2.1 Review: Multilinear Commitments via Random Linear Combinations

Recall that an error correcting code  $C: \mathbb{F}^k \to \mathbb{F}^n$  is an injective map with the property that any two codewords C(x), C(x'), for distinct messages  $x, x' \in \mathbb{F}^k$ , must be at least  $\delta$ -far from each other. The parameter  $\delta$  is called the distance of the code and rate of the code is k/n. We focus exclusively on linear codes which simply means that C is a linear function over  $\mathbb{F}$ .

Suppose we are given a code  $C: \mathbb{F}^k \to \mathbb{F}^n$  that has an extremely efficient encoding procedure. Brehm et al. (following [AHIV23,GLS<sup>+</sup>23]) suggest to view the message we wish to commit to as a  $k \times t$  size matrix  $X \in \mathbb{F}^{k \times t}$ , and then use C to encode each of its columns.<sup>6</sup> The resulting codeword can be viewed as a length k codeword over the alphabet  $\mathbb{F}^t$ , and is referred to as an interleaved codeword. The commitment to X is the interleaved encoding  $C^t(X)$ .

Suppose that the prover sends as the codeword the matrix A, which allegedly should be equal to  $C^t(X)$ , for some message  $X \in \mathbb{F}^{k \times t}$ . In the evaluation proof, recall that we are given as input a point  $z \in \mathbb{F}^{\log(k) + \log(t)}$  and value  $v \in \mathbb{F}$  and the goal is to convince the verifier that  $\hat{X}(z) = v$  (where we view view X as a function  $X : \{0,1\}^{\log k} \times \{0,1\}^{\log t} \to \mathbb{F}$  in the natural way). The prover first decomposes the claim about X into claim about its constituent columns. That is, it views z as  $z = (z_1, z_2) \in \mathbb{F}^{\log k} \times \mathbb{F}^{\log t}$  and sends (explicitly) to the verifier the function  $g : \{0,1\}^{\log t} \to \mathbb{F}$  defined as  $g(*) = \hat{X}(z_1, *)$ . The verifier now checks that  $\hat{g}(z_2) = v$ , which should be true in an honest execution since  $\hat{g}(z_2) = \hat{X}(z_1, z_2) = v$ .

Thus, if the original claim about X is false, the prover must lie and send an incorrect  $\tilde{g} \neq g$ . In order to detect this, the verifier now chooses a random linear combination  $r \in \mathbb{F}^t$  of the columns.

<span id="page-5-1"></span><sup>&</sup>lt;sup>5</sup>One may allow also the commitment phase to include interaction as proposed in [BFRW25]. Such interactive commitments seem necessary when working outside the unique decoding regime, which we avoid doing in this work.

<span id="page-5-2"></span><sup>&</sup>lt;sup>6</sup>Many of the prior works, such as [AHIV23], view the message as a  $t \times k$  matrix and encode the rows. We follow the "column-encoding" route (which is popular in the applied community) as it is more convenient with most of the linear algebra to come.

{6}------------------------------------------------

<span id="page-6-0"></span>Consider the vector Ar. In case A was honestly generated then  $A = C^t(X)$  and we have that  $Ar = C^t(X)r = C(Xr)$  and  $\widehat{Xr}(z_1) = \widehat{X}(z_1, r) = \widehat{g}(r)$ . Thus, the verifier would like to check that Ar is a valid encoding of a message x and, moreover, that  $\widehat{x}(z_1) = \widehat{g}(r)$ .

Via the proximity gap theorems for general linear codes [RVW13, AHIV23, BKS18] we can actually guarantee that for any matrix A to which the prover initially committed the following gap phenomenon occurs:

- A is close to an interleaved codeword  $C^t(X)$ ; or
- The random linear combination Ar is far from being a valid codeword (with high probability over r).

Thus, intuitively, it suffices for the verifier to run a proof of proximity on Ar and check that it is close to some valid codeword C(x) and, moreover, that  $\hat{x}(z_1) = \hat{g}(r)$ . The proximity test on a = Ar is done by emulating each query to a by reading the entire corresponding row of A.

The crucial point is that the resulting random linear combination x is smaller, by a factor of t, than the original message, and so we can afford to run a fairly expensive proximity test on it. Hence, to obtain an efficient multilinear PCS we need two components:

- 1. A code C with an extremely efficient encoding procedure.
- 2. An efficient procedure for checking proximity to a single copy of C (with an additional multilinear evaluation constraint).

#### 1.2.2 Sketched Codes

The core of our efficiency comes from our new error-correcting code, which we call the *sketched* code. We proceed to describe the simple construction, which is inspired by classical techniques from coding theory [GDP73, Spi96]. Let  $\alpha, \gamma \in (0, 1)$  be parameters. We utilize two main components:

- 1. A matrix  $H \in \{0,1\}^{\alpha k \times k}$  whose kernel does not contain low-weight vectors. That is, for any  $x \in \{0,1\}^n$  of Hamming weight at most  $\gamma n$ , it holds that  $Hx \neq 0$ . Put differently, H is the parity check matrix of a code of distance at least  $\gamma$ .
  - Specifically, we use the fact, due to Gallager [Gal62, Gal63], that a random sparse matrix is a good parity check matrix with high probability (the associated codes are called random low-density parity check codes, or random LDPCs for short).
- 2. Another linear error-correcting code  $C: \{0,1\}^{\alpha k} \to \{0,1\}^{(\alpha/\rho)k}$  for messages of length  $\alpha k$  with rate  $\rho > 0$  and relative distance  $\delta > 0$ .

The sketched code  $C_H$  is defined by encoding a message  $x \in \{0,1\}^k$  as  $C_H(x) = (x, C(Hx))$ . The intuition for why  $C_H$  has large distance is simple. First observe that C is a linear code and so it suffices to show that it does not have low weight codewords. Fix a message  $x \in \mathbb{F}^k$ . There are two options:

1. If x has high weight: then so does  $C_H(x)$  since we include x in the first component of the codeword.

{7}------------------------------------------------

<span id="page-7-3"></span>2. If x has low weight then by the parity check property, we know that Hx ̸= 0. Therefore, by the distance of C we have that C(Hx), the second component in the codeword, has high weight.

The above yields a code defined over bits. To obtain a code over a larger binary field, one can utilize bit slicing — that is, work over a binary extension field and encode each field element by separately encoding the individual bits.

Thus, the parity check matrix allows us to compress the given message into a sketch, which is then encoded under another code. This is beneficial since we can now encode the shorter sketch cheaply even using a slower code. As an additional benefit, even if the code C is non-systematic, we automatically get that the sketched code is systematic which, as noted above, can be very helpful in obtaining a fast SNARK prover.[7](#page-7-0)

We consider both an asymptotic and concrete instantiation of the above approach.

The Asymptotic Regime. In this regime, we take the base code C to be a linear-time encodable code with encoding time ck′ for messages of size k ′ , for some constant c ≥ 1. In this case we set the parameter α = ε/c, for a small ε > 0, so that the encoding time of C for messages of size αk is merely εk. If we take H to be a super-sparse random matrix with three 1's in each row, then computing Hx can be done in time 3k. Remarkably, Gallager's result shows that even such a sparse matrix is a parity check matrix of a code with some constant δ > 0 distance (with inverse polynomial probability over the choice of H).[8](#page-7-1)

By encoding the sketch using the code C we we get that the encoding time is 3k+c·αk = (3+ε)k as desired.

The Concrete Instantiation. Here we make a few choices that differ from the asymptotic ones. First, we do not take H to be maximally sparse as in the asymptotic case, so as not to make the distance δ be too small (recall that small distance translates to a large constant overhead in the proof size). Specifically, we set our shrinkage factor to be α = 1/8 and the the number of ones per row of the matrix to also be 8. Building on Gallager's analysis we show that the resulting parity check matrix induces a code of distance at least 0.013 (with high probability).[9](#page-7-2)

As for the code C, we simply take the Reed-Solomon code, with rate ρ = 1/2. We observe that due to the 8× shrinkage factor, we will be encoding relatively small messages. For such messages the Reed-Solomon encoding is already extremely fast and so there is no point to employ more elaborate techniques. Immediately switching to the Reed-Solomon code will also help us substantially when designing a proximity test for the sketched code.

For instance, consider the PCS application for a 30-variate multilinear over GF(232), corresponding to committing to N = 2<sup>30</sup> field elements). In this case we take an interleaving factor of t = 2<sup>7</sup> .

<span id="page-7-0"></span><sup>7</sup>Note that often codes can be made systematic but there is an associated cost. For example, to make the Reed-Solomon code systematic one needs to do an inverse FFT before performing the FFT (which increases the encoding cost by 50% in the popular rate 1/2 regime). RAA codes used in [\[BCF](#page-25-3)<sup>+</sup>25] are non-systematic. As Baweja et al. [\[BMMS25a\]](#page-26-0) point out, the code can be made systematic by prepending the message to the codeword, but this approach is purely theoretical since doing so significantly deteriorates both the rate and distance of the code.

<span id="page-7-1"></span><sup>8</sup>Unfortunately, Gallager's result is tight in the sense that reducing the sparsity further leads to poor distance, see [\[Gal63,](#page-26-9) Theorem 2.5].

<span id="page-7-2"></span><sup>9</sup>Observe that this is fairly close to the GV bound, which for the current setting of parameters, is roughly 0.0174. See [Section 6](#page-19-0) for details.

<span id="page-8-0"></span>This means that we need the code  $C_H$  to encode t messages of size  $k = N/t = 2^{23}$  each. With the shrinkage factor  $8\times$ , this means that the encoding involves  $t = 2^7$  FFTs of size  $\alpha k \cdot \rho^{-1} = 2^{21}$  each. Compared to just an interleaved Reed-Solomon encoding (as in [AHIV23, NA25], this reduces the total amount of FFT in the encoding by  $8\times$ , at the cost of having to compute H, which requires 8N XOR operations (but is actually more dominated by I/O).

On the Piece-wise Distance of the Sketched Code. As noted above, the sketched codes have the following distance guarantee. Every codeword  $(x,y) \in \mathbb{F}^k \times \mathbb{F}^{(\alpha/\rho)k}$  has the property that either x has relative weight at least  $\gamma$  or y has relative weight at least  $\delta$ , where  $\delta$  is the distance of the underlying code C. Hence, the codeword is guaranteed to have weight at least

$$\frac{\min\left(\gamma,\delta\cdot(\alpha/\rho)\right)}{1+(\alpha/\rho)}.$$

For our concrete instantiation, this is about 0.01.

However, we observe that we can actually work directly with the "piecewise" distance guaranteed above. Doing so allows us to work with the distance  $\gamma = 0.013$  of the first half and reduces the number of queries that we have to do to that part by about 25%.

#### 1.2.3 Proving Proximity to Sketched Codes

Thus, in our PCS we encode the data using an interleaved sketched code. But what about the evaluation proof? Recall that the goal in an evaluation proof is to convince the verifier that the polynomial that we committed to satisfies a multilinear polynomial evaluation constraint.

Using the random linear combination apporach discussed in Section 1.2.1, the question then boils down to a proof of proximity for a single instance of the sketched code (more precisely we also need to support a multilinear evaluation claim on the underlying message, but we defer the details on that to Remark 1.3 below).

Thus, we want to check that a given oracle  $(x, y) \in \mathbb{F}^k \times^{(\alpha/\rho)k}$  is close to a codeword  $C_H(x')$ , for some  $x' \in \mathbb{F}^k$ .

Recall that in our construction y should be a valid codeword under the base code C. Thus, as our first step, we run a proximity test of y. In our concrete instantiation we use the Reed-Solomon code as our base code and so we can run protocols such as WHIR [ACFY25] or Ligerito [NA25], which are specifically designed for this task.

Thus, we can ensure that y is close to some encoding C(u) for a message  $u \in \mathbb{F}^{\alpha k}$ . Our next goal is to check that x is close to  $H^{-1}(u)$ . A natural approach is to just have the verifier choose a few of the sparse constraints induced by u via H and check that they are satisfied by x. Intuitively, it seems to makes sense that if x satisfies many random constraints then it is close to a vector that satisfies all of them. Unfortunately, as shown by [BHR05] this intuition is actually false. Thus, we need to find a different way of checking consistency between x and u.

To do so, we follow the code-switching idea from [RR24], as implemented in [BCF<sup>+</sup>25]. Specifically, the prover sends a new *fresh* encoding  $w = \mathsf{RS}(x)$  of x under the Reed-Solomon code (with rate say 1/2). Thus, it suffices to check that:

- 1. w is close to a Reed-Solomon codeword RS(m).
- 2. x is close to m.

{9}------------------------------------------------

#### <span id="page-9-1"></span>3. Hm = u.

Before describing how to perform these checks, it is worthwhile to pause and wonder what is the point of re-encoding x with a Reed-Solomon code. Wasn't the entire purpose to avoid doing so? The point, is that we are only doing this re-encoding after the random linear combination was applied to the committed interleaved codeword. Hence, the prover can encode the t columns using the highly efficient code CH, and now only needs to encode the single random linear combination using the Reed-Solomon code.

Returning to our checks, it turns out that all three can be solved using existing techniques. The first can be directly checked using Reed-Solomon proximity checks such as WHIR or Ligerito. The second can be done by first doing spot checks — i.e., reading roughly λ/γ random coordinates of x and checking that m satisfies them, where λ is the security parameter. The checks on m can be first batched using sumcheck based techniques [\[RR24,](#page-27-3) [CBBZ23\]](#page-26-10) and then incorporated into protocols such as WHIR/Ligerito.

As for the third check, note that the verifier has access to w which is close to RS(x ′ ) and y which is close to RS(u) and needs to check that Hm = u. This is exactly the problem solved by lookup protocols. Specifically, we use the recent efficient protocol due to Bünz et al. [\[BCD25\]](#page-25-5).

<span id="page-9-0"></span>Remark 1.3 (Supporting a Multilinear Evaluation Proof). In the above description we focused only on the proximity check to C<sup>H</sup> and ignored the additional multilinear evaluation that we need to support. That is, not only do we need to check that (x, y) is close to a valid codeword CH(x ′ ), but moreover, that the multilinear extension of x ′ at a given point z is equal to a value v.

Recall that in the protocol the prover sends a fresh encoding RS(x). To handle the multilinear evaluation claim we simply incorporate this claim when batching the spot checks via the protocol of [\[RR24,](#page-27-3)[CBBZ23\]](#page-26-10).

Remark 1.4. Throughout this work we focus exclusively on the unique decoding regime. In all of the Reed-Solomon proximity checks described above, we could in principle utilize proximity gaps in the list decoding regime [\[BCI](#page-25-7)+23[,BCH](#page-25-8)+25]. This would require adding an "out-of-domain" sampling step and would only slightly reduce our query complexity, since the query complexity will be dominated by the spot checks to x.

A far more noticeable improvement would be obtained if good list-decoding proximity gap bounds could be shown for random LDPCs, and we leave this as a fascinating direction for future improvement, see [Section 1.3.](#page-10-0)

Cheaper Amortized Evaluation Proof. The most expensive part of the evaluation proof turns out to be the lookup argument of [\[BCD25\]](#page-25-5). We note however that the cost can be amortized as follows.

Recall that we use their protocol as follows: the verifier is given oracle access to strings that are respectively close to RS(m) and RS(u) and its goal is to check that Hm = u. Using the sumcheck protocol, this can be reduced to multilinear evaluation claims respectively to m, u and H. The claims about m (resp., u) can be incorporated directly into the proximity checks that we run on w (resp., y). Thus, we are left with a multilinear evaluation claim about the sparse matrix H.

We make two observations about this claim:

• First, it is totally independent of the committed message. Computing it, or proving correctness of its evaluation, could potentially be outsourced to a much more powerful server than what the committing player has.

{10}------------------------------------------------

<span id="page-10-3"></span>• Second, if we run several evaluation proofs, even accross different committed polynomials, we end up with multiple multilinear evaluation claims about the same matrix H. Using the sumcheck protocol we can batch these multiple claims to a single multilinear evaluation claim about H, and then only run the [\[BCD25\]](#page-25-5) protocol once on the batched instance.

## <span id="page-10-0"></span>1.3 Open Questions

Our work leaves several fascinating open questions:

- 1. On the theoretical side, what is the optimal computation rate of encoding? and of proofs? in particular can one construct either one with less than 3× overhead?
  - More generally, what is the best tradeoff between computation rate and distance in the case of codes, or with queries in the case of IOPs?
- 2. Can analogs of [Theorems 1.1](#page-3-0) and [1.2](#page-4-0) that use a deterministic code be constructed? Perhaps via expanders as in [\[Spi96\]](#page-28-2)?
- 3. As mentioned above, our works gives us a concrete motivation for studying proximity gaps for random LDPC codes. This is somewhat surprising because LDPC codes do not offer efficient encoding. However, the sketched code construction shows that they are nevertheless extremely useful. A promising direction for this are the works [\[MRR](#page-27-4)+20,[GG25\]](#page-26-11).
- 4. A compelling approach to constructing arguments with extremely fast provers and small proofs is to compose a fast prover protocol such as ours, with one that has a fast verifier. In order to do so in practice, we need to be able to prove our verification circuit using a verifier-efficient proof-system, without too much prover overhead.
  - We leave open the question of realizing a concretely efficient proof recursion using Bolt.
- 5. On a related note, Bolt does not provide hiding, aka zero-knowledge, guarantees. Indeed, zero-knowledge seems to be at odds with having a systematic encoding. One potential way to get around this is by recursing using a zero-knowledge proof.

Let F be a finite field. We denote the Hamming weight (i.e., number of non-zero entries) of a vector x ∈ F <sup>n</sup> by hw(x) and say that x has weight w. For two equal length vectors x, x′ ∈ F <sup>n</sup> we denote their relative Hamming distance by ∆(x, x′ ) = hw(x − x ′ )/n.

### <span id="page-11-3"></span><span id="page-11-0"></span>2.1 Multilinear Extension

Let F be a finite field and k be a positive integer. Any function f : {0, 1} <sup>k</sup> → F defined on the Boolean hypercube has a unique extension to the entire domain F <sup>k</sup> as a multilinear polynomial.

Definition 2.1 (Multilinear Extension). The multilinear extension (MLE) of f : {0, 1} <sup>k</sup> → F is the unique multilinear polynomial ˆf : F <sup>k</sup> → F that agrees with f on {0, 1} k .

The multilinear extension can be computed explicitly as follows. For any point z = (z1, . . . , zk) ∈ F k :

$$\hat{f}(z) = \sum_{b \in \{0,1\}^k} eq(z,b) \cdot f(b),$$

where eq is equality polynomial defined as:

$$eq(b,z) = \prod_{i=1}^{k} (b_i \cdot z_i + (1-b_i) \cdot (1-z_i)).$$

Throughout this work, for a vector v ∈ F <sup>N</sup> where N = 2<sup>k</sup> , we identify v with the function v : {0, 1} <sup>k</sup> → F such that v(b) = vint(b) , where int(b) is the integer with binary representation b. We denote the MLE of the vector by vˆ.

<span id="page-11-2"></span>Lemma 2.2 (Polynomial Identity Lemma). If P : F <sup>m</sup> → F is a non-zero total degree d polynomial then:

$$\Pr_{r \in \mathbb{F}^m}[P(r) = 0] < d/|\mathbb{F}|.$$

## <span id="page-11-1"></span>2.2 Interactive Proofs and IOPs

Interactive Oracle Proofs (IOP) [\[BCS16,](#page-25-9) [RRR16\]](#page-28-4) are a hybrid between interactive proofs, and probabilistically checkable proofs. In this work we consider IOPs for promise problems.

Recall that a promise problem consists of two distjoint sets YES and NO. In an IOP for a promise problem (YES, NO), the goal is for the verifier to accept x ∈ YES and reject x ∈ NO. There is no requirement for instances outside of YES ∪ NO. We assume the verifier is given oracle access to the main input x, and we count the number of queries (in particular, this lets us capture the notion of IOPPs [\[RRR16\]](#page-28-4)). The prover, who is given x explicitly, is allowed to interact with the verifier over multiple rounds. In each round the prover sends an oracle proof string, and the verifier responds with a random string. At the end of the interaction, the verifier accepts or rejects. For sake of efficiency it will also be convenient to allow the prover to send explicit (i.e., non oracle) messages.

The protocol satisfies completeness if for every x ∈ YES, the prover convinces the verifier with probability 1. The protocol has soundness error ϵ if for every x ∈ NO and every malicious prover P ∗ in the IOP model, the probability that the verifier accepts is at most ϵ.

An IOP for a language L is an IOP for the promise problem (YES, NO), where YES = L and NO = {0, 1} <sup>∗</sup>\L. Similarly, for δ > 0, a δ-IOPP for a language L is an IOP for the promise problem (YES, NO), where YES = L and NO = {x : ∆(x, L) > δ}.

Oracle Aided IOPs. We also consider IOPS in which the verifier has oracle access to some oracle O. We say in this case that the protocol is an O-oracle aided IOP.

#### <span id="page-12-5"></span>2.2.1 Interactive Proofs and the Sumcheck Protocol

We will make use of the celebrated sumcheck protocol [\[LFKN92\]](#page-27-5), with a linear-time prover [\[Tha13\]](#page-28-5), as described next:

<span id="page-12-4"></span>Lemma 2.3 (The Sumcheck Protocol). Let F be a finite field and m, d ∈ N. There exists a protocol between a prover, who is given as input a description of an m-variate polynomial Q : F <sup>m</sup> → F of individual degree d, and a verifier who gets as input α ∈ F and either rejects or outputs (z, v) ∈ F <sup>m</sup> × F. The protocol satisfies the following properties:

- Complexity: the protocol has m rounds. In each round the prover sends d + 1 field elements and the verifier responds with a single field element. The verifier performs O(d · m) field operations. The prover runs in time O((d + 1)m). In case Q is a product of d multilinear polynomials, the prover time decreases to O(d · 2 <sup>m</sup>).
- Completeness: if α = P <sup>x</sup>∈{0,1}<sup>m</sup> Q(x) then when interacting with the honest prover, the verifier always outputs (z, v) such that Q(z) = v.
- Soundness: if α ̸= P <sup>x</sup>∈{0,1}<sup>m</sup> Q(x) then, when interacting with any prover, with probability at least 1 − dm |F| , the verifier either rejects or outputs (z, v) such that Q(z) ̸= v.

Interactive Proofs for Batching Polynomial Evaluations. We recall an interactive protocol that reduce multiple multilinear evaluation claims into a single one due to [\[RR24,](#page-27-3) [CBBZ23\]](#page-26-10). We state the result from [\[CBBZ23\]](#page-26-10) due to its optimzied prover complexity.

<span id="page-12-1"></span>Theorem 2.4 ( [\[CBBZ23,](#page-26-10) Algorithm 4]). Let f : {0, 1} <sup>k</sup> <sup>→</sup> <sup>F</sup> be a function and denote by <sup>f</sup><sup>b</sup> its multilinear extension. There exists an interactive protocol that reduces t > 1 evaluation claims for <sup>f</sup><sup>b</sup> to a single evaluation claim for <sup>f</sup>b. The protocol has soundness error <sup>O</sup> k+log<sup>2</sup> (t) |F| , prover time O(t·2 k ), verifier time O(tk), communication complexity O(k+log(t)) and k+log<sup>2</sup> (t)+O(1) rounds. Furthermore, if only t ′ ∈ [t] of the evaluation points are outside the Boolean hypercube, then the prover times is reduced to O(t ′ · 2 k ). [10](#page-12-2)

### <span id="page-12-0"></span>2.3 Coding Theory

Let F be a finite field. A code is an injective function C : F <sup>k</sup> → F n . The rate of the code is defined as k/n and its minimal distance is minx̸=x′′ ∆(C(x), C(x ′′)). We say that C is linear C is a linear function.

We will be interested in proofs of proximity to codes. Actually, following [\[BCF](#page-25-3)+25,[ACFY25\]](#page-24-2) we will be interested in proofs of proximity to codewords that further satisfy a multilinear evaluation constraint.

<span id="page-12-3"></span>Definition 2.5 (Constrained Code). We define the constrained relation relative to C as:

$$R_C = \Big\{ \big( (z, v), c, m \big) \in \mathbb{F}^{\log k} \times \mathbb{F} \times \mathbb{F}^n \times \mathbb{F}^k : c = C(m) \, \wedge \, \hat{m}(z) = v \Big\},\,$$

where (z, v) is the explicit input, c is the implicit input and m is the witness.

<span id="page-13-2"></span>Interleaved Codes. In this work we utilize interleaved codes. For a parameter t ∈ N, the t-fold interleaving of C, denoted by C t , consists of all n × t matrices whose columns belong to C. We sometimes view C <sup>t</sup> as a code of blocklength n whose symbols are the entire rows of this matrix.

Thus, ∆(M, C<sup>t</sup> ) corresponds to the number of entire rows of M that need to be changed to derive a valid interleaved codeword.

Parity Check Matrices. A code with distance γ > 0 can be associated with a parity check matrix, defined as follows.

Definition 2.6 (Parity Check Matrix). Let F be a finite field. For n, k ∈ N and γ ∈ (0, 1), an (n, k, γ)-parity check matrix is a matrix H ∈ F k×n such that Hx ̸= 0 for all x ∈ F <sup>n</sup> with Hamming weight |x| ≤ γn.

#### 2.3.1 Reed-Solomon Code

Let F be a finite field and fix an evaluation set L ⊆ F. For a parameter d ≤ |F| − 1, recall that the RS[F, L, d] consists of all functions f : L → F that agree with a degree d univariate polynomial. We say that L is smooth if F is a prime field and L is a multiplicative subgroup, or if |F has characteristic 2 and L is an additive subgroup.

<span id="page-13-0"></span>Theorem 2.7 (Constrained RS-IOPP [\[ACFY25\]](#page-24-2)). Assume that |F| = Ω(k/|F|). Then, for every proximity parameter δ ∈ (0,(1 − ρ)/2), there exists a δ-IOPP for the constrained relation RRS with the following parameters (where λ is the security parameter):

- Soundness Error: 2 −λ .
- Input Query Complexity: <sup>λ</sup> − log(1−δ) .
- Proof Query Complexity: O((λ/δ) · log log n).
- Verifier Time: O(λ · log(n)).
- Prover Time: O(n log n).

#### 2.3.2 Proximity Gaps

We will make use of proximity gaps [\[RVW13,](#page-28-3) [AHIV23,](#page-24-0) [BKS18\]](#page-26-7) for general linear codes. We use a version that is nowadays referred to as "correlated agreement".

<span id="page-13-1"></span>Theorem 2.8 (Correlated Agreement for Linear Codes [\[BKS18,](#page-26-7) Theorem 4.1], see also [\[BCF](#page-25-3)+25, Corollary 2.4]). Let C ⊆ F n be a linear code with distance λ, let t ∈ N and let X ∈ F n×t be a matrix such that ∆(X, C<sup>t</sup> ) > δ. For any η > 0 such that δ − η < λ/3,

$$\Pr_{r \in \mathbb{F}^t} \left[ \Delta(X \cdot r, C) < \delta - \eta \right] \le \frac{\eta^{-1}}{|\mathbb{F}|}.$$

## <span id="page-14-0"></span>3 Sketched Coding

Let  $k \in \mathbb{N}$  and  $\alpha, \gamma \in (0,1)$ . Suppose that  $H \in \mathbb{F}^{\alpha k \times k}$  is a  $\gamma$ -parity check matrix. Let  $C : \mathbb{F}^{\alpha k} \to \mathbb{F}^{\alpha k/\rho}$  be a linear error-correcting code for messages of length  $\alpha k$ , with rate  $\rho \in (0,1)$  and distance  $\delta \in (0,1)$ .

We construct a new systematic linear error-correcting code as  $C_H : \mathbb{F}^k \to \mathbb{F}^{\left(1+\frac{\alpha}{\rho}\right)k}$  is defined as  $C_H(x) = (x, C(Hx))$ , for every  $x \in \mathbb{F}^k$ .

<span id="page-14-2"></span>**Distance Analysis.** The sketched codes exhibit a stronger notion of "piecewise distance":

Claim 3.1. For two distinct codewords  $(x,y), (x',y') \in C_H$ , either  $\Delta(x,x') > \gamma$  or  $\Delta(y,y') > \delta$ .

Proof. If  $y \neq y'$  then  $\Delta(y, y') > \delta$  due to the distance of C. Otherwise we are in the case that y = y' and  $x \neq x'$ . This means that Hx = Hx' or in other words  $H(x - x') = \mathbf{0}$ . Thus, as H is a partiy check matrix we have that  $\Delta(x, x') = \Delta(x - x', 0) > \gamma$ .

Claim 3.1 implies that the sketched code has distance at least:

$$\frac{\min(\gamma k, \delta \alpha k/\rho)}{(1 + (\alpha/\rho))k} = \frac{\min(\gamma, \delta \cdot (\alpha/\rho))}{1 + (\alpha/\rho)},$$

but in this work we will be focusing mostly on the stronger notion of piecewise distance established by Claim 3.1.

## <span id="page-14-1"></span>4 IOP for Sketched Codes

In this section we construct an IOP for proximity (under the piecewise distance) to a single sketched codeword. Later, in Section 5, we consider the case of interleaved codes.

For  $\alpha \in (0,1)$ , let  $H \in \mathbb{F}^{\alpha k \times k}$  be a  $(k, \alpha k, \gamma)$ -parity check matrix and  $C : \mathbb{F}^{\alpha k \to (\alpha/\rho)k}$  be a base code with rate  $\rho$  and distance  $\delta$ . Let  $C_H : \mathbb{F}^k \to \mathbb{F}^{(1+\alpha/\rho)k}$  be the sketched code of C as defined in Section 3. That is,  $C_H(x) = (x, C(Hx))$ .

Let  $R_C$  be the constrained relation relative to C (see Definition 2.5). Let  $\delta' \in (0, \delta/2)$  and  $\gamma' \in (0, \gamma/2)$  be parameters. (Jumping ahead, we will be assuming that  $R_C$  has an IOPP relative to distance  $\delta'$ , whereas  $\gamma'$  can be any parameter within the unique decoding radius of the code for which H is a parity check matrix.)

We consider a promise problem (YES, NO) defined as follows. First, define:

$$\mathsf{YES} = \left\{ \left( C_H(x), z, v \right) \in \mathbb{F}^{(1 + (\alpha/\rho))k} \times \mathbb{F}^{\log k} \times \mathbb{F} \mid x \in \mathbb{F}^k \text{ and } \hat{x}(z) = v \right\}.$$

We define NO to be the set of tuples  $((x,y),z,v) \in (\mathbb{F}^k \times \mathbb{F}^{(\alpha/\rho)k}) \times \mathbb{F}^{\log k} \times \mathbb{F}$  such that either:

- y is at least  $\delta'$ -far from C; or
- y is  $\delta'$ -close to some (unique) codeword C(u) but x is  $\gamma'$ -far from the affine space  $\{m \in \mathbb{F}^k : Hm = u\}$ ; or
- y is  $\delta'$ -close to some (unique) codeword C(u) and x is  $\gamma'$ -close to a (unique)  $m \in \mathbb{F}^k$  such that Hm = u, but  $\hat{m}(z) \neq v$ , where we view m as function  $m : \{0,1\}^{\log k} \to \mathbb{F}$ .

<span id="page-15-10"></span>(In both cases the uniqueness follows from the fact that the relevant vectors are in the unique decoding radius of the respective codes.)

<span id="page-15-1"></span>Lemma 4.1. Suppose that R<sup>C</sup> has a δ ′ -IOPP with input query complexity qinp, proof query complexity qprf , non-oracle communication cc, prover time T<sup>P</sup> , verifier time T<sup>V</sup> and soundness error 2 −λ .

Then, assuming |F| = Ω(2<sup>λ</sup> · (k + λ)), the promise problem (YES, NO) has an Hˆ -oracle aided IOP with the following parameters:

- Soundness error: 2 −Ω(λ) .
- Input Query Complexity: the verifier makes <sup>λ</sup> − log(1−γ ′) queries to x and qinp queries to y. It fully reads z and v.
- Proof Query Complexity: O(λ log log(k)) + qprf .
- Non-oracle Communication Complexity: O(log k).
- Prover Time: T<sup>P</sup> + O(k · log(k) + Sparsity(H)) field operations.
- Verifier Time: T<sup>V</sup> + O log(k) · λ γ ′ .

### <span id="page-15-0"></span>4.1 Proof of [Lemma 4.1](#page-15-1)

The input is (x, y) ∈ F <sup>k</sup> × F (α/ρ)k together with z ∈ F log <sup>k</sup> and v ∈ F. We simultaneously view x as a function x : {0, 1} log <sup>k</sup> → F. The protocol proceeds as follows:

- <span id="page-15-9"></span>1. The prover sends a fresh Reed-Solomon encoding, with rate ρ ′ ∈ (0, 1), of x, namely w = RS<sup>ρ</sup> ′(x) of x.
- <span id="page-15-7"></span>2. The verifier samples a random r ∈ F log(αk) and sends it to the prover. The prover responds with σ = ˆu(r), where u = Hx.
- <span id="page-15-5"></span>3. The prover and verifier use the sumcheck protocol of [Lemma 2.3](#page-12-4) to reduce the claim that σ = P i∈{0,1} log <sup>k</sup> Hˆ (r, i) · x(i) to a claim on Hˆ (r, s) and a claim that xˆ(s) = τ , for s ∈ F log(k)) and τ ∈ F. The claim about Hˆ is checked directly using the provided oracle.
- <span id="page-15-8"></span>4. The receiver chooses at random a (multi-)set I ⊆ {0, 1} log <sup>k</sup> of size t = λ − log(1−γ ′) . The receiver reads x(i) from the input for all i ∈ I.
- <span id="page-15-6"></span>5. The parties run the protocol of [Theorem 2.4](#page-12-1) to reduce the evaluations claims xˆ(z) = v, xˆ(s) = τ and {x(i)}i∈<sup>I</sup> to a single claim xˆ(z ′ ) = v ′ for some z ′ ∈ F log <sup>k</sup> and v ′ ∈ F.
- <span id="page-15-3"></span>6. The parties run the IOPP for constrained RS codes of [Theorem 2.7](#page-13-0) relative to input w, the evaluation point z ′ and evaluation v ′ .
- <span id="page-15-4"></span>7. The parties run the IOPP for the constrained relation of C relative to y, the evaluation point r and corresponding evaluation σ.
- 8. The verifier accepts if all of the above procedures accept, otherwise it rejects.

<span id="page-15-2"></span>Remark 4.2. In our actual implementation, the code C is also a Reed-Solomon code and so we can batch the protocols in Steps [6](#page-15-3) and [7.](#page-15-4) Additionally, the underlying IOPP for Reed-Solomon codes [\[ACFY25,](#page-24-2) [NA25\]](#page-27-0) support parallelizing the proximity check with a sumcheck which yields a further concrete optimization.

**Completeness.** Fix an input ((x,y),z,v) where  $(x,y)=C_H(x), z \in \mathbb{F}^{\log k}$  and  $v \in \mathbb{F}$  so that  $\hat{x}(z)=v$ , where recall that we view  $x:\{0,1\}^{\log k}\to\mathbb{F}$  also as a function. Denote u=Hx and note that y=C(u).

Recall that the prover sends in its first message  $w = \mathsf{RS}_{\rho'}(x)$ . Let  $r \in \mathbb{F}^{\log(\alpha k)}$  be the corresponding verifier message. Since Hx = u it holds that  $\sum_{i \in \{0,1\}^{\log k}} \hat{H}(r,i) \cdot x(i) = \hat{u}(r)$ .

Hence, the sumcheck protocol invoked in Step 3 is run on a correct claim. Thus, by completeness of the sumcheck protocol, the resulting claim on  $\hat{H}$  and  $\hat{x}(s) = \tau$  are correct.

Then, in Step 5 the parties invoke the protocol of Theorem 2.4 on a correct claim, and therefore  $\hat{x}(z') = v'$ .

As such, in Step 6 the prover and verifier invoke the protocol of Theorem 2.7 on a correct claim — that w is an encoding of a message x that is consistent with the claim  $\hat{x}(z') = v'$ . Similarly, in Step 7 the protocol is invoked on a correct claim — that y is an encoding of a message u such that  $\hat{u}(r) = \sigma$ .

**Soundness.** Fix an input  $((x,y),z,v) \in NO$ . Recall that  $x \in \mathbb{F}^k$ ,  $y \in \mathbb{F}^{(\alpha/\rho)k}$  and  $z \in \mathbb{F}$ .

Fix a (wlog deterministic) cheating prover strategy. We show that the verifier rejects the claim with high probability. We consider several cases:

Case 1: y is far. Suppose that y is  $\delta'$ -far from C. Then, in Step 7 we run the underlying IOPP on a NO instance and the verifier rejects with all but  $2^{-\lambda}$  probability.

Thus, we may assume that y is close to a (unique) codeword C(u) for some  $u \in \mathbb{F}^{\alpha k}$ .

Case 2: w is far. Denote the message sent by the prover in the first round by w. Suppose that w is  $((1-\rho')/2)$ -far from  $\mathsf{RS}_{\rho'}$ . Then, in Step 6 we run the protocol of Theorem 2.7 on a NO instance and the verifier rejects with all but  $O(k/|\mathbb{F}|)$  probability.

Thus, we may also assume that w is close to a (unique) codeword C(m) for some  $m \in \mathbb{F}^k$ .

Case 3:  $u \neq Hm$ . Consider now the case that  $u \neq Hm$ . Then, in Step 2, by Lemma 2.2, with all but  $\log(\alpha k)/|\mathbb{F}|$  probability, it holds that  $\hat{u}(r) \neq \sum_{i \in \{0,1\}^{\log k}} \hat{H}(r,i) \cdot m(i)$ . Assume that such an r was indeed chosen.

Let  $\sigma$  be the value claimed by the prover in Step 2. If  $\sigma \neq \hat{u}(r)$  then in Step 7 the claim is false and so the verifier rejects with all but  $2^{-\lambda}$  probability. Thus, we may assume that  $\sigma = \hat{u}(r)$  and in particular that  $\sigma \neq \sum_{i \in \{0,1\}^{\log k}} \hat{H}(r,i) \cdot m(i)$ .

Thus, in Step 3 the Sumcheck protocol is invoked on a false claim. Hence, with all but  $O(k/|\mathbb{F}|)$  probability, either the claim about  $\hat{H}(r,s)$  is false, in which case the verifier immediately rejects (via its oracle query to the multilinear extension of H), or the claim about m is false. That is,  $\tau \neq \hat{m}(s)$ . But this means that in Step 5 one of the claims is false and so it produces a false claim, with all but probability  $O(k + \log(t))/|F|$ , which then, in Step 6, makes the verifier reject with all but  $2^{-\lambda}$  probability.

Thus, we may assume that u = Hm.

Case 4:  $\hat{m}(z) \neq v$ . Then in Step 5 one of the claims is false and so, with all but probability  $O(k + \log(t))/|F|$ , it produces a false claim which then, in Step 6, makes the verifier reject with all but  $2^{-\lambda}$  probability.

<span id="page-17-1"></span>Case 5: x is γ ′ -far from m. In Step [4](#page-15-8) the verifier makes t random samples to x. Thus, with all but (1 − γ/3)<sup>t</sup> probability, it holds that x(i) ̸= m(i) for some i ∈ I. Since t = −λ/ log(1 − γ/3) this probability becomes 2 −λ .

Assume that the above event indeed occurs. But this means that in Step [5](#page-15-6) one of the claims is false and so, with all but probability O(k + log(t))/|F|, it produces a false claim which then, in Step [6,](#page-15-3) makes the verifier reject with all but 2 <sup>−</sup><sup>λ</sup> probability.

Case 6: x is γ ′ -close to m. In this case x is at distance at most γ/3 from m ∈ H−<sup>1</sup> (u) and moreover mˆ (z) = v, in contradiction to our assumption that ((x, y), z, v) ∈ NO.

Complexity. The query complexity is t = λ − log(1−γ ′) queries to x in Step [4](#page-15-8) and q queries to y in Step [7.](#page-15-4)

The prover time is dominated by O(k log(k)) for creating w in Step [1](#page-15-9) (via the FFT algorithm). ## <span id="page-17-0"></span>5 IOP for Interleaved Sketched Codes

We consider the same setup as in [Section 4.](#page-14-1) For α ∈ (0, 1), let H ∈ F αk×<sup>k</sup> be a [k, αk, γ] parity-check matrix and C : F αk→(α/ρ)<sup>k</sup> be a base code with rate ρ and distance δ. Let C<sup>H</sup> : F <sup>k</sup> → F (1+α/ρk be the sketched code of C defined as CH(x) = (x, C(Hx)).

For an interleaving parameter t ∈ N, which is a power of two, and proximity error η > 0, we consider the interleaved promise problem (YES<sup>t</sup> , NOt), defined as follows.

- The YES<sup>t</sup> instances are defined as (X, Y, z, v), where X ∈ F k×t , Y ∈ F (α/ρ)·k×t , z ∈ F log(k)+log(t) and v ∈ F such that Y = C t (HX) and Xˆ(z) = v
- The NO<sup>t</sup> instances are defined as (X, Y, z, v) such that one of the following conditions holds:
  - Y is (ρ/3 + η)-far from the interleaved code C t ; or
  - Y is (ρ/3 + η)-close to a (unique) interleaved codeword C t (U) for U ∈ F αk×<sup>t</sup> but X is (γ/3 + η)-far from the affine space {M ∈ F k×t : HM = U}; or
  - X is (ρ/3 + η)-close to a (unique) M such that HM = U but Mˆ (z) ̸= v.

(In both cases the codewords are unique as they are within the unique decoding radius of the respective codes.)

Lemma 5.1. Suppose that R<sup>C</sup> has a δ ′ -IOPP with input query complexity qinp, proof query complexity qprf , non-oracle communication cc, prover time T<sup>P</sup> , verifier time T<sup>V</sup> and soundness error 2 −λ .

Then, assuming |F| = Ω(2<sup>λ</sup> · (k + λ)), the promise problem (YES<sup>t</sup> , NOt) has an Hˆ -oracle aided IOP, with the following parameters:

- Soundness error: 2 −Ω(λ) .
- Query Complexity: the verifier makes <sup>λ</sup> <sup>−</sup> log(1−γ/3) queries to the rows of X and qinp queries to the rows of Y . It fully reads z and v.

- <span id="page-18-5"></span>• Proof Query Complexity:
- Proof Query Complexity:  $O(\lambda \log \log(k)) + q_{prf}$ .
- Non-oracle Communication Complexity:  $O(t + \log k)$ .
- Prover Time:  $2k \cdot t$  field additions and multiplications and an additional  $T_P + O(k \log(k)) + Sparsity(H)$ ) field operations.
- Verifier Time:  $T_V + O\left(t \cdot \log(k) \cdot \frac{\lambda}{\gamma'}\right)$ .

#### <span id="page-18-0"></span>5.1 Proof of Lemma 5.1

The protocol on input (X, Y, z, v) proceeds as follows:

- 1. Parse z as  $z = (z_1, z_2) \in \mathbb{F}^{\log k} \times \mathbb{F}^{\log t}$ .
- 2. The prover generates and sends the function  $g:\{0,1\}^{\log t}\to\mathbb{F}$  defined as  $g=\hat{X}(z_1,*)$ .
- <span id="page-18-2"></span>3. The verifier checks that  $\hat{g}(z_2) = v$ .
- <span id="page-18-3"></span>4. The verifier chooses a random  $r \in \mathbb{F}^t$  and sends it to the prover.<sup>11</sup>
- <span id="page-18-4"></span>5. The prover and verifier run the IOP of Lemma 4.1 on the input  $(x, y, z_1, \langle r, g \rangle)$ , where x = Xr and y = Yr. The protocol is invoked relative to parameter  $\delta' = \delta/3$  and  $\gamma' = \gamma/3$ . Each query to x (resp. y) is emulated via a single query to the corresponding full row of X (resp., Y).

**Completeness.** Fix an input ((X,Y),z,v) where  $(X,Y)=C_H^t(X), z\in \mathbb{F}^{\log k+\log t}$  and  $v\in \mathbb{F}$  so that  $\hat{X}(z)=v$ , where recall that we view X as a function  $X:\{0,1\}^{\log k+\log t}\to \mathbb{F}$ . Let U=HX and note that  $Y=C^t(U)$ .

Recall that the prover sends in its first message  $g = \hat{X}(z_1, *)$ . Note that the verifier's check in Step 3 passes since  $\hat{g}(z_2) = \hat{X}(z_1, z_2) = v$ .

Let  $r \in \mathbb{F}^{\log(\alpha k)}$  be the random vector chosen by the verifier in Step 4. Observe that, by linearity,  $y = Yr = C^t(HX)r = C^t(HXr) = C^t(HXr)$  and that:

$$\hat{x}(z_1) = \widehat{Xr}(z_1) = \langle r, \hat{X}(z_1, *) \rangle = \langle r, g \rangle$$

and so (X, Y, z, v) is a YES instance for the IOPP of Lemma 4.1 and so the verifier accepts.

**Soundness.** Fix a NO instance (X, Y, z, v). We consider several cases:

Case 1: Y is  $(\rho/3 + \eta)$ -far from  $C^t$ . By Theorem 2.8, we have that y = Yr is  $(\rho/3)$ -far from C with all but  $\frac{\eta^{-1}}{|\mathbb{F}|}$  probability. Assume that this is indeed the case. But this means that the input to the IOPP of in Step 5 is a NO instance and so the verifier rejects with all but  $2^{-\lambda}$  probability.

Thus, we may assume that Y is  $(\rho/3)$ -close to a unique codeword  $C^t(U)$ , for some  $U \in \mathbb{F}^{\alpha k \times t}$ . By linearity this means that y = Yr is also  $(\rho/3)$ -close to C(u), for u = Ur.

<span id="page-18-1"></span>One could partially derandomize r by using a suitable small bias generator as shown by Diamond and Posen [DP24b].

{19}------------------------------------------------

<span id="page-19-1"></span>Case 2: X is  $(\gamma/3 + \eta)$ -far from the affine space. Next, suppose that X is  $(\gamma/3 + \eta)$ -far from any M for which  $H \cdot M = U$ . Thus,  $X - H^{-1}(U)$  is  $(\gamma/3 + \eta)$ -far from the linear space  $H^{-1}(\mathbf{0})$ . Note that the latter space is exactly the interaleaving of the code whose parity check is H, and so has distance at least  $(\gamma/3 + \eta)$ . By Theorem 2.8, this means that with all but  $\frac{\eta^{-1}}{|\mathbb{F}|}$  probability, it holds that  $(X - H^{-1}(U)) \cdot r$  is at least  $(\gamma/3)$ -far from  $H^{-1}(\mathbf{0})$ . In other words, x = Xr is at least  $(\gamma/3)$ -far from:

$$H^{-1}(\mathbf{0}) + H^{-1}(U) \cdot r = H^{-1}(\mathbf{0}) + H^{-1}(Ur) = H^{-1}(u),$$

where u = Ur and both equalities follow from linearity.

Hence, the instance  $(x, y, z_1, \langle g, r \rangle)$  on which we run the protocol from 4.1 in Step 5 is a NO instance and the verifier rejects with all but  $2^{-\lambda}$  probability.

Thus, we may assume that X is  $(\gamma/2)$ -close to some M such that HM = U. This implies, in particular, that x = Xr is  $(\gamma/2)$ -close to M = Mr and that HM = u, where u = Ur.

Case 3:  $\hat{g}(z_2) \neq v$ . In this case the verifier immediately rejects in Step 3. Thus, we may assume that  $\hat{g}(z_2) = v$ .

Case 4:  $\hat{M}(z_1, z_2) \neq v$ . In this case, note that  $w = \hat{X}(z_1, *)$  and g are distinct vectors. Thus, with all but  $1/|\mathbb{F}|$  probability, it holds that  $\langle w.r \rangle \neq \langle g, r \rangle$ . Assume that this is indeed the case.

Thus, we know that y is  $\rho/3$  close to C(u) and that x is  $(\gamma/2)$ -close to M such that HM = u but  $\hat{m}(z_1) = \widehat{Mr}(z_1) = \hat{M}(z_1, *) \cdot r = \langle w, r \rangle \neq \langle g, r \rangle$ . But this means that in Step 5 we run the protocol of Lemma 4.1 on a NO instance, and the verifier rejects with all but  $2^{-\lambda}$  probability.

**Complexity.** The query complexity follows from the query complexity of Lemma 4.1. Prover and verifier time also follow directly from the protocol in combination with those of Lemma 4.1.

## <span id="page-19-0"></span>6 Random LDPCs

Gallager [Gal62] (see [Gal63]) introduced random low density parity check codes and analyzed their properties. We start by recalling his result.

The Gallager ensemble of (n, j, k) low-density parity-check (LDPC) codes is defined by a parity-check matrix H of dimension  $m \times n$ , where  $m = n \cdot (j/k)$ , constructed as the vertical concatenation of j submatrices  $H_1, H_2, \ldots, H_j$ , each of size  $(n/k) \times n$ . The first submatrix  $H_1$  is fixed deterministically such that its r-th row contains ones in the columns (r-1)k+1 through rk and zeros elsewhere, ensuring a row weight of k and a column weight of 1 within the strip. For each subsequent strip  $i \in \{2, \ldots, j\}$ , the submatrix  $H_i$  is generated by applying a random column permutation  $\pi_i$  to the columns of  $H_1$ .

**Theorem 6.1** ([Gal63, Theorem 2.4]). The minimal distance of the Gallager ensemble of (n, j, k) codes has minimal distance at least  $\delta$  with all but probability:

$$\frac{k-1}{2n^{j-2}} + o\left(\frac{1}{n^{j-2}}\right) + nC_j(\delta, n) \exp\left[-nB_{j,k}(\delta)\right],$$

{20}------------------------------------------------

<span id="page-20-1"></span>where the functions  $C(\lambda, n)$  and  $B_{jk}(\lambda)$  are defined as:

$$C_{j}(\lambda, n) = \left(2\pi n\lambda(1-\lambda)\right)^{\frac{j-1}{2}} \exp\left(\frac{j-1}{12n\lambda(1-\lambda)}\right)$$
$$B_{j,k}(\lambda) = (j-1)H(\lambda) - \frac{j}{k}\left(\mu(s) + (k-1)\ln 2\right) + js\lambda$$

with

$$H(\lambda) = -\lambda \ln \lambda - (1 - \lambda) \ln(1 - \lambda)$$

and the function  $\mu(s)$  is given by:

$$\mu(s) = \ln\left(2^{-k}\left((1+e^s)^k + (1-e^s)^k\right)\right).$$

and the parameter s can be arbitrary but the optimal choice is the unique solution to the equation  $\mu'(s) = \lambda \cdot k$  (where  $\mu'$  is the derivative of  $\mu$ ).

**Remark 6.2.** To make the error probability small, we first and foremost need to ensure that the parameter  $B_{j,k}$  is positive. If is indeed positive, the last term vanishes and the error becomes dominated by the  $\frac{k-1}{2n^{j-2}}$  factor.

Gallager suggests to expurgate the code, by checking that all weight 2 messages are not mapped to the all-zero syndrome. This eliminates the first error term.

#### <span id="page-20-0"></span>6.1 Concrete Analysis

We analyze Gallager's code for j = 8, k = 64 and distance 0.013.

Exact Expression for the Exponent  $B_{8,64}(0.013)$ . The derivative  $\mu'(s)$  is given by:

$$\mu'(s) = \frac{ke^s \left[ (1+e^s)^{k-1} - (1-e^s)^{k-1} \right]}{(1+e^s)^k + (1-e^s)^k}.$$

We need to find s such that  $\mu'(s) = \frac{0.01}{64}$ . Solving numerically, using a script that is available [GNR26] we find that the solution to the equation  $\mu'(s) = \lambda \cdot k$  is  $s \approx -4.087290$ . Plugging that back into the definition of B, we get that  $B \approx 0.000213$  is positive and so the exponent vanishes.

**Remark 6.3.** We remark that the correctness of the numerical estimation for s is not an assumption for our result. This is because Gallager's result holds for any value of s and we use the numerical analysis to derive a (close to optimal) choice for s.

Exact Expression for the Coefficient  $C_8(0.01, n)$ . Substituting the parameters into the coefficient function:

$$C_8(0.013, n) = (2\pi n(0.013)(0.987))^{3.5} \exp\left(\frac{7}{12n(0.013)(0.987)}\right) = O(n^{3.5}).$$

**Conclusion.** By Theorem 6.1, and using the analysis above, the probability that the code fails to have distance at least 0.013 is at most  $O(1/n^6)$ .

## <span id="page-21-4"></span><span id="page-21-0"></span>6.2 Theoretical Analysis: Proof of Theorem 6.4

We show that Gallager's result implies the following:

**Theorem 6.4.** For any  $\alpha > 0$  there exists a constant  $\gamma > 0$  such that a random matrix from the  $(n, 3, \alpha n)$  Gallager ensemble is a  $(n, \alpha n, \gamma)$ -parity check matrix with all but O(1/n) probability.

Let  $\alpha > 0$ . To prove Theorem 6.4 we instantiate Theorem 6.1 using j = 3 and  $k = 3/\alpha$ . to bound the error probability we need to bound the error terms  $B_{3,k}$  and  $C_3$ .

**Bounding**  $B_{3,k}$ . Recall that s in Theorem 6.1 can be set arbitrarily and so  $c = \frac{3}{k} \left( \mu(s) + (k-1) \ln 2 \right)$  and s are fixed constants independent of  $\delta$ . Thus, it suffices to find  $\delta > 0$  such that

$$2H(\delta)3s\delta > c$$
.

Such a  $\delta$  exists since the entropy function is asymptotically larger than any fixed linear function sufficiently close to 0.

**Bounding**  $C_3$ . Recall that  $C_3(\delta, n) = (2\pi n\delta \cdot (1 - \delta) \cdot \exp(\frac{1}{6n\delta \cdot (1 - \delta)})$ . For any fixed  $\delta > 0$  this term is O(n).

Conclusion. Overall, we get that the error probability from Theorem 6.1 is

$$\frac{k-1}{2n} + o(1/n) + O(n^2 \cdot \exp^{-nB_{j,k}(\delta)}) = O(1/n).$$

## <span id="page-21-2"></span>7.1 A Fast Code: Proof of Theorem 1.2

To prove Theorem 1.2, for a given desired rate parameter r > 0 and  $\varepsilon > 0$ , we need show that there exists  $\delta(r,\varepsilon) > 0$  and a family of codes that are computable by circuits of size  $(3+\varepsilon) \cdot k$ , such that a random code from the family has distance  $\delta$  with all but O(1/n) probability.

Let C be a linear-time error correcting code with constant rate  $\rho > 0$ , constant distance  $\delta > 0$ , and let c be a constant such that its encoding can be done in time  $c \cdot k$  (e.g., the linear-time encodable code due to [Spi96]). Instantiating the sketched code using C and the parity check matrix distribution of Theorem 6.4, using  $\alpha = \min\left(\varepsilon, \frac{1}{1+r}\right) \cdot c^{-1}$  we get a code  $C_H : \{0, 1\}^k \to \{0, 1\}^{(1+\alpha/\rho)}k$  that is encodable in time  $3k + c \cdot (\alpha k) \leq (3+\varepsilon)k$ . Note that  $c \geq 1/\rho$  and  $\alpha \leq \frac{1}{1+r}$  and so the rate of the code is at least r.

By the above analysis, the code has distance  $\frac{\min(\gamma, \delta \cdot (\alpha/\rho))}{1 + (\alpha/\rho)}$ , which is a positive constant.

The benchmarks were performed using an Apple Macbook Pro laptop with an M1 processor and 64 GBs of memory. All implementations incorporate SIMD and multi-threading. The results are reported in [Table 1.](#page-24-4)

| Brakedown          | 14.9            | 0               | 35MB       |
