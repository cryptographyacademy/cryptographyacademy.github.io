## Crossing with Confidence: Formal Analysis and Model Checking of Blockchain Bridges

Pyrros Chaidos<sup>∗</sup> , Pooya Farshim† , Denis Firsov‡ , Dimitar Jetchev†∗∗ Aggelos Kiayias∥¶, Markulf Kohlweiss∥¶, Anca Nitulescu§

Input Output, <sup>∗</sup>Greece, †Switzerland, ‡Estonia, ¶UK, §France <sup>∥</sup> University of Edinburgh, UK ∗∗ Microsoft Azure AI Platform, US

*Abstract*—We develop formal code-based security definitions for blockchain bridges and apply them to several bridge architectures deployed in practice. We derive both traditional pen-and-paper proofs and on the other, automated guarantees against bounded counterexamples. The latter is achieved via bounded model checking of our formally specified properties, implemented in Quint, a specification language and model checker closely related to TLA<sup>+</sup>.

Our definitions are expressed in a precise, code-based variant of the Universal Composition (UC) framework. This enables the modular use of hybrid functionalities—even for property-based definitions—and is essential for bounded model checking, where underlying primitives must be idealized.

Accordingly, we idealize and model-check all building blocks used in our protocols. Notably, we formulate a novel UC ideal functionality for Advanced Threshold Signatures (ATS) and modelcheck it for attacks to ensure its robustness.

*Index Terms*—Formal Security, Universal Composition, Model Checking, Blockchain Bridges, Advanced Threshold Signatures.

### I. INTRODUCTION

Major ledger networks, despite their combined financial worth of \$3T,[1](#page-0-0) currently lack native interoperability, relying instead on a variety of blockchain bridges. A blockchain bridge, is a protocol that allows the transfer of information, such as ownership of digital assets or smart contract state, between two distributed ledgers.

Several high-profile blockchain bridges rely on arrangements where a small fixed committee uses a threshold signature scheme to attest to the state of the sending chain. This strongly ties the security of these bridges to trusting the committees and their key management practices, and consequently has led to numerous serious security breaches including the Ronin attack (\$625M stolen) [\[43\]](#page-13-0). Moreover, subtle and technically sophisticated attacks on fixed committee bridges, such as the Poly Network attack (\$611M stolen) and Solana's Wormhole bridge attack (compromised assets of total value \$325M), highlight the need for security to carry through from design to implementation even when trust assumptions hold.

Motivated by the financial value locked in blockchain bridges and the resulting societal risks, we initiate a formal treatment of security for blockchain bridges. Our goals are twofold:

- Existing works on blockchain bridges typically focus on protocol constructions, and often treat security and protocol analysis informally. Our first goal is to complement these lines of work by defining explicit syntax and security models within the UC framework, a sufficiently general model of computation. This brings about a clear and shared understanding of security requirements needed to make the traditional pen-and-paper provable security approach meaningful. Indeed, we carry out such proofs for a number of bridge architectures deployed in practice: direct observation, fixed committee, native consensus certificate, ZK, and advanced threshold signature (ATS) bridges.
- We specify bridge protocols and security properties in a code-based language that is sufficiently precise to render protocols amenable to both easier implementation and analysis. To evidence this, we use Quint [\[44\]](#page-13-1), a userfriendly language for specifying and model-checking distributed protocols, to automatically analyze protocols and their idealized building blocks (referred to as hybrid functionalities in UC) for logical flaws. This significantly strengthens our trust in the robustness of our definitions, protocol specifications, and proofs. A conceptual insight that emerges is the need for perfectly secure building blocks, as model checkers typically cannot (easily) handle probabilistic or reductionist arguments.

We stress that our pen-and-paper proofs play a central role in validating the correctness and security of the core protocol design, providing conceptual clarity and theoretical assurance. Complementing this, the UC model checking framework we develop in Quint validates these theoretical efforts by providing automated, systematic evidence regarding the sanity and consistency of the formal definitions, protocols, and building blocks. Furthermore, this framework allows for the efficient exploration of protocols and is particularly effective at identifying logical flaws early in the design phase. Together, these two approaches reinforce one another: hand-written proofs guide sound protocol design, while model checking provides scalable, automated validation of the specification.

#### *A. Framework and security definitions*

To address the gap in formal definitions of security for bridges, we adopt a code-based language akin to that used in stand-alone (aka. game-based) security models [\[6\]](#page-12-0). However, in the setting of distributed protocols, setup assumptions, such as the type of PKI or the underlying network model, are of greater importance. This motivated us to employ the Universal Composability (UC) framework and extend it with a similar code-based approach that is, however, better suited for the distributed setting. (We give an overview in Section [II-B.](#page-3-0))

Although we operate within the UC framework, we nevertheless formulate our security definitions as trace *properties*. In other words, in our setting, an environment (which is an interactive distinguisher) tries to elicit an execution trace that violates a specific property. There are two main benefits to this approach: first, security properties are easier (often substantially so) to formulate. Second, the specified properties are compatible with model checking: a model checker searches for counterexamples for specific properties. We also retain the composition guarantees of UC when it comes to instantiating idealized building blocks.

We formulate bridge protocols as a collection of interfaces for generating bridge-maintenance keys (e.g., signature keys) and a root-of-trust data (e.g., the genesis blocks of respective chains, and initial public keys), as well as the core bridge maintenance protocol which submits transactions to the ledgers.

We introduce two new interfaces, Certify and the matching Verify. This abstraction allows us to shift our focus from the transfer of tokens or messages (which is commonly viewed as "the bridge") and instead study general types of statements that can be verified by smart contracts on a receiving ledger about the state of a sending chain. (Statements being about chains facilitates chain introspection.) More concretely, we split the design of bridges into two parts: a protocol for transferring (compressed) state—aka. finalized block headers to a receiving ledger, and a protocol for certifying and verifying properties of the sending chain using headers.

Relative to this syntax, we identify two core security properties for bridges: *bridge soundness* and *bridge liveness*. Intuitively, under certain trust assumptions, bridge soundness guarantees that if a "property certificate" for a statement about the sending ledger verifies on the receiving ledger, then the property indeed holds on the sending ledger with respect to all honest parties' ledgers. Bridge liveness, on the other hand, means that a true property of a sending ledger can eventually be proven on the receiving ledger. These properties roughly correspond to the functional and security goals informally stated for the bridge construction in [\[51\]](#page-13-2). Our effort consists of stating these properties in a more rigorous language suitable for proofs and model checking and generalizing them to cover a variety of bridge architectures. Although we focus on liveness and soundness, other notions of security can be formulated

using the approach we put forth here. In particular, applications built on top of bridges can be analyzed and model checked, as we do here. To support this modularity, we make the Verify interface callable by the environment.

### *B. Building bridges, an overview*

In this work, we provide a comprehensive treatment of security for a number of bridge architectures. We begin by describing two simple approaches for building bridges—direct observation and fixed committees—before discussing rotatingcommittee and ZK bridges, as well as bridges based on advanced threshold signatures (ATS), which is our main focus.

Two simple bridge architectures. In the *direct observation* (DO) approach, to prove that some statement holds for the sending ledger, the nodes of the receiving ledger directly observe the sending ledger and verify the statement. A slight refinement of this idea (which is used in other bridges below) requires the receiving ledger parties to only store headers that serve as (correctly observed) fingerprints of the sending ledger. Later, Certify (run by the nodes of the sending ledger) generates a proof of the statement with respect to a header value, which can be transferred off-chain to the nodes of the receiving chain, and verified against the recorded header. The main drawback of DO bridges is their high verification and storage overhead: receiving-ledger nodes must continuously track the sending ledger and validate its consensus rules. This significantly increases complexity and limits its scalability.

An alternative design for bridges relies on *fixed committees*, where a predefined set of parties attests to the validity of statements about the sending chain to a receiving ledger. While fixed-committee (FC) bridges are more efficient—yielding, for example, compact consensus certificates and minimal on-chain verification costs—they introduce a strong trust assumption: security now depends on the honesty and availability of a fixed subset of participants, making such bridges fragile and vulnerable to corruption, cartels, or long-term key compromise. We explore the security of the DO and FC bridges in Appendices [H-A](#page-27-0) and [H-C,](#page-34-0) respectively.

The two simple bridge architectures have complementary strengths. The Direct-Observation (DO) bridge is *trustless*, while the Fixed-Committee (FC) bridge has *compact consensus certificates* that are validated by the receiving ledger. We stress that consensus certificates allow validating the headers of sending chains on the receiving ledger, while property certificates validate *statements* against already validated headers.

Architectures using rotating committees. To overcome the limitations of these approaches, we consider an architecture that enables trade-offs between the efficiency of committeebased designs and the stronger security guarantees enjoyed by the direct-observation approach.

The core insight to avoid the downsides of fixed committees is to employ a *stake-based committee-selection* mechanism, which is often implicit and readily available in proof-of-stake consensus protocols. One then tasks (a threshold number of) the members of the committee to sign sufficient information to

establish the bridge. In particular, one needs to sign the header of the sending chain and enough information to determine the next committee to be used. Due to the ad-hoc nature of the committees and the desire to keep certificates small, Gazi, ˇ Kiayias and Zindros (GKZ) [\[27\]](#page-13-3) designed a dedicated signature scheme for building bridges, namely a threshold signature with signers picked from an ad-hoc committee. They refer to such a scheme as an *Ad-hoc Threshold Multi-Signature*. Due to their utility for building blockchain bridges and oracles, such signatures have recently received increased research attention under the guise of *Multiverse Threshold Signatures* (MTS) [\[4\]](#page-12-1) and *Silent-Setup Threshold Signatures* (STS) [\[25\]](#page-13-4). Architectures using traditional signatures and ZK. In blockchains where the consensus mechanism relies on iterated Byzantine Fault-Tolerant (BFT) protocols [\[18\]](#page-12-2), such as Tendermint [\[46\]](#page-13-5) or HotStuff [\[52\]](#page-13-6), quorums of validators create native *consensus certificates* that can be directly verified. For example, Cosmos uses Tendermint, and Ethereum supports both heavyweight consensus certificates through its beacon chain and a more light-weight variant through its light-client architecture. Such chains do not require a full external bridge protocol. We briefly explain what bridges following such a design look like, and how they can be captured in our definitional framework.

*Consensus-certificate bridge.* These bridges essentially transfer *native* consensus certificates on the state of the sending chain for verification on a receiving ledger. Thus, by relying on this built-in consensus mechanism, such bridges minimize trust assumptions and reduce attack surface. A number of deployed bridges (e.g., Rainbow [\[39\]](#page-13-7) and Polygon PoS [\[41\]](#page-13-8)) follow this blueprint. To give a complete landscape for bridges, we cover the formalization of this bridge in Appendix [H-B.](#page-31-0) We formulate a novel notion of secure and live consensus certificate as a (chain of) artifacts that can be extracted from a given chain. We then use this definition to define a simple native consensus-certificate bridge as outlined above.

*Compressing consensus certificates using SNARKs.* Native consensus certificates can be large. Rather than re-implementing the consensus logic of the sending ledger in the receiving ledger (which would be complex and expensive), the zkBridge construction of Xie at al. [\[51\]](#page-13-2) generates a succinct proof of *light-client execution*: the receiving ledger maintains a compact representation of the sending chain's state, typically a finalized block header. To advance this state, the bridge must verify on the receiving side that the sending chain has correctly progressed according to the light-client protocol this corresponds to succinctly proving and verifying (and thus compressing) a native consensus certificate.

We formalize Xie et al.'s work in our framework. Towards this, we formalize light-client protocols by restricting liveness and soundness of bridge definitions to an isolated "client-local" receiving ledger. One then establishes a bridge by running the client-local ledger in a smart contract on the receiving ledger. We develop these ideas in full detail in Appendix [I-D.](#page-40-0)

A key limitation of using consensus certificates is that many blockchains do not natively support them. Furthermore, interoperability poses other challenges: the receiving ledger must be capable of interpreting and verifying the certificate format and validation logic of the sending ledger, which often requires custom integration. This is an issue when bridging chains with different cryptographic primitives in their design, e.g., chains supporting different elliptic curve arithmetic or different signature schemes. Compression via SNARKs can help with interoperability but requires an expensive prover on the sending side. For example, Xie et al. use a two-layered deVirgo+Groth16 prover for balancing proof generation and verification efficiency.

#### <span id="page-2-0"></span>*C. An architecture based on Advanced Threshold Signatures*

Our ATS bridge considers bridge committees that are regularly rotated, analogous to block-production committees in ledger constructions (e.g., beacon committees in Ethereum, or slot leaders for Cardano). We follow GKZ [\[27\]](#page-13-3) and consider bridges where a secure committee is selected from the sending ledger, and then required to participate in the signing of header values via an Advanced Threshold Signature (ATS) scheme. An ATS is similar to threshold signature schemes in that only a threshold number of valid signatures need to be present, thus avoiding the need for all signers to be online. It differs from them in that it does not require a key-generation protocol for distributing key shares; instead each potential signer generates their own signing key. (Recently this has also been referred to as a Silent-Setup [\[25\]](#page-13-4).)

Our ATS bridge follows GKZ, but is streamlined in several aspects, and presented within the UC execution framework. Specifically, we assume that the ledger offers (or can be extended with) a committee-selection procedure that at the beginning of each epoch returns a sufficiently large committee satisfying a threshold honesty property. Each committee member signs the block header of an agreed prefix of the sending chain. An aggregator collects and aggregates the signatures into an aggregate threshold signature that it submits to a contract on the receiving ledger for verification.

Our protocol, being expressed in the UC framework relative to idealized building blocks, accurately accounts for network delays (more precisely we operate in the ∆-delayed synchronous setting), which influence the propagation of protocol messages, including updated aggregate keys after committee rotation. Bridge soundness and liveness are summarized informally in Theorem [1,](#page-11-0) and formally established in Theorems [10](#page-37-0) and [11,](#page-39-0) Appendix [I-C.](#page-37-1)

The ATS bridge abstracts features of rotating committee constructions such as Chainflip [\[19\]](#page-12-3) and Thorchain [\[48\]](#page-13-9), where the committee signs headers using a threshold signature scheme.

#### *D. Security analyses*

Proofs. In the appendices we carry out extensive pen-andpaper proofs of security for the five bridges above (advanced threshold signature (ATS), direct observation, fixed committee, consensus certificate, and ZK bridge). For each bridge, our theorems explicitly state the trust assumptions that are needed for liveness and soundness. We formulate our theorems with respect to ledger properties (as opposed to an ideal ledger functionality), as this results in *stronger* overall statements. Indeed, ideal ledgers can be shown—and have been model checked—to satisfy the ledger properties that we rely on. Our assumptions, by virtue of being weaker, allow our theorems to be applied to a potentially larger class of blockchains.

The proofs of our theorems are structured as sequences of games, similarly to game-based security analyses. This clearly isolates the assumptions (cryptographic, ledger, and trust) that the proofs rely on, and results in proofs that are easier to verify. It also allows us to derive concrete overall advantage bounds.

Model checking. We complement our theoretical analysis by presenting experimental evidence towards the security and correctness of our protocols and definitions. To this end, we carry out a systematic application of (bounded) model checking, implemented in the Quint specification language [\[44\]](#page-13-1), to the ATS bridge and all its underlying building blocks. These include, among others, checking network diffusion, zeroknowledge, ledgers, and the ATS primitive.

This effort is motivated in part by UC's subtle operational semantics, which often leads to complex ideal functionalities and protocols. Specifically, in UC, when simulators are called they do not need to respond immediately and can yield control to the environment. The latter can, in turn, choose to re-enter a functionality at a different point. Moreover, simulators can resume execution at *any* point where they were called. (This is how UC models concurrency in a single-threaded manner.) This nonlinear flow often hides concurrency bugs that are easily missed by manual inspection. A major effort in Quint consists of creating scaffolding to support UC's control flow.

Indeed, we found bugs/flaws in almost all functionalities that we model checked, even seemingly simple ones. For example, we identify a "flip-flop" bug in a seemingly innocuous definition of signature ideal functionality (see F-[Sig](#page-21-0) in Appendix [C](#page-20-0) and discussions in Section [V\)](#page-11-1), where a specific sequence of yields to and resumes by the simulator result in the system returning inconsistent verification results. As a second example, we designed our ATS ideal functionality *in tandem* with model checking. In this process, multiple correctness and security bugs were discovered (sometimes even when we were convinced that the functionality was "obviously correct"). The result is functionality F-[ATS](#page-37-2) (Appendix [I-A\)](#page-36-0), which despite its complexity has been thoroughly checked for a variety of trace and internal state properties. We note that precise code-based definitions were critical in this process.

Model checking relies on building blocks that are *ideally* secure, and thus matches UC's approach to security well. Indeed, model checking is typically non-probabilistic, meaning

that Quint tries to analyze all possible inputs and responses from pre-defined bounded spaces; even a single set of inputs and responses that violate a given property is not tolerated regardless of probabilities with which this case would arise during the execution.

The full Quint code, which comprises approximately 10K loc and is attached as auxiliary material, and in Section [V](#page-11-1) we provide an overview. Our work demonstrates that model checking can be a viable method to complement pen-and-paper proofs, even for fairly complex protocols in UC.

#### *A. Basic notation*

Given a tuple t = (a, b, ..), we use t.a to denote the acoordinate of t. For a list L we denote its i-th entry by L[i]. We assume throughout that ⊥ is the "none" option type and that □ denotes a value that is not set.

We let prefix(L, ℓ) denote the function that truncates a list L to length ℓ. We define isPrefix(L1, L2) := (L<sup>1</sup> = prefix(L2, |L1|)). Since isPrefix forms a partial order (it is reflexive, antisymmetric, and transitive), we denote it by ⪯ for succinctness.

We use the following font faces throughout: inputs and vectors for data, Algorithms for cryptographic schemes, methods of contracts, Interfaces of protocols and functionalities, Oracles of security games, and FINALIZATION procedures of games and property-based notions. All algorithms are randomized unless stated otherwise. We denote the (internal) state of an algorithm by st, written as Alg(input;st). We denote the empty string by ε. Unless stated otherwise, lists and sets are initialized to empty, integers to 0, and boolean variables to false.

For stateful algorithms, we need to store various data and compute on them as an execution unfolds. We adopt the following notation: if the state of an algorithm is st and we need to store some value under field x in st, we write st.x ← x. We also apply this convention to smart contracts and variables in their memories. When an input or an output of an algorithm is not used we replace it with " ".

#### <span id="page-3-0"></span>*B. The UC framework*

We rely on the Universal Composability (UC) framework [\[13\]](#page-12-4), [\[14\]](#page-12-5) as an execution model for our protocols. In this framework, distributed cryptographic protocols are modeled as sets of interactive Turing machines. Instead of an absolute statement of security (e.g., that an adversary cannot guess a bit), security is formulated in *relative* terms: for any adversary against a real protocol π there exists an ideal adversary (aka. a simulator) against an ideal counterpart ϕ so that no interactive distinguisher (aka. the environment) can distinguish the two. If this is the case, we say π UC-realizes ϕ.

This approach to security enables modular design: a (higherlevel) protocol ρ can be designed with respect to a simpler ideal process ϕ, and then ϕ is replaced by any real process π that UCrealizes ϕ. This is due to the fact that the effects of the higherlevel protocol ρ can be absorbed into the environment. (This

{4}------------------------------------------------

is essentially the UC composition theorem.) This replacement process, in turn, can be extended to multiple ideal building blocks, and enables a modular treatment of security for complex protocols. More details about the UC framework can be found in Appendix [B.](#page-14-0)

We specify protocols and ideal functionalities in a precise code-based language. This results in specifications that are ready to be model-checked in Quint. In more detail, we write protocols as collections of interfaces P.Interface(x) for each party P in some set P, which is a parameter of the protocol. In our specifications we use a simple programming language with basic data structures and operations (integers, lists, sets, etc.). We also allow *passing the execution token* to adversaries/simulators; this allows for (single-threaded) concurrency. For compactness, we adapt the convention that upon the first call to a functionality, a special Init() procedure is executed once.

Adversaries can *corrupt* a machine when instructed to do so by the environment. As a result, the adversary fully takes over the machine: (i) all inputs and outputs to the machine are passed to the adversary, and (ii) the adversary can replace the outputs before they are returned. With *static* corruptions, the adversary corrupts machines at the onset before any other interaction. As usual, we assume the real-world adversary is dummy (i.e., absorbed into the environment).

Our protocols operate in a ∆-delayed synchronous model, following standard approaches [\[2\]](#page-12-6), [\[5\]](#page-12-7), [\[32\]](#page-13-10). Specifically, we use a shared *clock functionality* G-[Clock](#page-19-0), which initializes a global counter, shared across all parties, to zero. G-[Clock](#page-19-0) increments the counter as soon as all machines registered with the G-[Clock](#page-19-0) have expressed their willingness to proceed to the next round via P.OK(). A P.Round() interface provides awareness of the current round r. The pseudo-code and further discussions can be found in Appendix [C.](#page-19-1)

We let view[r] denote the set of inputs and outputs in round r. In more detail, when interface P.Interface(x) is invoked in round r, an entry (handle, P.Interface, x) is added to view[r] where handle is a unique identifier for the call. (This is formally needed as there could be multiple calls to the same interface with the same input(s).) When this interface returns an output y in round r ′ entry (handle, y) is added to view[r ′ ]. The set of corrupted parties in round r is denoted by C[r].

#### *C. Property-based UC*

We formulate security of ledgers and bridges by requiring that certain *properties* of the execution hold with overwhelming probability. Properties are predicates over the full execution trace view and party corruption status C (as recorded for all rounds). We define a security property as a finalization procedure SECURITYPROPERTY that outputs a bit indicating an adversarial winning condition is met. It is called exactly once by the environment at the end of the execution. We write winning conditions by negating the property that is expected to hold for a secure construction. For example, collision resistance CR(x, x′ ) is written as check x ̸= x ′ ; return ¬(Hash(x) ̸= Hash(x ′ )).

Compared to standard game-based formulations, there are three noteworthy benefits of this approach: *First*, ideal building blocks that underlie a security game can be replaced by protocols that UC-realize them (as per UC composition theorem). *Second*, property-based UC allows one to make explicit various assumptions that are tacitly in place in traditional game-based modeling. These include underlying network assumptions, setup assumptions (such as a CRS or a PKI), and access to a shared clock. In game-based definitions these are typically handled by the challenger, often in a less principled way. *Third*, idealizing all cryptographic building blocks of a protocol makes it amenable to model checking.

#### *D. Idealized building blocks*

Besides G-[Clock](#page-19-0), we also use a network *diffusion functionality* F-[Diffuse](#page-20-1) that allows parties to diffuse a message among all parties connected to it with some delay [\[3\]](#page-12-8), [\[22\]](#page-13-11). When a party fetches messages from the diffusion network, all messages that were sent at least ∆ rounds in the past (together with some possibly more recent messages) are delivered to the party. During this process the adversary can *inject* messages of its choice. F-[Diffuse](#page-20-1) formalizes this resource and its code is listed under Functionality [6](#page-20-1) in Appendix [C.](#page-19-0)

We also use a standard F-[CRS](#page-20-2) functionality (see Functionality [7](#page-20-2) in Appendix [C\)](#page-20-1) to allow access to a common reference string generated via a string generation algorithm Setup, as well as F-[Beacon](#page-20-3) (Functionality [8](#page-20-3) in Appendix [C\)](#page-20-2) that allows access to a common fresh randomness in each round. (F-[Beacon](#page-20-3) is used to generate some initial random seed in a genesis block.) Finally, functionality F-[PKI](#page-20-0) idealizes a PKI and allows parties to register their public keys, so that later any party can authentically retrieve them; see Functionality [9](#page-20-0) in Appendix [C.](#page-20-3)

We rely on (succinct) non-interactive proofs of arguments [SNARK](#page-22-0) := (Setup, Prove, Verify) and collision resistant hash functions Hash. The syntax and security of these primitives are recalled in Appendix [D.](#page-21-1)

*Distributed ledger scheme*. A distributed ledger scheme (DLS) DLS consists of the following PPT algorithms.

- Setup(λ) → pp: The setup algorithm, on input the security parameter λ, outputs the common parameters pp. This algorithm is subsequently used to parametrize F-[CRS](#page-20-2).
- KGen(pp) → (sk, vk): The maintenance key-generation algorithm, on input the public parameters pp, outputs a key pair used for ledger maintenance (e.g., maintaining a blockchain containing the ledger). This algorithm is typically used in stake-based protocols; to model proofof-work-type ledgers, we allow KGen to be empty.
- Genesis(pp, vk, w, nonce) → G: The deterministic genesis block generation algorithm, on input the public parameters pp, a list of verification keys vk (for parties involved in maintaining a chain), corresponding stakes w, and a nonce nonce (generated via a randomness beacon), outputs a genesis block G. The nonce is used to model probabilistic Genesis algorithms that, for example, thwart grinding attacks. For proof-of-work protocols w is not needed.
- Val(G, L,tx) → b: The transaction validity predicate decides the validity of a given transaction tx wrt. a given ledger L, and genesis block G.
- BCVal(r, m,tx;st) → (m;st): The main maintenance algorithm is a stateful algorithm having subroutine access to Val that, on input a round number r, a list of incoming

network messages m, a transaction tx, and implicit state st, outputs a message m (to be sent out on the network) and an updated state st. We model parties that are silent in a certain round via message ⊥.

Chain(st) → C: The chain extraction algorithm, on input a state st, outputs a (public) chain C. A chain C is a list of blocks (where a block is a bit string of some given maximal length). Blocks, in addition to transactions, may also contain additional data such as headers, proofs of work, and consensus certificates.

Ledger(C) → L: The ledger extraction algorithm, on input a chain C, outputs a list of transactions as the ledger L. Typically, this entails extracting the list of transactions in each block of C, and then "flattening" the result.

getGenesis(C): Given a chain, this algorithm returns the genesis block of C as output by the Genesis.

Header(C) → h: The deterministic header extraction algorithm. On input chain C, it returns some header information h, which may be ⊥. Overloading, we define Header(C, ℓ) := Header(prefix(C, ℓ)), where the output is referred to as the header information at block height ℓ. Valid headers are required to be collision resistant, a property that is formalized in Appendix [E-E.](#page-24-0)

AChain(C) → C ′ : The deterministic agreed-prefix computation algorithm. On input a chain C, it outputs a prefix C ′ ⪯ C. The interpretation is that all honest parties share C ′ as a prefix (and are aware of it). We also define apl(C) := |AChain(C)|.

Trust assumptions. Following [28], we formulate our security properties with respect to assumptions explicitly checked by the finalization procedure (as opposed to requiring that adversaries respect them in theorem statements). We formalize assumptions as predicates over a set of parties  $\mathbf{P}$ , a collection  $\mathbf{C}[s]$  of subsets of  $\mathbf{P}$  for  $s \in [r]$  indicating corrupt parties in round s, and a collection of ledger views  $L(\mathsf{P},s)$  for each party  $\mathsf{P} \in \mathbf{P}$ , and  $s \in [r]$ . That is, assumptions take the form

$$\mathbb{A}(r; \mathbf{P}, \mathsf{G}, (\mathbf{C}[s], \mathbb{L}(\mathsf{P}, s))_{\mathsf{P} \in \mathbf{P}, s \in [r]})$$
,

where  $\mathbb A$  is an algorithm that returns a boolean. To ease notation, we will simply write "**check**  $\mathbb A(r)$ " in our finalization procedures, meaning that the procedures return false if the assumption is not met. We say that some assumption  $\mathbb A$  implies assumption  $\mathbb A'$  if this implication holds for all rounds. We further assume that assumptions are monotonic in that  $\forall r. \mathbb A(r+1) \Rightarrow \mathbb A(r)$  (e.g., if a party is honest in some round r+1, it was also honest in the preceding round r).

Ledger security. Employing property-based formulations, we follow the seminal works of Garay, Kiayias, and Leonardos (GKL) [23], Pass, Seeman, and Shelat (PSS) [40], as well as Tzinas, Sridhar, and Zindros (TSZ) [49].

We define ledger security as properties on the execution trace of a (real) UC protocol execution. Our definitions refer to the view of different parties in different rounds to define what an adversarial environment (with a dummy adversary) can and cannot do.

We start with two simple correctness requirements. *Genesis correctness* states that in every round, the genesis block output by P.Genesis matches that returned by getGenesis(C), where C is P's view of the chain. We also require *ledger validity*, which states that in every round, all transactions included in any honest party's view of the ledger, as computed via Ledger(C), are valid.

Ledger safety. We follow [49] in defining the safety property, which intuitively requires that the ledgers are append-only, and that the ledgers of any two honest parties are consistent in the sense that one is always a prefix of the other.

Ledger  $\delta$ -liveness. Liveness requires that if an honest party P submits a transaction at some round r and this transaction remains valid for at least  $\delta$  rounds in the view of all honest

parties that have not already included it in their ledger, then at most  $\delta$  rounds after r any other honest party Q will report the transaction in its local view of the ledger.

Agreed prefix. Our bridge constructions will need to report AChain headers of a sending ledger protocol's chain on a receiving ledger. To make this formal, we define an agreed prefix property which requires that in every round all parties can output an agreed upon prefix of everyone's chain by applying the AChain function to their chain. Moreover, the agreed prefix of the ledger at any round is required to extend the agreed prefix of any sufficiently old ledger. This is reminiscent of the "super-safety" property of [49, Definition 13] stated for chains. A second security property is APGROWTH requiring that the agreed prefix grows with the ledger; that is, any party's local chain (which grows due to liveness) is contained in the agreed prefix after some delay.

The formal definitions for the liveness, and (super) safety properties above are provided in Finalizations 16 and 17 in Appendix E.

Bridge liveness. A live bridge ensures successful verification of a true property of a sending ledger on a receiving ledger. #### B. The ATS ideal functionality

To facilitate model checking, which relies on *ideally secure* building blocks, we define an ideal functionality for ATS schemes. To the best of our knowledge, this functionality is novel; despite its complexity, it has been thoroughly checked for flaws. We outline this functionality below and describe how its design prevents (ideal-world) adversaries from breaking its correctness and unforgeability properties. This functionality is potentially of interest in other contexts involving threshold signatures [4], [25].

Functionality  $\mathcal{F}$ -ATS comprises interfaces that match the algorithms of an ATS scheme, with the exception of setup, which, consistent with UC approaches, is provided by independent idealized building blocks formulated via setup assumptions. The interfaces defer to the simulator in most cases, but they also enforce consistency, security, and correctness by maintaining state across interfaces, sanitizing simulator outputs, and enforcing the scheme's logic. We present its full specification in Functionality 24 of Appendix I-A.

We present the details in Bridge Scheme 1. Here, the bridge contract ATSBridge has two methods. The update method receives arguments of the form (h, avk') supported by aggregate signatures wrt. an existing avk. If verification is successful, avk is updated to avk' and the header is appended to the contract memory. The verify method verifies statements using mem, the contract memory.

Whilst we prove security against static adversaries, our formulation of committee selection is compatible with adaptive ones, albeit, with an implied corruption delay. Effectively, we need the committee member to stay honest from the time of selection, until the bridge contract updates. If we also introduce *key rotation* and secure erasures, this delay can be shortened: keys will rotate when the committee signs, so members need to stay uncorrupted only until that point. Aggregation and updating the bridge contract can take place even after the committee is corrupted.

#### V. MODEL CHECKING IN QUINT

<span id="page-11-1"></span>Bounded model checking is a formal verification technique that systematically searches for traces which violate the specification. By encoding system behavior and security properties into logical formulas, model checking uses SAT/SMT solvers to find counterexamples (bugs) that violate those properties. This methodology is justified by the so-called "small scope hypothesis" [\[30\]](#page-13-17), stating that in (natural) systems, bugs arise within traces that have relatively few objects.

We carry out a systematic model-checking of our UC protocols, which is a novel aspect of our work, and potentially of wider interest. Model-checking such protocols is motivated in part by subtleties in UC's operational semantics. Indeed, the specification of UC functionalities that we use appears deceptively simple: they are often interpreted as simple programming with procedure calls. However, the underlying operational semantics allows *resumptions*. Intuitively, this means that any call to the simulator pauses the execution of the functionality and allows the simulator to hand control back to the environment. The latter can then call the functionality again. Moreover, whenever the simulator is activated again, it has the option of resuming any pending calls from the functionality.

We model check the ATS bridge, including all its underlying building blocks, in Quint [\[44\]](#page-13-1). Quint is a specification language and a model checker with a focus on distributed and concurrent systems. Quint's theoretical foundations are closely related to TLA<sup>+</sup> [\[35\]](#page-13-18), [\[36\]](#page-13-19) and support first-order logic, state-transition systems, and linear temporal logic with actions. However, Quint has a more user-friendly syntax, supports various modes of verification and temporal properties, and is statically typed and effect aware.

By implementing our system in Quint we obtain executable specifications, which can be analyzed in a variety of ways. For example, we can: (i) execute the system for specific adversaries and environments (i.e., perform tests); this is a basic but limited form of guarantee that we can easily obtain; (ii) use Quint simulations to produce a large set of traces; (iii) convert Quint code to TLA<sup>+</sup> to use its toolset for further analysis or to the Apalache SMT solver [\[33\]](#page-13-20), [\[34\]](#page-13-21) for formal verification. We use Quint 0.29.1 (with Rust backend) and Apalache 0.51.1.

In many cases, we uncovered bugs, some arguably subtle, which lead to robust definitions of various standard ideal functionalities. It is worth noting that, while writing the Quint code, many bugs were caught simply by static type-checking.

For a more subtle bug, consider the example of the wellknown F-[Sig](#page-21-0) functionality, included in Appendix [C](#page-20-0) (cf. Ver. Bad and Ver. Good). This definition is only about 20 loc. It is fairly simple to convince oneself that Ver. B is indeed a good and fairly detailed idealization of digital signatures. However, when we implemented it in Quint, we immediately ran into an issue due to the resumption semantics mentioned above. Specifically, we discovered that this specification suffers from a "flip-flop" bug that allows the functionality to respond to verify requests inconsistently (i.e., the same request produced different responses). Quint reports a trace for the following attack: 1) the environment calls P.Verify(vk, m, σ); 2) the simulator yields the execution to the environment; 3) the environment calls P.Gen and the simulator allows it to run until the end; 4) the environment calls P.Sign(m), which the simulator allows to run until the end; 5) the environment calls P.Verify(vk, m, σ), where the simulator responds with 1; 6) the environment activates the simulator which in turn resumes the initially suspended execution of P.Verify(vk, m, σ) producing output 0. We found similar deficiencies in nearly all initial UC specifications that we had.

Our model checking relies on *idealized* building blocks, and thus matches UC's approach to security well. Indeed, model checking is typically non-probabilistic, meaning that Quint enumerates all possible inputs and responses from pre-defined bounded spaces. Even a *single* set of inputs and responses that violate a given property is not tolerated. Accordingly, we formulate idealized versions of the ATS scheme (Appendix [I-A\)](#page-36-0), and the ledger including committee selection and agreed-prefix length (Appendix [F\)](#page-25-1).

Model checking complements pen-and-paper proofs and substantially increases our confidence in the specifications of the overall bridge design and its building blocks. It also serves as a first (relatively low-cost) step towards the full formal verification of proofs and properties. This effort was made possible, in part, through unambiguous and falsifiable codebased specifications of various system components. Our work demonstrates the practical applicability of model checking to other complex UC protocols, whose specifications may rely on multiple building blocks. We hope this methodology helps create a robust library of precisely specified and well-tested ideal functionalities and (realistic) protocols.

### APPENDIX A BRIDGE-LIKE CONSTRUCTIONS

The terms blockchain and cross-chain bridge are used inconsistently across the literature, often conflated with a broad class of cross-chain applications that rely on some form of cross-chain communication (CCC) [\[53\]](#page-13-22).

We distinguish general-purpose bridges from such notions. In particular, we treat bridges as protocols that enable *generalpurpose* state verification or transaction synchronization between independent ledgers, whereas many cross-chain applications such as wrapped assets or shared liquidity pools merely operate atop such mechanisms, or implement them in a restricted way.

*HTLC*. A typical example of a cross-chain application that we do not consider a cross-chain bridge is hashed timelock contracts (HTLC) or adaptor signature-based atomic asset swaps [\[47\]](#page-13-23), [\[7\]](#page-12-9). Such swaps, for instance, do not guarantee liveness, as the counter party can simply ignore the lock. Thus the statement that holds at the end of the protocol execution is of the form: either one can get hold of an asset on the sending ledger, or one gets her own asset back on the receiving ledger (and vice versa). We consider support of such limited application specific statements to be too limited for a general purpose bridge.

*Simple payment verification*. In the context of cross-chain communication and lightweight verification, it is important to consider simplified payment verification (SPV) clients, particularly as used in Bitcoin. SPV clients verify transaction inclusion based only on block headers, without downloading full blocks; as such, they are not DO (Direct Observation) clients in the strict sense. While SPV is not itself a bridge mechanism, it can be used as a building block for bridge construction.

To adapt SPV-style verification into a DO-like bridge framework, several modifications are necessary: (1) the update function must verify that incoming headers extend the existing chain consistently; (2) it must handle chain reorganization or only retain headers buried at least k blocks deep for some k; and (3) Certify and Verify must be specialized to transaction inclusion proofs. While the connection between SPV and DO is not immediately obvious, it becomes clearer when abstracting their behaviors from the perspective of the receiving contract: both systems process externally provided updates and perform lightweight validation. The key difference lies in the validation predicate xcVal–SPV assumes minimal checks (e.g., Proof-of-Work validity), whereas DO requires alignment with the full view of the sending chain. It remains an open question whether relaxing xcVal and other DO bridge components can yield a secure and meaningful SPV-style bridge in our formal model.

*Light clients*. Light clients play a foundational role in several bridge designs, especially in scenarios where the receiving ledger must validate consensus or transaction inclusion without full state replication. The SoK on blockchain light clients [\[20\]](#page-12-10) provides a taxonomy of client types, including SPV clients, which verify inclusion proofs using only block headers. Fly-Client [\[9\]](#page-12-11) improves upon this model by allowing superlight clients to securely verify transactions with logarithmic communication. While SPV is not a bridge per se, it is a critical primitive in the sense that several bridge protocols (especially those compatible with Bitcoin) can be constructed using SPVstyle verification, augmented with chain continuity checks and rollback-aware update functions [\[38\]](#page-13-24).

Both bridge protocols and light clients benefit significantly from ledger finality properties, which reduce ambiguity around reorgs and improve the security and efficiency of cross-chain verification. Finality ensures that once a block is confirmed beyond a certain depth, it is unlikely (or impossible, in the case of strong finality) to be reverted—thus making proofs of inclusion or consensus status more reliable and simpler to verify.

### <span id="page-14-0"></span>APPENDIX B UC, PSEUDO-CODE, AND PROPERTIES

#### <span id="page-14-1"></span>*A. The UC framework*

We formalize our security statements in the Universal Composability (UC) framework of Canetti [\[13\]](#page-12-4), [\[14\]](#page-12-5). The UC framework provides strong security guarantees: protocols that are UC secure can be used securely and in a modular way in a wide variety of contexts. In a nutshell, UC compares the execution of a real protocol in the presence of an adversary to that of another ideal protocol in presence of another adversary (aka. a simulator). If for any adversary against the real protocol there exists an ideal adversary against the ideal protocol so that no interactive distinguisher (aka. the environment) can distinguish the two, the real protocol is said to UC-realize the ideal protocol.

This definition allows designing a higher-level protocol with respect to a simpler ideal protocol, and then replacing the ideal protocol with a real protocol that UC-realizes it. This is essentially due to the UC composition theorem, the proof of which relies on the fact that any higher-level protocol can be seen as being part of the environment. This replacement can be extended to multiple ideal protocols, and enables a modular treatment of security for complex protocols. It is thus particularly suitable for our goals.

While natural, defining a formal execution model that is sufficiently expressive to capture real protocols of interest turns out to be technical. For instance, one needs to delineate boundaries of protocol instances, appropriately model shared resources, consider corrupted parties, and constrain the environment so that the security is both feasible and meaningful.

We give a brief overview of the simpler UC framework for static systems. (Our protocols do not need to dynamically generate functionalities during runtime, and thus we do not need the full power of dynamic UC.) We refer to [\[14,](#page-12-5) Section 2] and [\[1,](#page-12-12) Section 2] for further details on UC for static systems (henceforth, called UC).

*Interactive Turing machines*. The UC framework models protocols as collections of *interactive Turing machines* (machines)

{15}------------------------------------------------

that interact through various tapes. An ITM resembles a Turing machine, having internal work tapes and a state transition function. However, an ITM (in UC) comes with three external tapes: input, subroutine-output and backdoor. The input tape holds inputs from calling machines and the subroutine-output tape contains return values for caller machines. The backdoor tape is used for communication with a special machine called the adversary (see below). To ease readability, we will simply use 'machine' instead of ITM.

*Identities*. UC machines possess unique identities of the form (P,sid = (pid, i)). P is a party identifier (PID) and is interpreted as a 'local cluster' of machines. pid is a 'process identifier' and is different for machines with different codes. For example, it can be set to the (hash of) code of the machine. i is an instance number and is used to indicate independent instances of the same process. We call sid = (pid, i) the session-identifier (SID), which are unique across all machines active in an execution.

Machines specify the destination identities of other machines that they provide inputs or subroutine outputs to. This is done by including the identities on tapes. Machines also include their own identities on tapes so that the destination machine knows the source. Every machine comes with a *communication set* which defines which other machines and in what role it can talk to.

*Protocols*. A protocol π is a *set of well-connected* machines in the following sense: (1) a machine in π can pass inputs to another machine in π if and only if the latter can pass outputs back to the former; (2) a machine in π can pass inputs to some destination identity only if there is a machine in π with that identity. The *external identities* of a protocol are the set of identities outside π where some machine in π provides outputs to. The *main* machines of π are those that have at least one external identity. As sets, we use ∪ to combine protocols into larger protocols, or remove components using \.

*PID-wise connectivity*. In UC one explicitly models network resources. Accordingly, we require that machines can only pass input to or receive outputs from other machines if they have the same PID. This means that the machines of a protocol communicate only 'locally' and use ideal functionalities to send network messages. We make exceptions to this rule when connecting to ideal functionalities, the environment, or the adversary (see below).

*The environment and the adversary*. The environment Z and the adversary A (which is called the *simulator* when run wrt. an ideal protocol to match attacks on a real protocol) are machines which have special PID and SID (e.g., 0 and 1 as their PIDs respectively, and ⊥ for their SIDs). The environment models honest parties in higher-level protocols, including protocols in other sessions; the adversary, on the other hand, models adversarial influence on the protocol, such as generation of cryptographic objects in ideal functionalities or actions of the corrupted parties (see below). The environment and the adversary can freely interact. The adversary can talk to all machines on the backdoor tape. Below we define how Z interacts with machines in a given protocol.

*PID-wise corruptions*. The environment may instruct the adversary to *corrupt* a machine with PID P different from those of the adversary, the environment, and any ideal functionality. Consequently, the contents of the internal work tapes of all machines with PID P are sent to the adversary. Moreover, these machines then act as dummies: they forward all their incoming communication to A, and receive instructions from A as to how to respond. This models PID-wise full Byzantine corruptions. The fact that the environment initiates corruptions ensures that in the real and ideal worlds corruptions are 'commensurate'. Throughout, we let C denote an execution-wide set of corrupt parties in the system.

Establishing security in the fully Byzantine and adaptive model of corruptions is oftentimes challenging. One often resorts to weaker models, e.g., of *static corruptions* whereby the environment instructs the adversary to corrupt machines of its choice at the onset before any other interaction. Another weaker notion of relevance is that of δ-delayed corruptions [\[28\]](#page-13-12) for some delay parameter δ. Here a machine that is instructed to be corrupted by the adversary becomes corrupt (its internal state is leaked and its control is handed over to the adversary) only after δ rounds.

*Ideal functionalities*. An *ideal functionality* models a 'trusted service' that collects inputs and carries out a specific task in a trusted manner. Given an ideal functionality F, the ideal protocol is obtained by attaching a number of dummy parties to F who act as forwarders. This gives the higherlevel protocols (or environment) the impression of interacting with a distributed protocol (rather than a single centralized service). By slight abuse of notation, we will use F for the ideal protocol for F. Ideal functionalities are *incorruptible*, though their guarantees can deteriorate depending on the corruption status of their dummy parties. Formally, we assume the PIDs of ideal functionalities take special forms, which the environment is prohibited from corrupting. Note, however, that the adversary can still interact with ideal functionalities both through the backdoor tape and on behalf of the corrupted dummy parties.

Note that through ideal functionalities, protocols get access to *joint state*. 'Local' functionalities typically have caller-sessionspecific states, while 'global' (or shared) functionalities can share state across all caller sessions. (No state is shared across the sessions of F, be it local or global.)

*Remark*. When specifying ideal functionalities, a core conceptual insight is that the simulator can influence the generation of cryptographic objects whose specific forms do not affect the security guarantees. For example, in a secure signature scheme, it does not matter what particular form the verification keys take, as long as no new verifying signature beyond those legitimately obtained through the ideal functionality, can be generated. This power, in turn, allows ideal functionalities to be realizable by concrete protocols. In UC, one typically abstracts away from secret keys (though on corruption they need to be simulated).

*Environmental restrictions*. The environment can send messages to machines in a protocol π *on behalf of other machines* as 

{16}------------------------------------------------

well as to the adversary. This is required so that  $\mathcal{Z}$  can indeed model higher-level protocols accessing  $\pi$  (and thus enable composition). The environment, however, can only pass inputs to the *main machines* of a protocol  $\pi$  on behalf of the *honest external identities*. (PID-wise connectivity is also required to be respected.) In this case we say that the communication set of  $\mathcal{Z}$  is compatible with  $\pi$ .

*Executions*. The execution of a protocol  $\pi$  with an adversary  $\mathcal{A}$ and an environment  $\mathcal{Z}$  starts by running  $\mathcal{Z}$ , which then sends messages to other machines as specified above. The execution terminates returning a bit, which we denote by  $\mathcal{Z}^{\pi,\mathcal{A}}$ . (This notation follows [6] and is more intuitive and arguably closer to diagrammatic descriptions. In UC lingo this is written as  $\mathrm{EXEC}_{\pi,\mathcal{A},\mathcal{Z}}$ .) In this execution, we assume that a concrete value for the security parameter is hard-coded to all machines. Efficiency. We assume that environments, adversaries, and protocols satisfy an appropriate notion of efficiency so that the entire execution can be efficiently emulated on a single machine. We give a concrete example. If each machine in the execution terminates-that is, halts and no longer sends any messages-after at most t steps over all its invocations, at least a single step is needed to send a message to another machine, and there are no chains of invocations of length more than c (any chain of executions of length at least c+1 has a cycle), the maximum number of steps can be shown to be that corresponding to a depth-c tree of processes each invoking t processes. The total number of steps is  $t^c$ , which is polynomial (in the security parameter) as long as t is polynomial and c is a constant. As a special case, if the total number of machines in a protocol is m, execution is clearly upper bounded by  $t \cdot m$ . Other notions of efficiency exist [12], [14], [29].

#### B. UC emulation and composition

We restate UC emulation and composition theorems, despite not directly employing them in our development. We do, however, rely on global subroutines when stating our propertybased definitions.

UC emulation. We say a protocol  $\pi$  UC-emulates another (typically more idealized) protocol  $\phi$  if for any efficient adversary  $\mathcal A$  there exists an efficient simulator  $\mathcal S$  such that for all environments  $\mathcal Z$  that are compatible with both  $\pi$  and  $\phi$ , the distance  $|\Pr[\mathcal Z^{\pi,\mathcal A}=1]-\Pr[\mathcal Z^{\phi,\mathcal S}=1]|$  is small.

Let  $\rho, \phi$ , and  $\pi$  be protocols. Recall:

- $\phi$  is a subroutine protocol of  $\rho$  if  $\phi \subseteq \rho$  as a set of ITMs.
- $\pi$  is *compatible* with  $\phi$  if there is an identity-preserving bijective correspondence between main machines of  $\pi$  and  $\phi$ , and the entries corresponding to external identities in the communication sets of the main machines are equal.
- If  $\phi$  is a subroutine protocol of  $\rho$ , then  $\pi$  is *identity-compatible* with  $\rho$  and  $\phi$  if no machine in  $\pi$  has the same identity as a machine in  $\rho \setminus \phi$ .

<span id="page-16-0"></span>We let  $\rho^{\phi \to \pi} := (\rho \setminus \phi) \cup \pi$ , i.e., a protocol that is similar to  $\rho$  except that protocol  $\phi$  is replaced by protocol  $\pi$ . Under the above conditions one can show that  $\rho^{\phi \to \pi}$  is indeed a protocol. Canetti [14] presents a composition theorem as follows.

**Theorem 2** (Essentially [14, Theorem 3]). Assume that  $\pi, \phi$ , and  $\rho$  are protocols such that  $\phi \subseteq \rho$ ,  $\pi$  is compatible with  $\phi$ ,  $\pi$  is identity-compatible with  $\rho$  and  $\phi$ , and  $\pi$  UC-emulates  $\phi$ . Then  $\rho^{\phi \to \pi}$  UC-emulates  $\rho$ .

Roughly speaking, the proof relies on the fact that the caller part of  $\rho$ , namely  $\rho \setminus \pi$ , can be absorbed into the environment, so that when this caller part is composed with  $\pi$  or  $\phi$ , the results are indistinguishable as  $\pi$  UC-emulates  $\phi$ . The fact that  $\rho$  and  $\rho^{\phi \to \pi}$  are protocols ensures that any environment against that is compatible with  $\rho$  and  $\rho^{\phi \to \pi}$  when converted to one against  $\pi$  or  $\rho$  with  $\rho$  absorbed, is also compatible with  $\pi$  and  $\rho$ .

#### C. Global functionalities

When designing systems, it is sometimes the case that certain components of the system should be modeled as global (or shared) functionalities. For instance, in our setting, the blockchain is typically a shared infrastructure that is used by different protocols. That is, one typically does *not* consider independent instances of blockchain for different applications. Examples of other shared functionalities include public-key infrastructure, or shared clocks that enable synchrony across different protocols. Thus, unlike local functionalities, global functionalities often have a single instance that may take input from, and provide outputs to multiple instances of (unknown) protocols. As such, the machines in a global functionality may be part of the main machines of a protocol, and thus the environment can directly call them. Thus, such functionalities can be in any state when they first get accessed by the protocol.

Complex frameworks exist to augment the UC framework to capture joint state or globals [15], [16]. The most recent is the UCGS framework of Badertscher et al. [1] that builds on the latest version of (dynamic) UC to model global functionalities as shared subroutines within the (plain) UC framework. This work, in particular, provides a simple and convincing treatment for the static fragment of UC [1, Section 2] that avoids complex frameworks. We give an overview next.

Given a protocol  $\rho$ , the UCGS framework (for static systems) treats its global sub-protocol  $\gamma$  together with other subroutines  $\pi$  as a sub-protocol  $\hat{\pi} := \pi \cup \gamma$ . To only replace  $\pi$  with  $\phi$ , leaving  $\gamma$  untouched, one considers  $\hat{\phi} := \phi \cup \gamma$  and invokes Theorem 2 with respect to  $\hat{\pi}$ ,  $\hat{\phi}$ , and  $\rho$ . Note that the main machines of  $\gamma$  may form a subset of the main machines of  $\pi \cup \gamma$ , which allows  $\gamma$  to be accessed from higher-level protocols.

**Theorem 3** (Composition with global subroutines). *Assume* that  $\pi$ ,  $\phi$ ,  $\gamma$ , and  $\rho$  are protocols. Suppose that  $\phi$ ,  $\gamma \subseteq \rho$ ,  $\pi$  is compatible with  $\phi$ ,  $\pi$  is identity-compatible with  $\rho$  and  $\phi$ , and  $(\pi \cup \gamma)$  *UC-emulates*  $(\phi \cup \gamma)$ . Then  $\rho^{\phi \to \pi}$  *UC-emulates*  $\rho$ .

Let  $\hat{\pi} := \pi \cup \gamma$  and  $\hat{\phi} := \phi \cup \gamma$ . The proof uses the following observations: 1) if  $\pi$  and  $\phi$  are protocols, then so are  $\hat{\pi}$  and  $\hat{\phi}$ ; 2) we have  $\hat{\phi} \subseteq \rho$ ; 3) if  $\pi$  is compatible with  $\phi$ , then  $\hat{\pi}$  is compatible with  $\hat{\phi}$ , and 4) if  $\pi$  is identity-compatible with  $\rho$  and  $\phi$  then  $\hat{\pi}$  is identity-compatible with  $\rho$  and  $\hat{\phi}$ . This allows Theorem 2 to be applied to  $\hat{\phi}$ ,  $\hat{\pi}$ , and  $\phi$ . We can conclude using the fact that  $\rho^{\hat{\phi} \to \hat{\pi}} = \rho^{\phi \to \pi}$ .

#### D. Pseudo-code conventions

The language. The language of Turing machines is too low level for practical purposes. English specifications, on the other hand, can be ambiguous and error prone. We follow the codebased framework of [6, Appendix B] and use a programming language to express functionalities and protocols. This simple language supports basic data types (integers, sets, lists, maps, etc.) and common operations on them. We use  $\perp$  for the option type indicating "none," and  $\square$  to indicate that some entry in a key-value store is "not set." We use ok for the unit type and \_ for unused values. Reserved words with special meanings used for program structure and syntax (if, then, for, return, etc.) are typeset in bold. We do not use while, loops, or goto statements. To help readability, we use various syntactic sugar such as indentation, use of line break instead of ";", subscripts and superscripts, Greek / Unicode characters, quantifiers over finite domains, and iteration "for  $x \in S$ " where x takes values in S in some natural order.

A major difference with conventional (stand-alone) games is that, besides subroutine calls, machines can *pass the execution* to another machine. This can be modeled via the language of *coroutines* [21]. Their use in this work is almost entirely confined to interactions with the simulator; calls to and between functionalities are expressed using subroutine calls/returns.

Interfaces and their connectivity. We express functionalities and protocols as collections of interfaces P.Interface for different parties P and session-ids sid, as well as special  $\mathbb{Z}$ - and Simspecific interfaces. An interface is a sequence of statements (program) in the programming language and corresponds to machines in UC. Parties P are required to always belong to a set of parties  $\mathbf{P}$ , which is a parameter of the functionality. For PID-wise connectivity, we require that the PID of the caller machine (which is another interface) always matches P, i.e., that of the called interface.

Session-ids are required to belong to a session-id set  $\mathbf{SID}$ , which is another functionality parameter. We will mainly consider  $\mathbf{SID}$  sets of two types. First, for local functionalities, which are designed to respond to a single caller, this is a singleton set  $\mathbf{SID} = \{ \text{csid} \}$ . Second, for global functionalities, which typically offer services to any caller process,  $\mathbf{SID}$  is set to the entire session-id space.

For locals, these restrictions amount to P.Interface checking if the caller's (P, sid) matches (P, csid). For globals, only the PID is checked as there are no restrictions on SIDs. (We will indicate this by an "any sid" annotation.)

Functionalities also have a session-id sid\*, which enables independent instances to be created. (In other words, program variables in different instances are indexed by sid\*.) Interfaces inherit this session-id and thus P.Interface's identity is (P, sid\*). This is used when calling other interfaces.

This operation can be set to random sampling or taking the lexicographically first good value for v.

*Real vs. ideal specifications*. Ideal functionalities and real protocols differ in a number of aspects. First, ideal functionalities are aware of the corruption status of parties in their party sets. Second, they can pass the execution to the simulator and may offer simulator-specific interfaces (see, e.g., F-[Diffuse](#page-20-1)). Finally, they may share state across parties.

Well-specified real protocols do not have any of these features (beyond those that are implicit in their underlying idealized building blocks). Moreover, real realization of ideal functionalities do not specify how leakage is computed upon corruptions; rather this is set to contents of work tapes.[6](#page-18-2)

In Appendix [C](#page-19-1) we give several examples of idea functionalities.

#### *E. Properties in UC*

*Views*. We define the environment's view of the protocol view in an execution as follows. When an interface Psid.Interface(x) is called by the environment, an entry (handle, Psid.Interface, x, □) is added to view, where handle is a unique handle/identifier for the call (needed as there could be multiple calls to the same interface with the same inputs). When this interface returns an output y to the environment entry (handle, y) is added to view.

To formalize some of our security properties, we need to talk about views in various rounds. This is a concept that only makes sense in the presence of a clock functionality G-[Clock](#page-19-0). We formalize this by requiring that the environment calls the Psid.Round() interface of G-[Clock](#page-19-0) immediately before each call it makes to other interfaces and immediately after it receives an output. The clock functionality is designed so that the Psid.Round() interface returns the current value of the clock's internal counter. Thus, inspecting this counter is an alternative way to define in which round a call was made or an output was received from. We note, however, that this is formally not a trace property.

*Properties*. An execution *property* is a predicate on the trace, view. Security requires specific properties hold for any environment/adversary. This is done via a special environmentspecific *finalization interface* that outputs a bit based on the execution trace. We write these interfaces in small-caps as SECPROP(..). Finalization interfaces can be called once by the environment at the end of the execution.

We express each SECPROP in code, with the interpretation that they are expected to hold for secure constructions. We thus adopt the convention that the return statement of a finalization procedure *negates* the target property. For instance, collision resistance is written as "check x ̸= x ′ ; return ¬(Hash(x) ̸= Hash(x ′ ))". Here "check C" is sugar for "if ¬C then return 0".

### <span id="page-23-0"></span>*B. Enabling smart contracts*

Recall that our ledger schemes are formulated relative to a validity predicate Val. Our bridge protocols will rely on deploying contracts in order to bridge two ledgers. We present a concrete validity predicate Val that supports smart-contract functionality. At a high level, Val interprets transactions as a piece of contract code and some arguments. It then evaluates the code on the current ledger and the provided arguments. This is done iteratively, starting from the genesis block (which is processed using a dedicated initial contract-state computation procedure initCS). In this execution, the code of a contract can call other deployed contracts and request to transfer funds.

More precisely, a key-value store CS, which we call the *contract state*, contains, for each contract-id cID, a tuple (code, mem, funds) consisting of the contract code, its memory, and funds. Following standard practices, we assume that cID is computed via hashing the code. Predicate Val relies on an updateCS procedure that given a contract state CS and a transaction tx outputs an updated contract state (which may be ⊥ indicating an invalid transaction). The updateCS procedure is applied iteratively to each of the transactions in a given ledger in order to compute the final contract state. Procedure contractState starts this iterative process using initCS, typically by reading off the list of ledger-maintenance keys and initial stakes from the genesis block. We also assume the genesis block encodes standard account contract code for the initial stakeholders, which initCS deploys.

In subsequent invocations, updateCS takes CS and parses a given transaction tx as (cID, method, args; v,sID) comprising a contract ID cID, a method of the contract method, and arguments args, a received-fund amount, and a fund-sender contract-id sID. For top-level contract calls it checks that the received value v is 0 and that the caller contract-id sID is the empty string.[9](#page-23-1) If the specified method is install, updateCS parses args further as code and sets CS[cID] to (code, □, 0). (If a contract is already deployed at cID, the request is ignored.) We note that in this emulation CS is a *local* variable and is computed "from scratch" given a ledger (i.e., a list of transactions) L.

<span id="page-23-1"></span><sup>9</sup>We do not currently model gas costs, minting (besides genesis) or offchain-to-contract transfers.

This emulation contains standard instructions that are applied to the current memory. Additionally, a contract can call a method method′ of some other contract dID on args′ , v′ . The call is implemented in the natural way by applying the updateCS to the current contract state CS and transaction (dID, method′ , args′ ; v ′ , cID). The value v ′ indicates the amount of funds transferred between contracts. updateCS checks if the funds to be transferred are positive and that the sending contract cID has sufficient funds. If so, it goes on to reduce the funds of caller contract cID and increments those of the receiving contract dID by the given value. This design is inspired by Ethereum's approach for transferring funds, though we have simplified it to fit the scope of the paper.

#### *D. Ledger security, an extended discussion*

We define ledger security as properties of the execution trace of a UC protocol. Our definitions depend on the views of different parties at different rounds (e.g., regarding their local chains and ledgers). We define a "helper" function C(P, r) that returns the output of Psid.Chain() in round r from view[r] as long as no Psid.Submit calls are placed after it (and ⊥ otherwise). We recall two simple correctness requirements already stated in the body of this work. Genesis correctness states that in every round, the genesis block returned by Psid.Genesis matches that returned by getGenesis(C), where C is P's view of the chain. We also require ledger validity: no invalid transactions are included in any (honest) party's (local) view of the ledger in any round. See Finalization [17.](#page-26-0)

#### <span id="page-24-0"></span>E. Other ledger properties

Self-persistence. This property requires that the local ledger of any party is append-only in the sense that if an entry is added to a specific position in a party's ledger in some round, it will

persist in that position from that round onwards. See property SELFPERSISTENCE in Finalization 17.

GKL ledger persistence. We formulate two alternative persistence notions that follow those in GKL and PSS. The first property, GKLPERSISTENCE, fully decouples persistence from liveness and requires that a valid transaction that is reported in one party's ledger at a specific position is also reported in any other party's ledgers in the same position, provided the position is not empty  $(\bot)$ . Note that this property does *not* imply self-persistence, as (in a bad ledger) a valid transaction might switch back to  $\bot$ . However, when combined with the APPENDONLY property, self-persistence is implied.

**PSS** second ledger persistence. The property PSSPERSISTENCE is arguably closer to [40, Definition 7.2]. Here, ledger persistence under an assumption A posits that if a valid transaction is reported at some position in an honest party's ledger, it will eventually be reported in this position after a sufficient number of rounds elapses. Thus, this definition has a flavor of liveness mixed into it, though it does not guarantee full liveness since it does not guarantee that a submitted transaction will indeed appear in the party's local ledger. This notion of persistence implies GKLPERSISTENCE, but still it does not imply self-persistence (for essentially the same reason that GKLPERSISTENCE does not).

Collision resistance. This property states that headers form succinct fingerprints for a given chain. Model checking requires ideal building blocks. We present an idealization of a ledger. To simplify, we focus on defining a *ledger* rather than a chain, which is sufficient for our purposes since we focus on ledger properties such as transaction inclusion.

Our ledger functionality  $\mathcal{G}\text{-}\mathrm{Ledger}$  comes with the following interfaces.

- A P<sub>sid</sub>.KGen() interface for the preliminary stage of genesis generation where potentially some keys (for proof-of-stake protocols) are generated.

- All transactions older than  $\Delta_{lat}$  rounds (the latency parameter) are included in the log. Then, for each party, the simulator picks a pointer to a prefix of this log. This pointer is sanitized to be increasing wrt. the party's last pointer, and to ensure a sufficiently recent prefix of the log. (Thus different parties' pointers are within a "neighborhood" of each other.) A slack parameter  $\Delta_{slack}$  determines how far back from the current round the pointer can be, and thus how synchronized parties are. The validity of transactions is decided using the Val function.
- In the  $\mathsf{P}_{\mathsf{sid}}.\mathsf{ComSel}[c,\rho]()$  interface, we ask the simulator to output a committee sanitized to satisfy the properties defined in Finalization 4. See Functionality Extension 19 for the details.
- Similarly, in the  $P_{sid}$ .ALedger $[\Delta_{apl}]$ () interface, we ask the simulator for a pointer, which is sanitized to be at most the minimum of all parties' pointers, and at least lags from this by some delay parameter. See Functionality Extension 20 for the details.
- Interface  $P_{sid}$ .ALedgerHeader() computes a collision-resistant header value for the agreed ledger. Its definition borrows ideas from  $\mathcal{F}$ -Hash. See Functionality Extension 21 for the details.

## APPENDIX H ANALYSES OF BRIDGES

In order to show that a certain statement x holds for the sending ledger L we will rely on a proof system to show that (1) x holds for *some* ledger L, and (2) that L has a fingerprint (hash value) that matches one that is included as part of a bridge transaction. $$\mathcal{R}_{\mathsf{zk}}((\mathsf{x},\mathsf{h}),\mathsf{C}) := \big( \mathtt{Header}_i(\mathsf{C},\mathtt{apl}_i(\mathsf{C})) = \mathsf{h} \wedge \mathcal{R}(\mathsf{x},\mathsf{C}) \big) \; ,$$

The Verify algorithm obtains the CRS from the memory mem. It then checks if the

hash value h (included with the proof) appears in mem. If not, the proof is rejected; otherwise it is verified according to the verification algorithm of the proof system.

#### <span id="page-31-0"></span>B. Consensus-certificate bridges

From a high-level perspective, a number of bridge constructions presented herein rely on *consensus certificates for the headers* of the sending ledger. These are certificates that guarantee that honest parties agree (have consensus) on headers for a prefix of their chain. By virtue of the collision resistance of headers, these certificates also guarantee that parties agree on a prefix of local ledgers. Many ledger protocols *already* come equipped with such consensus certificates. For example, iterated-BFT-type protocols typically prepare quorum certificates on the tip of the chain. In this section, we formalize what it means for a ledger to offer secure consensus certificates. We then show how to use these to construct bridges.

We start by defining auxiliary algorithms associated with generating and verifying consensus certificates.

 $\underline{\mathrm{epoch}(r) \to e}$ : This is a deterministic algorithm that returns an epoch number for a given round number. For concreteness, we assume this function takes the form  $\underline{\mathrm{epoch}(r)} := \lfloor r/R \rfloor$  for some integer  $R \geq 1$ . Thus, the smallest r for which  $\underline{\mathrm{epoch}(r)} = e$  is eR, while the largest such r is (e+1)R-1. We call this contiguous sequence of adjacent rounds an epoch.

#### <span id="page-34-0"></span>C. Fixed-committee bridge

A second type of bridge design is one where each member of a *fixed* group of bridge operators  $\mathbf{P}$  observe the state of the sending ledger (thus,  $\mathbf{P} \subseteq \mathbf{P}_i$ ) and submit a signature on the hash of the agreed prefix of the sending ledger to the receiving ledger (thus,  $\mathbf{P} \subseteq \mathbf{P}_j$  as well). A contract deployed on the receiving ledger collects the submitted signatures. When a threshold number of signatures is reached, the hash value is declared as accepted. This is a simple bridge design and has been used in the Ronin [43] and Wormhole bridge [50]. However, fixing the committee makes it brittle and vulnerable to committee attacks [43].

Nevertheless, and due to its popularity, we formalize this bridge in this section. The public parameters include a CRS for a proof system, and bridge maintenance keys are a pair of standard signature keys (see Bridge Scheme 4). The root of trust includes the CRS and the list of all bridge maintenance verification keys.<sup>16</sup>

The Br algorithm in round  $r_0+2$  prepares and deploys a contract FCBridge[rot] on the receiving ledger, which has the root of trust data rot hardwired. This is similar to the direct-observation bridge except that FCBridge is implemented differently. From rounds  $r_0+3$  onward, the Br algorithm computes a digital signature  $\sigma$  on the header of the agreed prefix and its verification key, and prepares it as an argument to be submitted to the update method of the contract. The update method checks the validity of  $\sigma$  with respect to vk on message (h, vk). If ok, it adds (h, vk) to its memory mem. It then checks if greater than a  $\rho$  fraction of bridge operator keys appear in memory for a specific h. If so, the contract finalizes h in its memory.

The second method of FCBridge, verify, is implemented analogously to that in the direct-observation bridge. The Certify algorithm also operates similarly to that in the direct-observation bridge with respect to the same zk-relation  $\mathcal{R}_{zk}$ . Algorithm getVerData extracts the memory of FCBridge. Algorithm Verify retrieves the CRS from mem, checks that its input h is in its memory, and applies the verification procedure of the SNARK. As these are similar to those in the direct observation bridge, we omit their formal descriptions.

We next prove that the fixed-committee bridge is sound and live as defined in Section III-B as long as the sending and receiving ledgers are secure (persistent and live), at most a  $\rho$  fraction of bridge parties are corrupt. For this, the proof system needs to be knowledge sound and complete, and the headers need to be collision-resistant.

#### <span id="page-36-0"></span>*A. The ATS ideal functionality*

To facilitate our model checking of the ATS bridge, which relies on *ideally secure* building blocks, we give a definition of an ideal functionality for ATS schemes. To the best of our knowledge, this functionality is novel, and although complex, had been thoroughly checked for flaws. We give an outline of this functionality below, and describe how its design prevents (ideal-world) adversaries from breaking its correctness and unforgeability properties. This functionality is potentially of interest in other contexts involving threshold signatures.

Functionality F-[ATS](#page-37-2) comprises interfaces that match the algorithms for an ATS scheme, except for setup, which compatible with UC approaches, is provided by independent idealized building blocks formulated via setup assumptions. We give an overview below and present its full specification in Functionality [24.](#page-37-2)

Interfaces P.Gen, P.Sign, and P.Ver are defined similarly to a standard signature ideal functionality F-[Sig](#page-21-0). In particular, individual verification keys that are generated (via the simulator) are sanitized so that there are neither collisions nor pre-forgeries wrt. them. The generated signatures are also sanitized so that they will always verify. Verification ensures that the output bit is overwritten by 0 if the signer is honest and the input signature was not generated through the P.Sign interface.

Interface P.AKey aggregates a given list of keys (possibly with repetitions). For correctness, it ensures that, independently of what the simulator outputs, honestly generated individual keys always successfully aggregate. Interface P.ACheck checks if an aggregate verification key is correct essentially through recomputation. These two interfaces keep track of lists of verification keys and their aggregates for consistency.

Interface P.ASign returns an aggregate signature τ . It proceeds by computing an avk associated with the input list of verification keys, returning ⊥ if the simulator rejects the vk list. It then checks if there are enough verifying σ in S, returning ⊥ if this is not the case. Verification bits are computed as usual via the simulator. Finally, a sanitized aggregate signature is returned, which is ensured to pass aggregate verification.

Interface P.AVer is designed so that a list of individual verification keys vk<sup>∗</sup> corresponding to a given avk is computed *via the simulator*. The interface counts, *with multiplicity*, if the total number of non-honest keys that have signed the message, corrupt keys, and other keys that appear in vk<sup>∗</sup> is less than the threshold t. In this case, the interface overwrites the simulator's bit with 0. This ensures unforgeability.

In this functionality, we keep track of the maliciously generated keys in a set CVK, which P.Gen interface avoids. This allows the functionality to distinguish honest and malicious keys, which is then used to define correctness of signature aggregation.

A quick glance at the functionality reveals that there are many details, spanning over 100 lines of code. This, in turn, gives rise to the possibility of bugs. We have designed the functionality in tandem with model-checking, where we have a variety of trace and state properties. Indeed, our initial attempts at formulating F-[ATS](#page-37-2) had subtle bugs, some of which were arguably confusing. For instance, F-[ATS](#page-37-2) maintains two lists AVK and VK that intuitively keep a correspondence between lists of individual keys and aggregate verification keys. Due to the simulators resumption capabilities, the presence of maliciously generated keys, and the fact that the lists are populated under different interfaces, bugs were present. We used Quint's output trace vectors, to debug the functionality. For example, this led us to keep track of malicious keys in a list CVK and avoid them. In its final formulation, Quint reports that F-[ATS](#page-37-2) passes a number of properties, including correctness and unforgeability for both individual and aggregate signatures; see Section [V.](#page-11-1)

#### *B. ATS correctness*

In Game [5,](#page-38-0) we extend Game [1](#page-9-0) so that in addition to unforgeability, we also define three correctness properties. We require that 1) any set of keys that are aggregated, even if some of the keys are not honestly generated, pass ACheck (AVKC[ORRECTNESS](#page-9-0)); 2) individual signatures pass (individual) signature verification (SIGC[ORRECTNESS](#page-9-0)); and 3) ATS aggregation of any set of t+ 1 individual vk/signatures pairs that verify with some vk, verifies against wrt verification procedure (t-ASIGC[ORRECTNESS](#page-9-0)) subject to the condition that vk in the provided pairs appear in a unique position in vk. Note that this still allows for duplicate entries in vk to account e.g. for weighted votes, but ensures that the provided signatures do not exceed their specified quota. Thus, "ad-hocness" refers to the fact that this set can be chosen in an ad-hoc manner after the signatures are generated. We note that this property must hold even with respect to malicious keys. In particular, and surprisingly, the correctness notion defined in the original work [\[28\]](#page-13-12) is *not* sufficient for the liveness of our ATS bridge.

As shown in [\[28,](#page-13-12) Section 5], an ATS based on a multisignature scheme from [\[8\]](#page-12-20) (which also includes Merkle proofs of inclusion in a Merkle commitment to a set of verification keys) achieves t[-UF-CMA](#page-9-0) security under adaptive corruptions. Our adaptive notions of correctness (where some malicious keys may be included in the aggregate verification key) are also satisfied. Indeed, AVKC[ORRECTNESS](#page-9-0) relies on AKey being deterministic, and SIGC[ORRECTNESS](#page-9-0) on the perfect correctness of the underlying multi-signature scheme and Merkle proofs, which also hold for maliciously generated keys. The t-ASIGC[ORRECTNESS](#page-9-0) property is met because of the soundness of Merkle trees and perfect correctness of the underlying multi-signature scheme.

#### <span id="page-40-0"></span>D. Light-client-based and ZK bridges

Cross-chain bridges and light clients are intimately connected, and many real-world bridges are (or claim to be) based on light clients [9], [20]. In this work, we show a connection that (to a limited extent) also goes the other direction, at least in terms of definitions. Namely, we define the security features required from a low-level light client in terms of a bridge to a local one-party ledger<sup>18</sup> corresponding to the client's device that can locally verify the statements about the ledger that the client is aware of.

*Local ledgers*. We define the local ledger scheme  $DLS_0$  wrt.  $Val_0$  to be a ledger scheme where  $Setup_0$ ,  $AKGen_0$ ,  $KGen_0$  and  $Genesis_0$  return the empty string  $\varepsilon$ . Algorithms  $Chain_0(st) \to C$  and  $Ledger_0(C) \to L$  are identity maps. Algorithm  $BC_0^{Val_0}(r, m, tx; st)$  ignores the input round r and messages m and operates by calling  $Val_0$  to check the validity of transactions before appending the list of valid transactions in tx to its state st.

Note that  $\mathtt{Ledger}_0(\mathtt{Chain}_0(\mathsf{st}))$  is simply the list of accepted transactions. We define the local ledger protocol associated with a party P and  $\mathtt{Val}_0$  to be  $\Pi\text{-Ledger}[\mathbf{P},\mathbf{w},\delta,\mathsf{DLS}_0],$  where  $\mathbf{P}:=\{\mathsf{P}\},\,\mathbf{w}_\mathsf{P}:=0,\,\delta:=0,$  and  $\mathtt{DLS}_0$  is the local ledger scheme associated with  $\mathtt{Val}_0.$  This ledger satisfies  $(\mathbb{A}_0,0)\text{-LedgerPersistence}$  and  $(\mathbb{A}_0,0)\text{-LedgerLiveness}$  under  $\mathbb{A}_0:=\mathsf{True},$  i.e., under no trust assumptions. For light-client-based bridges, we need the local ledger, i.e., its predicates  $\mathtt{Val}_0$ , to support smart contracts, e.g., as defined in Algorithm 14.

Light-client schemes and protocols. We define a lightclient scheme LC simply as a bridge scheme that is used to build a bridge protocol from some sending ledger  $\Pi$ -Ledger<sub>i</sub> to a local receiving ledger  $\Pi$ -Ledger<sub>0</sub>. We call the resulting bridge protocol a light-client protocol  $\Pi$ -LC. Note that this results in a notion of a light client supporting a relation  $\mathcal{R}$ . This conceptualization of light clients immediately leads to the notions of A-LIGHTCLIENTSOUNDNESS := A-BRIDGESOUNDNESS  $(\mathbb{A}, \delta)$ -LIGHTCLIENTLIVENESS and := $(\mathbb{A}, \delta)$ -BridgeLiveness where  $\mathbb{A}$  and  $\delta$  need to be sufficient to bridge to the local ledger. The protocol part of a light client, which supplies the bridge transactions, is seen as the light-client *server*, and the local ledger (including any contracts deployed) is seen as the light client.

The work of Xie et al. [51] introduces informal notions of liveness and consistency for light clients. Briefly, liveness requires blockchain headers to be eventually included in the light-client state, and consistency essentially requires that the header list of any light client node is a prefix of that of any full node. Setting the bridge relation  $\mathcal{R}(x,C)$  to one that checks if x matches the header of C, we recover formal notions that

imply these. Indeed, liveness (in our sense) means that the header must be included so that the relation can be checked, and soundness precludes the inclusion of any incorrect header as this would result in an incorrect statement that successfully verifies. Note that this corresponds to a 'bare' light client which needs to be dressed up to verify, e.g., transaction inclusion. This extended light client can be seen as a bridge (in our syntax) where the bridge relation is  $\mathcal{R}(x,C) := (x \in \text{Ledger}(C))$ .

Succinctness. One desirable aspect of practical light clients is succinctness: in each invocation, they require at most O(1) communication and computation to synchronize their internal state [51]. In our formulation of light clients, succinctness means that the underlying bridge protocol uses O(1) communication to communicate with the local ledger (though the bridge parties may have a higher communication complexity) and further that the local ledger uses O(1) time to update its state, the non-trivial component of which is  $Val_0$  (e.g., by memoizing all but the last transaction).

Instantiating light-clients from bridges. A light-client scheme can be instantiated using any of the bridge schemes defined in the previous sections except for the direct-observation bridge. This is because the direct-observation bridge requires the predicate  $Val_0$  to observe the sending chain, which a local ledger does not support (or else if it is extended to support it, the resulting light client would become a full node).

LC-based bridges. Let  $\Pi$ -Ledger $_i$  and  $\Pi$ -Ledger $_j$  be two ledger protocols, and let LC be a light-client scheme for  $\Pi$ -Ledger $_i$  wrt.  $\mathcal{R}$  (i.e., a bridge scheme from  $\Pi$ -Ledger $_i$  to the local ledger). We construct a bridge scheme BBS from  $\Pi$ -Ledger $_i$  to  $\Pi$ -Ledger $_j$  using LC supporting the same relation  $\mathcal{R}$ . Informally, this bridge operates a single "on-chain party" running the local ledger scheme on the receiving ledger. Note that the local ledger is non-interactive as it does not rely on any incoming network messages. We establish the bridge by bridging  $\Pi$ -Ledger $_i$  to this local ledger using the light-client scheme.

Instead of explicitly running the local ledger on the receiving ledger – which even if O(1) can be costly as it might require verifying a constant but large number of signatures from a committee– we attach succinct proofs of correct transition of the local ledger state. The details of this protocol are shown in Bridge Scheme 5.

*Proof sketch.* We give an overview of the structure of the proof.

G<sub>0</sub>: This is the BRIDGESOUNDNESS game.

- $\underline{G_1}$ : This game aborts if an invalid transaction is included in ledger  $\Pi$ -Ledger $_j$ . In particular a transaction for which SNARK.Verify or Verify fail. The probability of this event is upper bounded by LEDGERVALIDITY $_j$  as in previous bridge soundness proofs.
- $G_2$ : For every bridge update transaction  $btx_j = (clD, update, (st', \pi))$ , this game runs the knowledge extractor for the SNARK to obtain a bridge transaction  $btx_0$  for the local ledger, and aborts if the extracted transactions do not result in the submitted state st' when run with respect to state st. The probability of this event is bounded by KnowledgeSoundness. Note that we have to run the extractor for each update transaction. This implies the need for either straight-line extraction or tightly secure extraction, that allows to recursively build extractors from extractors without a blow up in the extractor's running time.
- $\underline{G_3}$ : This game runs with respect to the extracted witnesses. Since the effect of witnesses are the same as the submitted states and proofs, we have that  $\Pr[G_3] = \Pr[G_2]$ .

We now show that an adversary that violates BridgeSoundness in  $G_3$  can be used to win LightClientSoundness, the soundness of the light

client; that is,

$$Pr[G_3] \leq Pr[LightClientSoundness]$$
.

To see this, note that the bridge construction simply runs the Certify and Verify procedures of the light client, and by rules of  $G_3$ , the state of the ledger on chain evolved according to the bridge transactions submitted to it. Thus any violation of A-BridgeSoundness directly gives rise to a violation of  $A_{lc}$ -LightClientSoundness.
