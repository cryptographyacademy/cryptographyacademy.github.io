# vkproof: Succinct verification of indexed verifying keys using modular compilation and polynomial fingerprinting

Antonio Mej´ıas Gil†,1 and Xueqin Zhao†,2

<sup>1</sup>Provable, <antonio@provable.com> <sup>2</sup>Chengdu Neusoft University, <zhaoxueqin@nsu.edu.cn>

#### Abstract

We introduce vkproof, a preprocessing SNARG which enables verification of the Varuna verifying key (or that of any similar proof system based on Marlin [\[Chi+20\]](#page-60-0)) for the R1CS compiled from a given higher-level program. It has constant proof size and affords linear verifier costs in the number of instructions of the program rather than the density of the compiled R1CS, which makes it especially appealing in contexts where complex-to-arithmetise functions (such as hashing) appear as program instructions frequently. This verifier succinctness is achieved through modular compilation of programs and the use of fingerprints to verify polynomial correctness. We augment the algebraic holographic proof (AHP) model of Marlin by allowing oracles to witness polynomials in the instance and queries to linear combinations of indexed polynomials, resulting in a primitive we refer to as extended algebraic holographic proofs (eAHP).

Acknowledgements. The authors are indebted to Pratyush Mishra and Victor Sint Nicolaas for many valuable discussions, as well as to the authors of Marlin, which the extended algebraic holographic proof primitive is based on.

## <span id="page-2-0"></span>1 Introduction

Interest in compiling high-level languages into some representation suitable for succinct proving of execution has surged in recent years, with programmable blockchains as one of the main driving forces. Motivating examples for this paradigm include rollups, i. e. L2 blockchains where execution of a number of transactions is proved using a succinct proof system and the resulting proof is sent to an L1 smart contract for verification (for instance Polygon[1](#page-2-1) and Scroll[2](#page-2-2) ); and privacy-focused blockchains where validators verify a proof of correct execution of the private component of a transaction (such as Aztec[3](#page-2-3) ).

In Aleo[4](#page-2-4) , for instance, each program written in the custom-domain high-level language Leo, or its lower-level counterpart Aleo instructions, is compiled into a rank-one constraint system (R1CS) for proving and verification with the Varuna proof system[5](#page-2-5) , a variation on Marlin [\[Chi+20\]](#page-60-0). For reference, Marlin proving keys include three polynomials row, col and val for each of the three R1CS matrices (for a total of nine), whereas Varuna replaces val by the closely related rowcol and rowcolval for efficiency. Although we focus on the latter, the discussion applies to Marlin keys without any substantial changes.

Each program deployed onto the blockchain therefore comes with a fixed associated Varuna key pair: a proving key, used by the party executing the program (or any delegated entity proving that execution on their behalf); and a verifying key, used by any party wishing to verify the proof of execution – most notably, network validators. The question thus arises of who should generate those keys. In some conceivable designs, the blockchain validators that receive the verifying key alongside the program must simply trust said key. Note that a maliciously generated verifying key would enable critical attacks, as validators would be satisfied with proofs of execution of a functionality different than the program it purportedly corresponds to.

A strawman approach would be to have each validator generate the verifying key by themselves whenever a new program is deployed (we note at this point that Varuna verifying keys are deterministically computed from Aleo programs given the code of the latter). There are two fundamental issues with this approach:

• Large R1CS: the constraint systems that arise even from moderate programs are very large: both in terms of the number of rows and columns of their matrices and, more importantly for the purposes of Varuna, their densities (that is, the number of the non-zero entries they contain). Even though the R1CS compiled from a typical Aleo program tends to be very sparse (that is, a very large proportion of its entries have the value zero), computing the Varuna verifying key is challenging in practice even for such a constraint system[6](#page-2-6) . This is largely due to operations such as hashing (for arithmetic and, especially, bit-based Hash functions) that have complex arithmetisations in terms of finite-field operations.

• Expensive compilation: even producing the R1CS (or its sparse description) given the program code can be computationally unfeasible in restricted environments. Compilers for relatively expressive virtual-machine environments such as Aleo's snarkVM[7](#page-3-1) into R1CS (or any other typical SNARK-friendly arithmetisation) tend to be quite complex, as we briefly touch upon when discussing monolithic compilation below.

In order to address these challenges, we introduce vkproof, a preprocessing SNARG for Varunaverifying-key correctness. Any party with access to a program P (for instance, the entity deploying P onto the blockchain) can compute its Varuna verifying key vk together with a proof πvk attesting to its correctness. Other parties can then, given P and vk, verify πvk and be satisfied that vk is correct and hence suitable for future verification of Varuna proofs of execution of P.

The main goals when designing vkproof were:

• Succinctness: verifier work should essentially be O(|P|), where |P| is the number of instructions in P. In particular, it should have dependence at most logarithmic in the complexity of P, by which we understand the density of the R1CS compiled from it. Note that, as a general guiding principle in SNARK design, O(|P|) is optimal in that the verifier always needs to at least read the full instance – which, in this case, includes the program. This is a qualitative improvement over the O(d · log(d)) field-arithmetic and O(d) elliptic-curve-arithmetic work needed to produce the verifying key, with d denoting the density of the R1CS. The costs of vkproofare dominated by two fixed-base elliptic-curve MSMs of a constant number of points.

As a second aspect of succinctness, proofs should ideally be short. vkproof achieves constant proof size of ∼ 1.7 KB assuming ∼ 256-bit base and scalar fields when compiled with the Marlin (algebraic-group-model, non-hiding) PCS.

- Reduced compilation costs: the aspect of program compilation performed by the verifier in the proof system (which it necessarily will, since the verifying key encompasses the functionality of P) should be less computationally expensive than the original compilation algorithm, which, as discussed, is prohibitively costly in some relevant contexts.
- No modifications to the Varuna proof system: the above improvements should not come at the cost of substantially modifying how the Varuna proof system operates[8](#page-3-2) . This would be problematic for existing pipelines using Varuna and also excessively expand the boundaries of the design space.

## <span id="page-3-0"></span>1.1 Our contributions

The proposed design achieves those aims by means of three main tools:

1) Modular compilation: A natural approach to designing an abstract compiler which arithmetises programs over a given language into R1CS triples is the following: for each instruction ν in the language, precompute an R1CS (three matrices (Aν, Bν, Cν)) capturing its functionality. On input a program P, incorporate the R1CS of each instruction in P into a running

<sup>8</sup>For instance, at an early stage of the project, expanding Varuna to include a permutation argument to enforce relations between program variables existing as duplicate columns of the R1CS was considered.

overall-program one. If an instruction ν<sup>i</sup> is applied to some variables that appeared earlier in the program, some columns of (Aν<sup>i</sup> , Bν<sup>i</sup> , Cν<sup>i</sup> ) are displaced to certain positions in the running R1CS to force the variables they represent to coincide with their previous appearances in the program. We refer to this paradigm as monolithic compilation, because this "column shuffling" makes it difficult to relate individual-instruction R1CS matrices to the overall-program one – and this relation is something a proof system hoping to incur linear verifier costs in the size of P would likely rely on. Modular compilation also imposes a significant computational burden on the compiler itself, which must really materialise the overall-program R1CS to capture the shuffling of entries.

Instead, we propose a modular compiler where each individual-instruction R1CS really appears as a (contiguous) submatrix of the overall-program one. More specifically, the Aν<sup>i</sup> are diagonally concatenated to one another, and similarly for Bν<sup>i</sup> and Cν<sup>i</sup> . This means that a vkproof verifier[9](#page-4-0) does not need to materialise the full R1CS: it only needs a virtual description of it, which is precisely the program. In order to capture the relations between variables that are lost when diagonally concatenating individual-instruction R1CS, the compiler also adds a number of structural constraints. The resulting full-program R1CS does have more non-zero entries than those produced by monolithic compilation, which translates increased computational costs for the Varuna prover. However, structural constraints (the only component introducing new non-zero entries with respect to monolithic compilation) are extremely sparse, which makes the additional prover costs negligible if the program has complex instructions.

2) Preprocessed data: In order to make vkproof verifier costs only grow as O(|P|) regardless of the complexity (more formally: density) of each of the |P| instructions in the program, each instruction in the language is preprocessed into some data that can be used to process any given program. This preprocessing is carried out by a trusted party known as the indexer in the context of algebraic holographic proofs [\[Chi+20\]](#page-60-0) (AHP)[10](#page-4-1) and other proof systems. This indexer only needs to be trusted once: after a fixed set of language instructions are preprocessed, verifying-key proofs can be generated for any (admissible) program over that language – in other words, no party needs to be trusted on a per-program basis. In fact the preprocessed data is publicly verifiable without any trust assumptions and can be updated with new instructions after the initial indexing phase.

Some readers may appreciate the irony of resorting to holography (more informally, preprocessing) in order to verify indexed verifying keys of a proof system which is in itself holographic - indeed, Varuna (and, more generally, Marlin) verifying keys are meant to provide fast verification of R1CS instances by preprocessing the constraint system in question.

3) Polynomial fingerprinting: As already mentioned, a Varuna verifying key essentially consists of three polynomials per matrix (for a total of nine; three of which are in fact replaced by six related ones for increased efficiency) which, with some caveats, interpolate the row positions, column positions and values of the matrix non-zero entries. Instead of preprocessing those same polynomials for each individual-instruction R1CS, we turn to fingerprint polynomials. By the fingerprint of a polynomial f at a point α ∈ F we simply mean the sum of its evaluations (over some specific set) each scaled by a different power of a fixed element

<sup>10</sup>This then gives rise to the "preprocessing" in "preprocessing SNARGs" after AHP compilation.

- α. The fingerprint polynomial of f is the polynomial that maps each α to the fingerprint of f at α. vkproof checks the fingerprints of the polynomials in the Varuna verifying key against their expected values, computed largely from preprocessed data. Switching from preprocessing the Varuna-key polynomials to polynomials which compute their fingerprints has several important benefits:
  - It makes the proof size constant as opposed to O(|P|). Without delving into the details, the main reason is that fingerprint polynomials are easier to collate into full-matrix ones than those in the Varuna verifying key. When collating the latter, their variables are shifted in a way which would require the verifier to query them at a distinct point for each polynomial, thus forcing the prover to send an evaluation per Varuna-key polynomial per instruction, resulting in linear proof sizes. Furthermore, depending on the PCS chosen for compilation, checking batched opening proofs may still require one elliptic-curve pairing per queried point (this is, for instance, the case in the Marlin PCS [\[Chi+20\]](#page-60-0)), which would be a prohibitively high cost.
  - It significantly reduces the cost associated to structural constraints. The fingerprints (at some α ∈ F) of the Varuna polynomials of a full-program R1CS easily split into one part corresponding to the submatrix containing the instruction submatrices, and a second one corresponding to structural constraints. Computing the latter has a cost (in terms of field multiplications) which is linear in the number of instructions of the program, and concretely very small. Attempting the same split with the original Varuna polynomials (instead of their fingerprints), would require to evaluate a row, col or val polynomial interpolating the structural-constraint part of the R1CS and zeros everywhere else in the full Varuna domain. This means the costs in terms of field arithmetic would still be O(d log(d)) in the density d of the full program R1CS (assuming FFTs were used).
- 4) Extended algebraic holographic proofs: Our information-theoretic protocol is inspired by the algebraic holographic proof (AHP) primitive introduced in Marlin [\[Chi+20\]](#page-60-0) yet does not quite fit that formalism. One reason is that the relation we eventually want to prove with a preprocessing argument, which captures correctness of Varuna verifying keys, includes polynomial commitments in its instances: indeed, an RVarunaVK instance consists of the twelve commitments to the polynomials in the corresponding Varuna proving key, as well as the program P they purportedly capture. Furthermore, our goal of achieving constant-time proofs is only enabled by the fact that the verifier can query a linear combination of polynomials without querying each of them individually (thus avoiding having to receive that many evaluations from the prover).

In order to address both challenges at the information-theoretic-protocol level, we expand the AHP by allowing the instance to contain oracles to distinguished polynomials, and allowing the verifier to query linear combinations of (certain) polynomial oracles. The result is a primitive we term extended algebraic holographic proofs (eAHP). We introduce this primitive, provide a black-box compiler which produces a succinct argument given an eAHP and a polynomial commitment scheme (formally introducing the homomorphic property required of the latter along the way) and prove the completeness and soundness of the compiled protocol.

## <span id="page-6-0"></span>1.2 Related work

Since our protocol is aimed at shifting verification work on indexed verifying keys from the verifier to the prover and relies, among other techniques, on modular compilation of programs, related work comes from several research areas: proof systems for virtual-machine (VM) environments, distributed or delegated computation of proving and verifying keys, and attempts to verify key correctness through circuit recursion. In this subsection, we emphasize the differences in design goals and threat models between those paradigms and our protocol.

Modular-compilation-based approaches for VM environments A large body of recent work has tackled the problem of proving the execution of complex programs in a virtual machine defined by some instruction-set architecture (ISA), including models enriched with some notion of public state to capture the expressivity of programmable blockchains. These approaches compile a program into a sequence of instructions and allow the prover to show the correctness of each instruction's semantics, the program's execution trace, and memory operations. Such techniques are already in use in production systems including SP1's zkVM[11](#page-6-1), RISC Zero[12](#page-6-2), Polygon's zkEVM[1](#page-2-1) , and zkSync's EraVM[13](#page-6-3). The cryptography involved typically relies on techniques such as lookups (with Jolt[\[AST24\]](#page-60-1), which leverages Lasso [\[STW24\]](#page-60-2) lookups, as a prime example), STARKs and recursion.

The crucial difference between all of the above and vkproof is that our protocol is not aimed at (and indeed does not enable) proving and verification of program execution itself: that is still handled by the Varuna proof system – which does not take particular advantage of modular, ISA-based circuitry. Our only reason for introducing modular compilation is making Varuna proving and verifying keys themselves succinctly provable and verifiable.

Delegated and distributed computation of verification keys A natural way to address the question of verifying-key correctness in contexts where synthesis is computationally intensive is to outsource key generation to a trusted party. However, this entails a non-trivial trust assumption which, in the case of a programmable blockchain with private execution such as Aleo, occurs for each new program function deployed onto the chain. In order to mitigate this, key generation can be performed it in a distributed fashion (for instance, a multi-party computation (MPC) ceremony) with minimal trust assumptions, which only requires an inexpensive verification step afterwards for any interested parties.

Classical and production-grade MPC systems such as Global-Scale Secure MPC [\[WRK17\]](#page-60-3) and MP-SPDZ [\[Buc+25\]](#page-60-4) support outsourcing linear algebraic computations while preserving privacy[14](#page-6-4) . However, these general-purpose MPC systems typically incur large communication and computation overhead, making them ill-suited for latency-sensitive applications such as blockchains. In the context of zero-knowledge proofs, systems like Eos [\[Chi+23\]](#page-60-5) and Siniel [\[Yan+24\]](#page-60-6) have investigated

privacy-preserving verifiable computation delegation, where zkSNARK computation is parallelized across multiple workers.

Our approach differs fundamentally in that it targets Marlin-based zkSNARK systems with circuitspecific setup phases and does not involve delegated or distributed key (or proof) generation. The only trust assumption in vkproof is in the one-off indexing phase (which is publicly verifiable, although computationally expensive): every individual-program key can subsequently be locally verified against malicious key generation. Existing work on distributed or outsourced verification key generation differs from this in terms of the underlying security model, communication cost (which is minimal in vkproof), computational overhead and fault tolerance.

Circuit-based verification of verifying keys A conceptually simple way to ensure correctness of verifying-key synthesis would be to encode the compilation and key-generation process itself as a circuit in some standard DSL (e. g. Circom) or lower-level arithmetisation (R1CS, PLONK [\[GWC19\]](#page-60-7), etc.) and then use an existing compatible proof system (such as Zcash' Halo2[15](#page-7-1) for PLONK) to ensure correctness of each program-to-key execution. There is, however, a sizeable practical obstacle to this approach: the arithmetisation of a program such as the snarkVM compiler would be extraordinarily complex. Observe that the resulting arithmetised circuit would capture each possible execution path of the compiler (that is, each possible input program) simultaneously, which in the case of languages as expressive and complex as Aleo instructions (which includes, for instance, input- and output-record encryption and hashing) would be prohibitively expensive to produce proofs for.

The verifier of a proof of execution of such a vast arithmetised compiler would be faced with similar computational challenges, which could be mitigated with the usual recursion techniques. However, proof recursion increases the prover work (which was already too high) and introduces other architectural challenges such as curve chaining, cross-curve compatibility and handling of trusted setups within recursion (such as long structured reference strings). Even when recursion is feasible, the handling of low-level components such as polynomial commitments often incurs a significant overhead, further reducing real-world applicability.

Instead, vkproof offers a lightweight, tailored SNARG which avoids the arithmetisation explosion and the engineering burden of recursive constructions.

## <span id="page-8-0"></span>2 Extended algebraic holographic proofs (eAHP)

Our protocol is formulated as a variant of the algebraic holographic proof (AHP) primitive introduced in Marlin [\[Chi+20\]](#page-60-0). Recall that an AHP is a special type of interactive proof system where

- a trusted indexer precomputes certain polynomials given an index i and a finite field F.
- the prover receives the same index i and field F as the indexer (alongside the instance x and witness w) and the verifier receives the same field F as the indexer (alongside x).
- all prover messages to the verifier are polynomials over F with preestablished degree bounds.
- all verifier messages are sequences of elements in F of preestablished length.
- assuming the AHP is public coin (that is, verifier messages are uniformly random strings of field elements), the verifier's operation is split into the following phases, in order:
  - an interactive phase, where it receives messages from the prover and replies with its own messages.
  - a query phase, where it consults the evaluations at any desired points of polynomials precomputed by the indexer or sent by the prover.
  - a decision phase, where it performs a number of calculations (possibly involving the evaluations learnt during the query phase) and either accepts or rejects.

More details can be found in section 4 of the cited article. As explained in op. cit., one can compile such an AHP together with an extractable polynomial commitment scheme (PCS) which enforces degree bounds into a preprocessing argument with universal SRS by essentially having the indexer and prover commit to polynomials instead of directly sending them to the verifier, and subsequently replacing the verifier's query phase by a number of commitment openings and opening checks, which can be batched for efficiency.

If the original AHP was public coin, the preprocessing argument compiled as above can furthermore be made non-interactive using the Fiat-Shamir transform, thus becoming a preprocessing SNARG with universal SRS[16](#page-8-1) (potentially a SNARK and potentially zero knowledge depending on the properties of the AHP and the PCS). [\[Chi+20\]](#page-60-0) furthermore restricts its attention to non-adaptive AHPs, where the points queried by the verifier during the query phase depend only on its internal randomness and input (and not, for instance, on prover messages or the replies to other queries). This allows the evaluation opening and opening checks to take place in a single execution of the PCS's batched opening and checking methods.

In order to accurately model our protocol and evaluate its security, the AHP primitive must be augmented with two additional features:

i) The instance x can include oracles to certain polynomials in the witness w.

<span id="page-8-1"></span><sup>16</sup>In the sequel, we may abbreviate this to "preprocessing SNARG" and "preprocessing argument with universal SRS" to "preprocessing argument".

{9}------------------------------------------------

This is dictated by the nature of our problem, where the instance to be verified (which includes a Varuna verifying key) contains commitments to some witness polynomials (namely, to those in the Varuna proving key).

ii) The verifier has the ability to query not only polynomials exposed by the indexer, but also linear combinations of them – that is, expressions of the form  $s_0 \cdot f_0 + \ldots + s_n \cdot f_n$ , where the  $f_i$  are such polynomials and the  $s_i$  are elements of  $\mathbb{F}$ .

This need arises from the vkproof verifier combining preprocessed individual-instruction fingerprint polynomials (or, after compilation with a PCS: their commitments) into their full-program analogues.

Note that, for reasons that will become apparent shortly, the possibility to query linear combinations of polynomials does not extend to polynomial oracles in the instance or to polynomials sent by the prover during the interactive phase.

Some other minor differences are only detailed in the formal definitions below. For instance, degree bounds of round polynomials must be allowed to depend on the (non-oracle part of the) instance x instead of only the index i, as do the degree bounds of the newly introduced instance polynomials. Motivated by this, indices are required to include an overall degree bound to control the aforementioned, instance-dependent bounds.

We refer to an interactive protocol as above as an extended algebraic holographic proof (eAHP). In this section, we formally define eAHPs, describe their compilation to a preprocessing argument with universal SRS and prove the soundness of the latter. Although minor, the introduced modifications give rise to some subtleties (for instance, the need for a formal notion of homomorphic and separable PCS for efficiency or the appearance of commitments in the relevant indexed relation) which are carefully addressed. We do not define or study knowledge soundness or zero knowledge for eAHPs as those are not relevant notions to our protocol of interest: in the Varuna verifying-key relation (detailed at the start of section 4), the witness consists only of the twelve Varuna prover-key polynomials, which can be easily recovered from the program P contained in the instance.

### <span id="page-9-0"></span>2.1 Definition of eAHPs

The formal definition of extended algebraic holographic proofs largely follows that of standard algebraic holographic proofs in [Chi+20] section 4. For convenience, a full definition is given below and changes with respect to the original are highlighted in blue. Before that, we define a version of indexed relations whose instances may oracles to witness polynomials. This is not an infrequent technique (cf. for instance [Che+23] definition 2.3), but we tailor it to our specific needs.

<span id="page-9-1"></span>**Definition 1.** An **indexed polynomial-oracle relation**  $\mathcal{R}$  over a family  $\mathcal{F}$  of finite fields is a set of triples (i, x, w) where

- i) the index i contains the description of a finite field  $\mathbb{F}_i \in \mathcal{F}$  and a tuple of degree bounds  $(N_{i,j})_{j=1}^d \in \mathbb{N}^d$ . We set  $N_i = \max_{1 \leq j \leq d} N_{i,j}$  and refer to it as the **index degree bound**.
- ii) the witness w is a pair  $w = (w_o, \widetilde{w})$ , with  $w_o = (f_i)_{i=1}^n$  in turn consisting of a (possibly empty) tuple of univariate polynomials  $f_i \in \mathbb{F}_{i}[x]$ .

{10}------------------------------------------------

iii) the instance x is a pair  $x = (x_0, \tilde{x})$ , with  $x_0 = ([[f_i]], d_i)_{i=1}^n$  containing oracles  $[[f_i]]$  to all witness polynomials  $f_i$  and degree bounds<sup>17</sup>  $d_i \in \mathbb{N}$  such that  $\deg(f_i) \leq d_i$  for all i.

The number n of polynomials, degrees and oracles is the same in the instance and witness but can vary for each triple in  $\mathcal{R}$ . The **language** defined by  $\mathcal{R}$  is the set  $\mathcal{L}(\mathcal{R})$  of all pairs (i, x) such that  $(i, x, w) \in \mathcal{R}$  for some w.  $\square$ 

<span id="page-10-3"></span>**Definition 2.** An extended algebraic holographic proof (eAHP) for an indexed polynomial-oracle relation  $\mathcal{R}$  over a family  $\mathcal{F}$  of finite fields is a tuple

$$eAHP = (k, s, d, I, P, V)$$

such that:

- k, s and d are (deterministic) polynomial-time functions  $\{0,1\}^* \to \mathbb{N}$  specifying the number  $\mathsf{k}(|\dot{\mathfrak{s}}|)$  of rounds of interaction between **I** and **P**; the number  $\mathsf{s}(\dot{\mathfrak{s}},0)$  of polynomials output by **I** during preprocessing and the number  $\mathsf{s}(\dot{\mathfrak{s}},i)$  of them sent by **P** in the *i*-th round of interaction<sup>18</sup>; and the degree bounds of each polynomial, respectively. More precisely,
  - o d(i, 0) is the degree bound for *all* indexed polynomials given i (and does not only depend on |i|).

- I, known as the *indexer*, is a non-interactive deterministic polynomial-time algorithm which receives an index i and outputs polynomials  $f_{0,j}, \ldots, f_{0,s(i,0)} \in \mathbb{F}_i[x]$  with  $\deg(f_{0,j}) \leq \mathsf{d}(i,0)$  for all  $j \in \{1, \ldots, \mathsf{s}(i,0)\}$ .

  - $\circ$  **P**, known as the *prover*, receives as input (i, x, w).
  - $\circ$  **V**, known as the *verifier*, receives oracle access to i (including the description of  $\mathbb{F}_i$ ) and to the polynomials  $f_{0,i}$  output by the indexer on input i; and receives as input the full instance x.

## <span id="page-17-0"></span>2.3 Preprocessing arguments for relations with commitments

The existence of commitments to witness polynomials in the instances our relation of interest (capturing the correctness of Varuna verifying keys compiled from programs), which already motivated the introduction of indexed polynomial-oracle relations in subsection [2.1,](#page-9-0) also requires a minor modification of the "preprocessing argument with universal SRS" primitive from [\[Chi+20\]](#page-60-0) section 7. We illustrate the issue in the next few lines (which can be safely skipped until definition [7](#page-17-1) introducing the updated primitive itself).

Observe first that it is not possible to relegate the commitments to witness polynomials to the nonoracle part <sup>x</sup><sup>e</sup> of the instance <sup>x</sup> = (xo, <sup>x</sup>e) of the (uncompiled) indexed polynomial-oracle relation. This would result in the task of PCS openings and checks being deferred to the eAHP itself, yet the eAHP model is too rigid to accommodate the opening and checking procedures of polynomial commitment schemes of interest. Recall that, in that model, all messages from the prover to the verifier are polynomial oracles. Furthermore, placing the aforementioned oracles in <sup>x</sup><sup>e</sup> would prevent the black-box compilation we propose below to batch the opening of the corresponding commitments with those coming from eAHP-prover messages and indexed polynomials.

A second challenge appears when defining the result of compiling the indexed polynomial-oracle relation with a PCS – which largely means replacing the oracles to witness polynomials by commitments to them. In our PCS formalism, the only relation between polynomials and their commitments is the PCNH.Commit method. Note that in other paradigms, such as LegoSNARK [\[CFQ19\]](#page-60-10), polynomial commitment schemes come equipped with commitment-verification methods which check whether a given commitment indeed corresponds to a certain polynomial. In our case, when capturing the notion that "the commitments in the instance are indeed commitments to the witness polynomials", one must somehow refer to the committer key ck. Quantifying it with an existential quantifier ("there exists a ck such that (ci) = PCNH.Commit(ck, f<sup>i</sup> , . . .)") has several evident issues. Placing ck in the witness or index of the compiled relation (or even placing its vk counterpart in the instance) gives rise to formal difficulties. For instance, completeness of the preprocessing argument relies on the prover and verifier using a committer key generated by the algorithm G which could have nothing to do with the one contained in the witness or index (and could even correspond to a different security parameter λ, field, toxic waste, etc.).

The upshot from the above is that the committer key has to be a first-class citizen of the compiled relation separate from the index, witness and verifier; and it has to be related to both the output of G and the prover's commitment procedure. We capture this in the following definition, where we again highlight changes with [\[Chi+20\]](#page-60-0) in blue.

#### <span id="page-19-0"></span>2.4 Compilation of eAHPs with polynomial commitment schemes

Intuitively, compiling an eAHP into a succinct argument using a PCS proceeds as in the AHP case with the addition that

- i) instance oracles to witness polynomials are replaced with commitments to them<sup>26</sup>. Queries about their evaluations are simply added to the batched opening of the compiled argument system.
- ii) whenever the eAHP verifier would query a linear combination  $[[f]] = s_1 \cdot [[f_1]] + \ldots + s_n \cdot [[f_n]]$  of oracles to indexed polynomials as above at a point  $\alpha$ , the argument verifier instead computes the commitment  $c = s_1 \cdot c_1 + \ldots + s_n \cdot c_n$  using the available commitments  $c_i$  to the  $f_i$ , and requests c to be opened at  $\alpha$  (which is batched with the rest of the openings). Note that this requires all  $c_i$  to have been committed to with the same degree bound by the definition of homomorphic PCS. The prover computes c as well as  $f = s_1 \cdot f_1 + \ldots + s_n \cdot f_n$  itself in order to be able to open the combined commitment.

## <span id="page-27-0"></span>3 Modular compilation of programs

This section offers a detailed description of the modular compiler, which takes as input a program P and produces an R1CS which encapsulates its functionality; and also of the relations between various polynomials describing such an R1CS. The first subsection introduces some notation concerning matrices and the Varuna polynomials associated to them, as well as the notion of fingerprints of polynomials (or, equivalently, of matrices and vectors). In subsection 3.2, we delve into the properties of each individual instruction in the language required for the modular compiler to operate (corresponding to point i) above). Subsection 3.3 describes additional constraints added by the modular compiler to, among other aims, correctly relate the inputs and outputs of the various instructions in the program. In the last subsection, we combine the two previous aspects, introduce the crucial notion of hybrid fingerprints and state the relevant relations between individual-instruction polynomials and their full-matrix counterparts.

We understand a program P as a number N of input-output variables and a list of **commands**  $(\nu, \rho)$ , each of which contains:

<span id="page-27-1"></span>i) An **instruction**  $\nu$  (for example: add) which specifies the operation to perform. It is important here to distinguish "programmer-facing" instructions from *concrete* instructions, by which we mean the following: an instruction such as add behaves very differently if the operands are field elements in  $\mathbb{F}$ , 8-bit unsigned integers u8 or, for instance, points of some elliptic curve. Even though the programmer may be oblivious to the low-level differences between those three manifestations of add, the resulting R1CS are very different and should be treated separately.

Note that a concrete instruction is anything that should be reflected in the R1CS in the form of constraints. For instance, the *declaration* of a variable v of type bool may result in the incorporation of a constraint of the form  $v \cdot (1 - v) = 0$  in the constraint system and should therefore be treated as an instruction for the purposes of the modular compiler.

Unless otherwise stated, we shall only be concerned with concrete instructions in the sequel, and abstract away the task of identifying which concrete variant of a programmer-facing instruction is being used to a higher-levelcompiler.

ii) Operand information  $\rho$  which relates the instruction's operands to other variables in the program or constants. It can, for instance, indicate that the first input to the current command is the output of the command three steps above. This and other types of structural relations are specified in detail later in this subsection.

Recall that the distinction between inputs and outputs of an instruction is purely semantic in the context of R1CS: both are simply entries in the instance part of the assignment vector (or, equivalently, instance columns in the instruction's R1CS). In other words, what a programmer might write as  $v_3 = \operatorname{add}(v_1, v_2)$  may be regarded by the compiler as something like  $\operatorname{add}(v_1, v_2, v_3)$ . For simplicity, we treat the "output"  $v_3$  as one more operand of  $\operatorname{add}$ .

We fix a finite field  $\mathbb{F}$  and denote its group of units by  $\mathbb{F}^* = \mathbb{F} \setminus \{0\}$ . For each  $0 \le t \le t_{\max}$ ,  $K_{2^i}$  denotes the unique subgroup of  $\mathbb{F}^*$  of order  $2^t$ , for which fix a distinguished generator  $\omega_{2^t}$  in a compatible way:  $\omega_{2^t} = (\omega_{2^{t+1}})^2$ . ## <span id="page-31-0"></span>3.2 Individual-instruction information

<span id="page-31-2"></span>For each instruction ν in the language, the modular compiler requires the following data:

- i) Three matrices Aν, Bν, C<sup>ν</sup> ∈ M(F) encoding ν into an R1CS. The numbers of rows and columns and the density of these matrices do not need to equal a power of two: this does not affect proving or verifying costs of either Varuna or vkproof. The first column of each of those matrices corresponds[34](#page-31-1) to the constant 1.
- ii) dν,A, dν,B, dν,C ∈ N: the density of the matrices Aν, B<sup>ν</sup> and Cν, respectively. These are the same as dA<sup>ν</sup> , dB<sup>ν</sup> and dC<sup>ν</sup> in the notation of definition [12,](#page-28-2) but we avoid the latter in order to mitigate subscript and superscript clutter.
- iii) rν, c<sup>ν</sup> ∈ N \ {0}: the number of rows and columns of Aν, B<sup>ν</sup> and Cν.
- <span id="page-31-3"></span>iv) Input-output information IO<sup>ν</sup> = (IOν,1, . . . ,IOν,t), where IOν,i ∈ N is the index of the column corresponding to the i-th input-output variable of ν in Aν, B<sup>ν</sup> and Cν. We denote by |IOν| the number t of such variables.

For instance, if ν has two inputs and one output located in columns 2, 4 and 7 of the R1CS, respectively, one would have IO<sup>ν</sup> = (2, 4, 7) ∈ N <sup>3</sup> and |IOν| = 3.

This gives rise to the following fingerprint polynomials for each M<sup>ν</sup> ∈ {Aν, Bν, Cν}:

- For each power of two ⌈ 2 rν⌉ ≤ 2 <sup>t</sup> ≤ 2 <sup>t</sup>max , the row fingerprint polynomial φrow,Mν,<sup>2</sup> <sup>t</sup> (x) of depth t.
- For each power of two ⌈ 2 cν⌉ ≤ 2 <sup>t</sup> ≤ 2 <sup>t</sup>max , the column fingerprint polynomial φcol,Mν,<sup>2</sup> <sup>t</sup> (x) of depth t.
- The (single) value fingerprint polynomial φval,M<sup>ν</sup> (x).

This amounts to at most 3 · (2 · tmax + 1) polynomials per instruction.

The four types of structural constraints produced by the modular compiler are as follows:

• Operand equals preexisting program variable: If ρi,j is of the form[36](#page-33-0) var(v) and v already appeared as an operand in a previous command or is a public program input/output variable, the compiler adds the structural constraint formed by the triple of rows

It follows that the above structural constraint enforces equality between the two R1CS variables corresponding to the program variable v.

If the compiler encounters an operand ρi,j = var(v) where v has not appeared in the program before (and is not a public program input/output), no structural constraint is added: instead, the compiler simply stores the information that v is located at column c<sup>v</sup> = 1+N + ˜ci+ IOνi,j for later.

• Operand equals constant: If ρi,j is of the form const(k), the compiler simply adds the structural constraint

$$(e(1 + N + \tilde{c_i} + IO_{\nu_i,j}), e(0), k \cdot e(0)).$$

This forces the corresponding R1CS variable to equal k times the global constant 1.

• Local 1 equals global 1: As mentioned in subsection [3.2,](#page-31-0) the first column of each individualinstruction R1CS (Aν<sup>i</sup> , Bν<sup>i</sup> , Cν<sup>i</sup> ) is reserved for the constant 1. In order to enforce the honesty of this value within the full-program solution vector, one could introduce a new Varuna instance variable, but we advise against this for various performance and formal reasons. The following structural constraint can be added instead:

$$(e(1+N+\tilde{c}_i),e(0),e(0)).$$

This enforces equality between the variable at column e(1 + N + ˜ci) (the local constant 1 of the individual instruction) and that at column 0 (the global constant 1).

• Zero knowledge: As part of its zero-knowledge design, the current Varuna specification and implementation add three private variables zka, zk<sup>b</sup> and zk<sup>c</sup> at the end of the solution vector and forces them to satisfy zk<sup>a</sup> · zk<sup>b</sup> = zk<sup>c</sup> with the constraint

are the cumulative densities and row positions of the individual matrices (with  $\tilde{d}_1 = \tilde{r}_1 = 0$ ). We also define column and value hybrids

$$\widetilde{\varphi}_{\mathsf{col},\mathcal{M},2^t,\alpha}(x) = \sum_{i=1}^n \omega_{2^t}^{\tilde{c}_i} \cdot \alpha^{\tilde{d}_i} \cdot \varphi_{\mathsf{col},M_i,2^t}(x) \tag{10}_{\mathsf{col}}$$

and

<span id="page-35-3"></span>
$$\widetilde{\varphi}_{\mathsf{val},\mathcal{M},\alpha}(x) = \sum_{i=1}^{n} \alpha^{\tilde{d}_i} \cdot \varphi_{\mathsf{val},M_i}(x),$$
(10<sub>val</sub>)

where  $\tilde{c}_i$  is defined analogously to  $\tilde{r}_i$ 

The crucial property of these objects is that they satisfy

$$\widetilde{\varphi}_{\text{row},\mathcal{M},2^t,\alpha}(\alpha) = \varphi_{\text{row},M_1 \wedge \dots \wedge M_n,2^t}(\alpha)$$
 (11)

and analogously for col and val, as induction on lemma 13 iii) shows.

We note that, while the evaluation  $\widetilde{\varphi}_{\mathsf{row},\mathcal{M},\alpha}(\alpha)$  depends only on  $M_1 \wedge \ldots \wedge M_n$  (as shown by the above equality), the same is not true of the hybrid polynomial  $\widetilde{\varphi}_{\mathsf{row},\mathcal{M},2^t,\alpha}(x)$ , which does depend on the family  $\mathcal{M}$  itself – this is the reason  $\mathcal{M}$  needs to be reflected in the notation. For instance, consider two matrices  $M_1$  and  $M_2$  and the two families  $\mathcal{M} = (M_1 \wedge M_2)$  and  $\mathcal{M}' = (M_1, M_2)$ . Then  $\varphi_{\mathsf{row}}$  of the diagonal concatenation is the same in both families, yet  $\widetilde{\varphi}_{\mathsf{row},\mathcal{M},2^t,\alpha}(x) \neq \widetilde{\varphi}_{\mathsf{row},\mathcal{M}',2^t,\alpha}(x)$  in general: the latter has a lower degree (this will also result in improved prover costs later on). Broadly speaking, for a family  $\mathcal{M} = (M_i)_{i=1}^n$  as above, the fingerprint polynomials  $\varphi_{\mathsf{row},M_1\wedge\dots\wedge M_n,2^t}(x)$ ,  $\varphi_{\mathsf{col},M_1\wedge\dots\wedge M_n,2^t}(x)$  and  $\varphi_{\mathsf{val},M_1\wedge\dots\wedge M_n}(x)$  have degree approximately equal to  $\widetilde{d}_{n+1} = \sum_{j=1}^n d_j$ , whereas their hybrid counterparts  $\widetilde{\varphi}_{\mathsf{row},\mathcal{M},2^t,\alpha}(x)$ ,  $\widetilde{\varphi}_{\mathsf{col},\mathcal{M},2^t,\alpha}(x)$  and  $\widetilde{\varphi}_{\mathsf{val},\mathcal{M},\alpha}(x)$  have degree approximately equal to  $\max_{i=1}^n d_i$ .

## <span id="page-38-0"></span>4 Protocol

We first present the protocol eAHPvkproof as an eAHP for the following indexed polynomial-oracle relation RVarunaVK:

$$\{2^i - 1, 2^i - 2 : 1 \le i \le t_i\},$$

### <span id="page-40-0"></span>4.1 Overview of the eAHP

Before delving into the indexer, prover and verifier algorithms in detail, we give an overview of their operation. Using vague terminology to simultaneously account for  $\mathcal{P}$  and  $\mathcal{V}$ , the idea for each  $M \in \{A, B, C\}$  is as follows:

- Send ( $\mathcal{V}$ : receive) a purported value polynomial  $\mathsf{val}_M$  analogous to  $\mathsf{row}_M$  and  $\mathsf{col}_M$ .
- Reduce the correctness of  $\mathsf{rowcol}_M$  and  $\mathsf{rowcolval}_M$  to that of  $\mathsf{row}_M$ ,  $\mathsf{col}_M$  and  $\mathsf{val}_M$  using quotients  $q_{\mathsf{rowcol},M}$  and  $q_{\mathsf{rowcolval}_M}$ .
- Verify the two quotient relations from the previous point at a random point.

From now on, the correctness of  $row_M$ ,  $col_M$  and  $val_M$  is established by showing that their actual fingerprint at a random point  $\alpha$  matches the honest fingerprint:

- Receive ( $\mathcal{V}$ : sample and send)  $\alpha \stackrel{R}{\leftarrow} \mathbb{F}$ .
- Send ( $\mathcal{V}$ : receive) a polynomial  $\mathsf{pow}_{M,\alpha}$  interpolating  $(1,\alpha,\ldots,\alpha^{d_M-1})$  over the domain  $K_M$ , where  $d_M = \lceil^2 d_{\mathsf{Mfull}} \rceil = \lceil^2 d_{\mathsf{inst},M} + d_{\mathsf{stru},M} \rceil$  and  $K_M = K_{d_M}$ .

#### Honest fingerprint:

- Using  $(10_{\text{row}})$   $(10_{\text{val}})$ , compute the *honest* hybrid fingerprint polynomials  $\widetilde{\varphi}_{\text{row},\mathcal{M},r_{\text{full}},\alpha}(x)$ ,  $\widetilde{\varphi}_{\text{col},\mathcal{M},c_{\text{full}},\alpha}(x)$  and  $\widetilde{\varphi}_{\text{val},\mathcal{M},\alpha}(x)$ , where  $\mathcal{M}$  is as in subsection 3.4 (after definition 15), then reveal  $(\mathcal{V}: \text{query})$  their evaluation at  $\alpha$ .
- Manually compute the *honest* fingerprints  $\varphi_{\mathsf{row},M_{\mathsf{stru}},r_{\mathsf{full}}}(\alpha)$ ,  $\varphi_{\mathsf{row},M_{\mathsf{stru}},c_{\mathsf{full}}}(\alpha)$ ,  $\varphi_{\mathsf{val},M_{\mathsf{stru}}}(\alpha)$  at  $\alpha$ .
- Using the previous values, compute  $\varphi_{\overline{\mathsf{vec}}_{\mathsf{row}},M_{\mathsf{full}},r_{\mathsf{full}}}(\alpha)$ ,  $\varphi_{\overline{\mathsf{vec}}_{\mathsf{col},M_{\mathsf{full}}},c_{\mathsf{full}}}(\alpha)$  and  $\varphi_{\overline{\mathsf{vec}}_{\mathsf{val}},M_{\mathsf{full}}}(\alpha)$  by means of  $(12_{\mathsf{row}})$   $(12_{\mathsf{val}})$ .

#### Check against the actual fingerprint:

• Perform univariate sumchecks on the polynomials  $\mathsf{row}_M \cdot \mathsf{pow}_{M,\alpha}$ ,  $\mathsf{col}_M \cdot \mathsf{pow}_{M,\alpha}$  and  $\mathsf{val}_M \cdot \mathsf{pow}_{M,\alpha}$  over the domain  $K_M$  using as the expected sum of evaluations the fingerprint at  $\alpha$  from the previous point. Accept if and only if sumcheck verification passes.

The final interactive protocol is the result of transforming the above outline in several ways:

• In line with the eAHP formalism, the verifier's operation is split into an interactive phase, where all it does is send uniformly random field elements; a query phase, where the evaluations

{41}------------------------------------------------

of various polynomials at certain points are learnt; and a decision phase, where all expected relations are checked using the learnt values.

- Several steps are batched together across the three matrices (and the different functions f where possible), most notably the check on qrowcol and qrowcolval as well as the final univariate sumchecks. The powers of a random element β are used as batching randomness.
- Several steps which require a pass over all instructions (most notably, the computation of the various hybrid fingerprint polynomials and of the fingerprint of Mstru at α) are bundled together into a single pass.
- The relations between structural-constraint fingerprints from remark [16](#page-37-1) are incorporated.

What follows are the full indexer, prover and verifier algorithms of the eAHP. As explained in section [2,](#page-8-0) most polynomials in the eAHP (more concretely: those output by the indexer, sent by the prover or appearing in the witness) will appear to the verifier as commitments once the protocol is compiled into a preprocessing argument. In the remainder of this section, we denote the index field F<sup>i</sup> simply by F.

### <span id="page-41-0"></span>4.2 Indexer

$$\mathbf{I}(\mathbf{i}) \to \{ \{ \varphi_{\mathsf{row}, M_{\nu}, 2^{t}}(x) \}_{t}, \{ \varphi_{\mathsf{col}, M_{\nu}, 2^{t}}(x) \}_{t}, \varphi_{\mathsf{val}, M_{\nu}}(x) \}_{M \in \{A, B, C\}, \nu} :$$

For each iν<sup>i</sup> = ((Aν<sup>i</sup> , Bν<sup>i</sup> , Cν<sup>i</sup> ),(dνi,A, dνi,B, dνi,C),(rν<sup>i</sup> , cν<sup>i</sup> ),IOν<sup>i</sup> ) in i, I computes and outputs the following polynomials described in subsection [3.2:](#page-31-0)

- For each power of two ⌈ 2 rνi ⌉ ≤ 2 <sup>t</sup> ≤ 2 ti , the row fingerprint polynomials φrow,Aνi ,2 <sup>t</sup> (x), φrow,Bνi ,2 <sup>t</sup> (x) and φrow,Cνi ,2 <sup>t</sup> (x) of depth t.

### <span id="page-41-1"></span>4.3 Prover

## P(i, x, w):

The prover receives an index <sup>i</sup>, an instance <sup>x</sup> = (([[fM]], <sup>d</sup>f,M )f∈Fkey,M∈{A,B,C} , P) containing a program <sup>P</sup> = (N,((ν1, ρ1), . . . ,(ν<sup>I</sup> , ρ<sup>I</sup> ))) and a witness <sup>w</sup> = ((fM)f∈Fkey,M∈{A,B,C} , ⊥) as described at the beginning of this section.

#### 1) First round:

← Receive from V an empty message[40](#page-41-2) ⊥ ∈ F 0 .

<span id="page-41-2"></span><sup>40</sup>This is caused by the eAHP formalism, which requires the verifier to make the first move (mirroring the AHP model). Our eAHP is in essence a sigma-protocol.

{42}------------------------------------------------

- For M ∈ {A, B, C}, compute the value vector vecval,M corresponding to the full matrix Mfull compiled from the program P, whose length is by definition the density dinst,M + dstru,M . Set d<sup>M</sup> = ⌈ <sup>2</sup>dMfull⌉ = ⌈ <sup>2</sup>dinst,M + dstru,M ⌉ and K<sup>M</sup> = Kd<sup>M</sup> , then compute the polynomial valM(x) which interpolates vecval,M over K<sup>M</sup> – that is, it maps the i-th element of K<sup>M</sup> to the value of the i-th non-zero entry of M if i < dinst,M + dstru,M and to 0 otherwise. Note val<sup>M</sup> can also be obtained directly by interpolating the values (rowcolvalM(k)/rowcolM(k))k∈K<sup>M</sup> over KM.
- → Send to V the polynomials valA(x), valB(x) and valC(x).

### 2) Second round:

- ← Receive from V (α, β) ∈ F 2 .
- For[41](#page-42-0) M ∈ {A, B, C}, compute the power polynomial[42](#page-42-1) powM,α(x) ∈ F <d<sup>M</sup> [x] which maps the i-th element of K<sup>M</sup> to[43](#page-42-2) α i .

Instead of using an FFT, this can be done in linear time using lemma [18](#page-62-1) [ii\)](#page-62-2) and [iii\).](#page-62-3) Note furthermore that all the inverses inside the sum in [\(25\)](#page-62-4) featured in the lemma can be computed, using Montgomery's trick, with a single batch inversion in which the overall product QdM−<sup>1</sup> <sup>i</sup>=0 (α · ω <sup>−</sup><sup>i</sup> − 1) to be inverted is simply 1 − α <sup>d</sup><sup>M</sup> .

• Set dmax = max{dA, dB, dC} and K = Kdmax . Compute the univariate-sumcheck polynomials h and g uniquely defined by

$$\sum_{M \in \{A,B,C\}} \beta^M \cdot S_{K,K_M}(x) \cdot \mathsf{pow}_{M,\alpha}(x) \cdot \sum_{\mathsf{f} \in \mathcal{F}_{\mathsf{mat}}} \beta^\mathsf{f} \cdot \mathsf{f}_M(x) = h(x) \cdot v_K(x) + x \cdot g(x) + \Phi \quad (13)$$

with deg(g) < dmax − 1 and Φ ∈ F, where

$$\beta^A = \beta^0, \quad \beta^B = \beta^1, \quad \beta^C = \beta^2, \quad \beta^{\mathsf{row}} = \beta^0, \quad \beta^{\mathsf{col}} = \beta^3, \quad \beta^{\mathsf{val}} = \beta^6$$

and vK(x) = x <sup>d</sup>max − 1 is the vanishing polynomial over K. If P's inputs are honest, one has

<span id="page-42-3"></span>
$$\Phi = \frac{\sum_{M \in \{A,B,C\}} \sum_{f \in \mathcal{F}_{\text{mat}}} \beta^M \cdot \beta^f \cdot \varphi_{\overline{\text{vec}}_{f,M_{\text{full}}}}(\alpha)}{d_{\text{max}}}.$$
 (14)

• Compute the batched rowcol and rowcolval quotient polynomials

$$q_{\mathsf{rowcol}}(x) = \sum_{M \in \{A, B, C\}} \beta^M \cdot \frac{S_{K, K_M}(x) \cdot (\mathsf{row}_M(x) \cdot \mathsf{col}_M(x) - \mathsf{rowcol}_M(x))}{v_K(x)} \tag{15}$$

and

$$q_{\mathsf{rowcolval}}(x) = \sum_{M \in \{A,B,C\}} \beta^M \cdot \frac{S_{K,K_M}(x) \cdot (\mathsf{rowcol}_M(x) \cdot \mathsf{val}_M(x) - \mathsf{rowcolval}_M(x))}{v_K(x)}, \quad (16)$$

then batch them into

$$q(x) = q_{\mathsf{rowcol}}(x) + \beta^3 \cdot q_{\mathsf{rowcolval}}(x). \tag{17}$$

→ Send to V the polynomials powA,α(x), powB,α(x), powC,α(x), h(x), g(x) and q(x).

<span id="page-42-0"></span><sup>41</sup>Since K<sup>M</sup> has size dM, which is a power of two which pads dinst,M + dstru,M , it is possible two or more among KA, K<sup>B</sup> and K<sup>C</sup> coincide with relative frequency. In that case, the corresponding power polynomials powM,α also coincide and do not need to be sent (in the compiled SNARG: committed) separately or have their correctness tested separately in the verifier's decision phase.

<span id="page-42-1"></span><sup>42</sup>The use of these polynomials, which always take the value 1 at the point 1 by definition, admits a small optimisation where the prover and verifier operate on (f(x) − 1)/(x − 1) for each of them instead. This reduces the number of queries by 3 and the number of points queried by 1.

<span id="page-42-2"></span><sup>43</sup>For α = 0, we set powα(1) = 1 for formal reasons.

{43}------------------------------------------------

## <span id="page-43-0"></span>4.4 Verifier

$$\mathbf{V}^{\mathtt{i},\mathbf{I}(\mathtt{i})}(\mathtt{x})$$
:

The verifier receives an instance <sup>x</sup> = (([[fM]], <sup>d</sup>f,M )f∈Fkey,M∈{A,B,C} , P) with oracle access to the witness polynomials and containing program P = (N,((ν1, ρ1), . . . ,(ν<sup>I</sup> , ρ<sup>I</sup> ))). It then operates as follows:

### 1) First round:

- ← Send to P an empty message[40](#page-41-2) ⊥ ∈ F 0 .
- → Receive from P the polynomials valA(x), valB(x) and valC(x).

### 2) Second round:

- ← Sample (α, β) R ←− F <sup>2</sup> and send it to P.
- → Receive from P the polynomials powA,α(x), powB,α(x), powC,α(x), h(x), g(x) and q(x).

### 3) Query phase:

• Sample γ R ←− F.

Before querying any polynomials, V must determine the (single) linear combination φbatch,inst(x) of indexed polynomials which will be queried. Imporant note: only in this query phase, we abuse notation by writing expressions such as

$$\overline{\varphi}_{\mathsf{row},\mathsf{inst},M}(x) = \overline{\varphi}_{\mathsf{row},\mathsf{inst},M}(x) + \overline{\omega}_{\mathsf{inst},\mathsf{row}} \cdot \overline{\alpha}_{\mathsf{inst},M} \cdot \varphi_{\mathsf{row},M_{\nu_i},r_{\mathsf{full}}}(x)$$

Such a formula simply means that the eAHP verifier is updating the coefficient for the label of φrow,Mνi ,rfull(x) in the running linear combination φrow,inst,M (x) by adding[44](#page-43-1) ωinst,row ·αinst,M ∈ F.

The computation of φrow,inst,M (x) proceeds as follows:

- For each M ∈ {A, B, C}, compute the density[45](#page-43-2) dstru,M of Mstru. Recall this is the same for A and B, namely the number of structural constraints in the program; and that dC,stru is the same magnitude minus the number of operands of the form const(0).
- Compute dinst,M = P<sup>I</sup> <sup>i</sup>=1 dMνi and the padded full-program density d<sup>M</sup> = ⌈ <sup>2</sup>dinst,M + dstru,M ⌉, then set K<sup>M</sup> = Kd<sup>M</sup> with distinguished generator ω<sup>M</sup> = ωd<sup>M</sup> .
- For M ∈ {A, B, C}, initialise αinst,M = ωinst,row = 1 ∈ F and ωinst,col = ω 1+N <sup>c</sup>full <sup>∈</sup> <sup>F</sup>.
- For M ∈ {A, B, C} and f ∈ Fmat = {row, col, val}, initialise φf,inst,M (x) as the zero linear combination of indexed polynomials.

<span id="page-43-1"></span><sup>44</sup>After compilation with a homomorphic PCS, this would translate to a corresponding operation on commitments. Note, however, that the verifier does not update the commitment to the linear combination instruction by instruction, but instead computes the final coefficients and then performs a single fixed-base MSM. Aside from the gains obtained from efficient MSM algorithms, this prevents the argument verifier from scaling the same commitment several times if the corresponding instruction appears more than once in the program.

<span id="page-43-2"></span><sup>45</sup>This step already necessitates the construction of the locations dictionary introduced a few lines below. The reason is that whether an operand ρ<sup>i</sup> introduces a new structural constraint or not depends on whether the variable it refers to (if any) has already appeared in the program or not. We only defer the exposition of the construction of said dictionary to step [iii\)](#page-45-0) below for clarity.

{44}------------------------------------------------

- For each command  $(\nu_i, \rho_i)$  in the program:
  - $\circ$  For  $M \in \{A, B, C\}$ , update

$$\overline{\varphi}_{\mathsf{row},\mathsf{inst},M}(x) = \overline{\varphi}_{\mathsf{row},\mathsf{inst},M}(x) + \overline{\omega}_{\mathsf{inst},\mathsf{row}} \cdot \overline{\alpha}_{\mathsf{inst},M} \cdot \varphi_{\mathsf{row},M_{\nu_i},r_{\mathsf{full}}}(x),$$

$$\overline{\varphi}_{\mathrm{col,inst},M}(x) = \overline{\varphi}_{\mathrm{col,inst},M}(x) + \overline{\omega}_{\mathrm{inst,col}} \cdot \overline{\alpha}_{\mathrm{inst},M} \cdot \varphi_{\mathrm{col},M_{\nu_i},c_{\mathrm{full}}}(x),$$

and

$$\overline{\varphi}_{\mathsf{val},\mathsf{inst},M}(x) = \overline{\varphi}_{\mathsf{val},\mathsf{inst},M}(x) + \overline{\alpha}_{\mathsf{inst},M} \cdot \varphi_{\mathsf{val},M_{\nu_i}}(x).$$

- o Update the counters: set  $\overline{\alpha}_{\mathsf{inst},M} = \overline{\alpha}_{\mathsf{inst},M} \cdot \alpha^{d_{\nu_i,M}}$  for  $M \in \{A,B,C\}$ ,  $\overline{\omega}_{\mathsf{inst},\mathsf{row}} = \overline{\omega}_{\mathsf{inst},\mathsf{row}} \cdot \omega^{r_{\nu_i}}_{r_{\mathsf{full}}}$  and  $\overline{\omega}_{\mathsf{inst},\mathsf{col}} = \overline{\omega}_{\mathsf{inst},\mathsf{col}} \cdot \omega^{c_{\nu_i}}_{c_{\mathsf{full}}}$ .
- Finally, batch the nine polynomial accumulators into a single polynomial

<span id="page-44-0"></span>
$$\overline{\varphi}_{\mathsf{batch},\mathsf{inst}}(x) = \sum_{M \in \{A,B,C\}} \sum_{\mathsf{f} \in \mathcal{F}_{\mathsf{mat}}} \beta^M \cdot \beta^\mathsf{f} \cdot \overline{\varphi}_{\mathsf{f},M}(x) \tag{18}$$

with

$$\beta^A = \beta^0$$
,  $\beta^B = \beta^1$ ,  $\beta^C = \beta^2$ ,  $\beta^{\mathsf{row}} = \beta^0$ ,  $\beta^{\mathsf{col}} = \beta^3$  and  $\beta^{\mathsf{val}} = \beta^6$ .

At the end of the computation, the accumulators  $\overline{\varphi}_{\mathsf{row},M}$ ,  $\overline{\varphi}_{\mathsf{col},M}$  and  $\overline{\varphi}_{\mathsf{val},M}$  are precisely the hybrid fingerprint polynomials

$$\widetilde{\varphi}_{\mathsf{row},\overline{\mathcal{M}},r_{\mathsf{full}},\alpha}(x) = \widetilde{\varphi}_{\mathsf{row},\mathcal{M},r_{\mathsf{full}},\alpha}(x),$$

$$\widetilde{\varphi}_{\mathsf{col},\overline{\mathcal{M}},c_{\mathsf{full}},\alpha}(x) = \omega_{c_{\mathsf{full}}}^{1+N} \cdot \widetilde{\varphi}_{\mathsf{col},\mathcal{M},r_{\mathsf{full}},\alpha}(x) \text{ and }$$

$$\widetilde{\varphi}_{\mathsf{val},\overline{\mathcal{M}},\alpha}(x) = \widetilde{\varphi}_{\mathsf{val},\mathcal{M},\alpha}(x)$$

respectively, with  $\mathcal{M} = (M_{\nu_i})_{i=1}^I \in \{\mathcal{A}, \mathcal{B}, \mathcal{C}\}$  and  $\overline{\mathcal{M}} \in \{\overline{\mathcal{A}}, \overline{\mathcal{B}}, \overline{\mathcal{C}}\}$  as in section 3.4.

The polynomial querying now proceeds as follows:

 $\circ$  Query all polynomials in the instance x at the point  $\gamma$ :

$$\mathsf{row}_A(x), \mathsf{col}_A(x), \mathsf{rowcol}_A(x), \mathsf{rowcolval}_A(x),$$
 $\mathsf{row}_B(x), \mathsf{col}_B(x), \mathsf{rowcol}_B(x), \mathsf{rowcolval}_B(x),$ 
 $\mathsf{row}_C(x), \mathsf{col}_C(x), \mathsf{rowcol}_C(x) \text{ and } \mathsf{rowcolval}_C(x).$ 

 $\circ$  Query all polynomials sent by  $\mathcal{P}$  in the first round at the point  $\gamma$ :

$$\mathsf{val}_A(x), \mathsf{val}_B(x) \text{ and } \mathsf{val}_C(x).$$

 $\circ$  Query all polynomials sent by  $\mathcal{P}$  in the second round at the point  $\gamma$ :

$$\mathsf{pow}_{A,\alpha}(x), \mathsf{pow}_{B,\alpha}(x), \mathsf{pow}_{C,\alpha}(x), h(x), g(x) \text{ and } q(x).$$

- For each  $M \in \{A, B, C\}$ , query the power polynomial  $pow_{M,\alpha}$  sent by  $\mathcal{P}$  in the second round at the points 1 and  $\omega_M^{-1} \cdot \gamma$ .
- Query the batched hybrid fingerprint polynomial  $\overline{\varphi}_{\mathsf{batch},\mathsf{inst}}(x)$ , which is a linear combination of indexed polynomials, at the point  $\alpha$ .

This amounts to

{45}------------------------------------------------

- 18 polynomials queried at the same point γ.
- 3 polynomials (powA,α(x), powB,α(x) and powC,α(x)) queried at three points each, the first one being the same γ as above, the second one being 1 and the third one being (potentially) different for each polynomial.
- 1 polynomial queried at α.

### 4) Decision phase:

In each of the checks described below, the verifier rejects if the tested equality (denoted ?=) does not hold.

<span id="page-45-1"></span>Set dmax = max{dA, dB, dC} and K = Kdmax .

i) Verification of the rowcol and rowcolval quotients: Verify that

$$q(\gamma) \cdot v_{K}(\gamma) \stackrel{?}{=} \sum_{M \in \{A,B,C\}} \beta^{M} \cdot S_{K,K_{M}}(\gamma) \cdot (\mathsf{row}_{M}(\gamma) \cdot \mathsf{col}_{M}(\gamma) - \mathsf{rowcol}_{M}(\gamma))$$

$$+ \beta^{3} \cdot \sum_{M \in \{A,B,C\}} \beta^{M} \cdot S_{K,K_{M}}(\gamma) \cdot (\mathsf{rowcol}_{M}(\gamma) \cdot \mathsf{val}_{M}(\gamma) - \mathsf{rowcolval}_{M}(\gamma))$$

$$(19)$$

with β<sup>M</sup> as in the query phase. All evaluations come from the query phase except for vK(γ) and SK,K<sup>M</sup> (γ), which can be computed with O(log(dmax)) squarings in F and will be reused in several subsequent steps.

<span id="page-45-5"></span>ii) Verification of the power-polynomial quotients: For M ∈ {A, B, C}, verify that

<span id="page-45-4"></span>
$$\alpha \cdot \mathsf{pow}_{M,\alpha}(\omega^{-1} \cdot \gamma) - \mathsf{pow}_{M,\alpha}(\gamma) \stackrel{?}{=} \frac{\alpha^{d_M} - 1}{d_M} \cdot v_{K_M \setminus \{1\}}(\gamma), \tag{20}$$

(cf. lemma [18\)](#page-62-1) and

<span id="page-45-3"></span>
$$\mathsf{pow}_{M,\alpha}(1) \stackrel{?}{=} 1.$$

All evaluations involved come from the query phase except for vKM\{1} (γ), which can be computed as (γ <sup>d</sup><sup>M</sup> −1)/(γ −1) if γ ̸= 1 and simply equals d<sup>M</sup> when γ = 1. Note that some of these values have likely already been computed when evaluating SK,K<sup>M</sup> (γ) in step [i\).](#page-45-1)

- <span id="page-45-0"></span>iii) Computation of structural fingerprints: Perform a single pass[46](#page-45-2) over the program to compute the fingerprints φrow,Mstru,rfull(α), φcol,Mstru,cfull(α) and φval,Mstru (α) for M ∈ {A, B, C}. By remark [16,](#page-37-1) the main computation is restricted to four values out of those nine. In preparation for this step, do the following:
  - Initialise an empty dictionary locations: VarIdentifiers → N. Let v1, . . . , v<sup>N</sup> be the identifiers of the variables in the Varuna instance – that is, the public input-output variables of the program P. For 1 ≤ i ≤ n, set locations[v<sup>i</sup> ] = i (location 0 is reserved for the global constant 1).
  - Initialise ωstru,row = 1 ∈ F and reset ωinst,col to ω 1+N <sup>c</sup>full <sup>∈</sup> <sup>F</sup>.

<span id="page-45-2"></span><sup>46</sup>In the compiled SNARG, this could be integrated in the same pass as the computation of the (commitments to) the hybrid fingerprint polynomials from the query phase. If done this way, the same counters ωinst,row and ωinst,col can be used for both purposes instead of resetting them as indicated in the second point below.

{46}------------------------------------------------

- Initialise  $\overline{\alpha}_{\mathsf{stru},C} = 1 \in \mathbb{F}$ .
- Initialise the fingerprint accumulators

$$\overline{\varphi}_{\mathsf{row},\mathsf{stru},C} = \overline{\varphi}_{\mathsf{col},\mathsf{stru},A} = \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} = \overline{\varphi}_{\mathsf{val},\mathsf{stru},C} = 0 \in \mathbb{F}.$$

Then, for each command  $(\nu_i, \rho_i)$  in the program, do the following:

• Accumulate the "local constant 1 equals global constant 1" constraint: set

$$\begin{split} \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} + \overline{\omega}_{\mathsf{stru},\mathsf{row}} \cdot \overline{\alpha}_{\mathsf{stru},C} \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},A} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},A} + \overline{\omega}_{\mathsf{inst},\mathsf{col}} \cdot \overline{\alpha}_{\mathsf{stru},AB} \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} + \overline{\alpha}_{\mathsf{stru},C} \\ \overline{\varphi}_{\mathsf{val},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{val},\mathsf{stru},C} + \overline{\alpha}_{\mathsf{stru},C} \end{split}$$

- Update the structure-related counters: set  $\overline{\omega}_{\mathsf{stru},\mathsf{row}} = \overline{\omega}_{\mathsf{stru},\mathsf{row}} \cdot \omega_{r_{\mathsf{full}}}$  and  $\overline{\alpha}_{\mathsf{stru},C} = \overline{\alpha}_{\mathsf{stru},C} \cdot \alpha$ .
- For each operand information  $\rho_{i,j} \in \rho_i = (\rho_{i,1}, \dots, \rho_{i,|\mathrm{IO}_{\nu_i}|})$ :
  - o If  $\rho_{i,j}$  is of the form var(v) and v is not a key in locations: Set  $|\text{locations}[v]| = \overline{\omega}_{\text{inst,col}} \cdot \omega_{c_{\text{full}}}^{\text{IO}_{\nu_i,j}}$ , where  $|\text{IO}_{\nu_i,j}|$  is the j-th element of  $|\text{IO}_{\nu_i}|$  as defined in subsection 3.2 (that is, the relative column position of the j-th operand in  $\nu_i$ 's R1CS). Then move onto the next operand, skipping the remaining steps in the inner loop.
  - $\circ$  Else if  $\rho_{i,j}$  is of the form const(k):

Accumulate the "operand equals constant (with value k)" constraint: set

$$\overline{\varphi}_{\text{col,stru},A} = \overline{\varphi}_{\text{col,stru},A} + (\overline{\omega}_{\text{inst,col}} \cdot \omega_{c_{\text{full}}}^{\text{IO}_{\nu_i,j}}) \cdot \overline{\alpha}_{\text{stru},AB}$$

Furthermore, if  $k \neq 0$ , also set

$$\begin{split} \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} + \overline{\omega}_{\mathsf{stru},\mathsf{row}} \cdot \overline{\alpha}_{\mathsf{stru},C} \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} + \overline{\alpha}_{\mathsf{stru},C} \\ \overline{\varphi}_{\mathsf{val},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{val},\mathsf{stru},C} + k \cdot \overline{\alpha}_{\mathsf{stru},C} \end{split}$$

and update  $\overline{\alpha}_{\mathsf{stru},C} = \overline{\alpha}_{\mathsf{stru},C} \cdot \alpha$ .

 $\circ$  Else (necessarily:  $\rho_{i,j}$  is of the form var(v) and v is a key in locations): Accumulate the "operand equals preexisting program variable (at column locations[v])" constraint:

$$\begin{split} \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} + \overline{\omega}_{\mathsf{stru},\mathsf{row}} \cdot \overline{\alpha}_{\mathsf{stru},C} \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},A} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},A} + \left( \overline{\omega}_{\mathsf{inst},\mathsf{col}} \cdot \omega_{c_{\mathsf{full}}}^{\mathsf{IO}_{\nu_i,j}} \right) \cdot \overline{\alpha}_{\mathsf{stru},AB} \cdot \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} + \mathsf{locations}[v] \cdot \overline{\alpha}_{\mathsf{stru},C} \\ \overline{\varphi}_{\mathsf{val},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{val},\mathsf{stru},C} + \overline{\alpha}_{\mathsf{stru},C}. \end{split}$$

Also set  $\overline{\alpha}_{\mathsf{stru},C} = \overline{\alpha}_{\mathsf{stru},C} \cdot \alpha$ .

- Update the remaining structure-related counter: set  $\overline{\omega}_{\mathsf{stru},\mathsf{row}} = \overline{\omega}_{\mathsf{stru},\mathsf{row}} \cdot \omega_{r_{\mathsf{full}}}$ .
- Update the instruction-related column counter: set  $\overline{\omega}_{inst,col} = \overline{\omega}_{inst,col} \cdot \omega_{c_{full}}^{c_{\nu_i}}$ .

After going through all program commands, computation of the structural fingerprints concludes as follows:

{47}------------------------------------------------

• Expand the four fingeprints into the expected nine by setting

$$\begin{split} \overline{\varphi}_{\mathsf{row},\mathsf{stru},A} &= \overline{\varphi}_{\mathsf{row},\mathsf{stru},B} = \sum_{i=0}^{d_{\mathsf{stru},A}-1} (\alpha \cdot \omega_{r_{\mathsf{full}}})^i \cdot \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},B} &= \sum_{i=0}^{d_{\mathsf{stru},B}-2} \alpha^i \\ \overline{\varphi}_{\mathsf{val},\mathsf{stru},A} &= \overline{\varphi}_{\mathsf{val},\mathsf{stru},B} = \sum_{i=0}^{d_{\mathsf{stru},A}-1} \alpha^i, \end{split}$$

where the sums can be computed with the usual geometric-sum formula (when the ratio is not 1).

• Accumulate the final "zero-knowledge" constraint: set αstru,A = αstru,B = α dstru,AB (note that the density of Astru and Bstru is equal to the number of rows of either) and

$$\begin{split} \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{row},\mathsf{stru},C} + \overline{\alpha}_{\mathsf{stru},C} \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},A} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},A} + \overline{\alpha}_{\mathsf{stru},A} \cdot \overline{\omega}_{\mathsf{inst},\mathsf{col}} \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},B} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},B} + \overline{\alpha}_{\mathsf{stru},B} \cdot \omega_{c_{\mathsf{full}}} \cdot \overline{\omega}_{\mathsf{inst},\mathsf{col}} \\ \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} &= \overline{\varphi}_{\mathsf{col},\mathsf{stru},C} + \overline{\alpha}_{\mathsf{stru},C} \cdot \omega_{c_{\mathsf{full}}}^2 \cdot \overline{\omega}_{\mathsf{inst},\mathsf{col}} \\ \overline{\varphi}_{\mathsf{val},\mathsf{stru},M} &= \overline{\varphi}_{\mathsf{val},\mathsf{stru},M} + \overline{\alpha}_{\mathsf{stru},M} \quad \text{for } M \in \{A,B,C\}. \end{split}$$

Observe that the zero-knowledge constraint part of the remaining fingerprints was added in the previous step.

• Pad the structural fingerprints and prepare them for addition to the instruction ones (cf. equuations [\(12](#page-36-0)row) - [\(12](#page-37-0)val) or definition [12](#page-28-2) [iii\)\)](#page-29-2) by setting

$$\begin{split} \overline{\varphi}_{\text{row,stru},A} &= \omega_{r_{\text{full}}}^{r_{\text{inst}}} \cdot \alpha^{d_{\text{inst},M}} \cdot \overline{\varphi}_{\text{row,stru},A} + \sum_{i=d_{\text{inst},M}+d_{\text{stru},M}}^{d_M-1} \alpha^i \\ \overline{\varphi}_{\text{col,stru},A} &= \alpha^{d_{\text{inst},M}} \cdot \overline{\varphi}_{\text{col,stru},A} + \sum_{i=d_{\text{inst},M}+d_{\text{stru},M}}^{d_M-1} \alpha^i \end{split}$$

for M ∈ {A, B, C}.

<span id="page-47-0"></span>iv) Computation of batched full-matrix fingerprint: Batch the structural fingerprints with the same coefficients as their instruction analogues and combine the two by setting

$$\Phi = \overline{\varphi}_{\mathsf{batch},\mathsf{inst}}(\alpha) + \sum_{M \in \{A,B,C\}} \sum_{\mathsf{f} \in \mathcal{F}_{\mathsf{mat}}} \beta^M \cdot \beta^\mathsf{f} \cdot \overline{\varphi}_{\mathsf{f},\mathsf{stru},M},$$

where the first term in the right-hand side comes from the query phase. The resulting value Φ is precisely the (batched) full-program fingerprint

$$\sum_{M \in \{A,B,C\}} \sum_{\mathsf{f} \in \mathcal{F}_{\mathsf{mat}}} \beta^M \cdot \beta^{\mathsf{f}} \cdot \varphi_{\overline{\mathsf{vec}}_{\mathsf{f},M_{\mathsf{full}}}}(\alpha).$$

{48}------------------------------------------------

#### <span id="page-48-3"></span>v) Verification of the batched sumcheck proof: verify that

$$\sum_{M \in \{A,B,C\}} \beta^{M} \cdot S_{K/K_{M}}(\gamma) \cdot \mathsf{pow}_{M,\alpha}(\gamma) \cdot \sum_{f \in \mathcal{F}_{\mathsf{mat}}} \beta^{\mathsf{f}} \cdot \mathsf{f}_{M}(\gamma) \stackrel{?}{=} h(\gamma) \cdot v_{K}(\gamma) + \gamma \cdot g(\gamma) + \frac{\Phi}{d_{\max}}, \tag{21}$$

where β<sup>M</sup> and β <sup>f</sup> are as in the query phase. Note all values therein come from the query phase or have been computed in previous steps.

## <span id="page-48-0"></span>4.5 Compilation into a preprocessed SNARG

Since eAHPvkproof is public coin, so is the preprocessing argument universal SRS compiled with any (separable homomorphic) PCS as described in subsection [2.4.](#page-19-0) Applying the Fiat-Shamir transform to the argument therefore results in a preprocessing SNARG.

We do not study the soundness of the latter, or indeed of the outcome of the double compilation process from a black-box eAHP into a SNARG ([\[Chi+20\]](#page-60-0) does not define a notion of AHP soundness aimed at Fiat-Shamir security). Observe, however, that our soundness analysis for the eAHP in section [5.1](#page-48-2) can be converted into an analysis of (intuitively) "round-by-round soundness", which is, in contexts where it is formally defined, a sufficient condition for the security of the Fiat-Shamir transform (cf. [\[BCS16\]](#page-61-0), [\[Can+19\]](#page-61-1), [\[Hol19\]](#page-61-2), [\[Blo+23\]](#page-61-3)).

#### Computing $p_1$

If, for some  $M \in \{A, B, C\}$ ,  $\mathsf{rowcol}_M^*$  and  $\mathsf{row}_M^* \cdot \mathsf{col}_M^*$  do not coincide over  $K_M$ , one has  $v_{K_M}(x) \nmid \mathsf{row}_M^*(x) \cdot \mathsf{col}_M^*(x) - \mathsf{rowcol}_M^*(x)$  or, equivalently,  $v_K \nmid S_{K,K_M}(x) \cdot (\mathsf{row}_M^*(x) \cdot \mathsf{col}_M^*(x) - \mathsf{rowcol}_M^*(x))$ . By a classic result (cf. for instance [GWC19] claim 4.6), one has

$$\Pr_{\beta \xleftarrow{R} \mathbb{F}} \left( v_K(x) \text{ divides } \sum_{M} \beta^M \cdot S_{K,K_M}(x) \cdot (\mathsf{row}_M^*(x) \cdot \mathsf{col}_M^*(x) - \mathsf{rowcol}_M^*(x)) \\ + \beta^3 \sum_{M} \beta^M \cdot S_{K,K_M}(x) \cdot (\mathsf{rowcol}_M^*(x) \cdot \mathsf{val}_M^*(x) - \mathsf{rowcolval}_M^*(x)) \right) \le \frac{5}{|\mathbb{F}|}.$$

## <span id="page-58-0"></span>6 Future work

We conclude by outline some natural directions to explore in future work:

- 1) Implement the protocol for instance, in the setting of snarkVM. This will, on one hand, offer a more concrete idea of the speedup achieved by verifying proofs of key synthesis compared to the current option of key re-generation; and, on the other, illustrate the challenges of adapting a complex, real-world language such as Leo (or Aleo instructions) to the clean, idealised formalism of modular compilation.
- 2) One could introduce into the protocol a separate class of "super-sparse instructions" which, due to their simplicity, can be more efficiently treated analogously to structural constraints (that is, having the verifier compute the value of their fingerprints on the fly) than to fullblown instructions (where the verifier has to operate on the indexed commitments to their fingerprint polynomials). A prime candidate for this is, for instance, the field-multiplication instruction mul, containing a single constraint.
- 3) Determine to what extent modular compilation enables faster proving-key generation[56](#page-58-1) . More specifically, just like the vkproof indexer precomputes fingerprint polynomials for each individual-instruction R1CS, it could also expose the plain row, col and val (possibly rowcol and rowcolval) polynomials of said R1CS[57](#page-58-2). A party wishing to generate the proving key for a specific program P could then combine the relevant precomputed polynomials using the

Only a small FFT would be needed in order to incorporate the information about Astru. One may optimistically expect a speedup from the O(d · log(d)) of current proving-key generation to O(d) using modular compilation and precomputation, but more careful analysis is needed in order to substantiate any complexity claims.

Note that this pursuit is independent of the vkproof eAHP and should instead be regarded as a positive side-effect of switching to modular compilation.

- 4) Extend the protocol to attest to the correctness of not only verifying keys, but also proving ones. This would open the door to proving-key generation as a service, where a party interested in a program P's proving keys may delegate their synthesis to a different entity and simply verify a succinct proof of honest key generation. On one hand, the usefulness of this is somewhat unclear as proving keys are of interest mainly to proving parties which typically have the necessary resources to directly synthesise the proving key by themselves. On the other hand, the changes to the vkproof protocol needed to encompass this functionality are minimal: simply move the full Varuna-key polynomials into the instance and replace the opening of their commitments in the original vkproof by direct evaluation. This has a cost of O(d), with d denoting the density of the full-program R1CS, as opposed to the O(d · log(d)) asymptotics of proving-key synthesis.
- 5) Study which other existing languages and R1CS proof systems our techniques can be applied to without substantially departing from the current design. As already mentioned, vkproof encompasses both the compilation step and the proof system itself, which means it is likely only extendable to scenarios where both components exist. Marlin-based proof systems are good candidates for exploration.
- 6) Formally define a notion of knowledge soundness and zero knowledge for the eAHP primitive and study its behaviour under compilation with a PCS, which should likely follow the template in [\[Chi+20\]](#page-60-0). This has not been treated in this text since it is not directly relevant to vkproof, as already explained.

## <span id="page-62-0"></span>Appendix A Power polynomials

Let  $n \in \mathbb{N} \setminus \{0\}$  and  $\mathbb{F}$  be a field whose unit group  $\mathbb{F}^*$  contains a subgroup H of order n, for which we fix a distinguished generator  $\omega$ . Given  $\alpha \in \mathbb{F}$ , let  $\mathsf{pow}_{\alpha}(x) \in \mathbb{F}^{< n}[x]$  be the unique polynomial of degree less than n which maps  $\omega^i$  to  $\alpha^i$  for  $0 \le i < n$  with the convention that  $\mathsf{pow}_0(1) = 1$ .

We denote by  $\hat{v}_H(x) = (x^n - 1)/(x - 1) = \sum_{i=0}^{n-1} x^i \in \mathbb{F}^{-n-1}[x]$  the vanishing polynomial over  $H \setminus \{1\}$ .

#### <span id="page-62-1"></span>Lemma 18.

i) The relation

<span id="page-62-7"></span>
$$\alpha \cdot f(\omega^{-1} \cdot x) - f(x) = \frac{\alpha^n - 1}{n} \cdot \hat{v}_H(x), \tag{24}$$

is satisfied by  $f = pow_{\alpha}$ .

<span id="page-62-2"></span>ii) If  $\alpha \notin H$ , then<sup>58</sup>  $pow_{\alpha}(x)$  is the unique polynomial  $f \in \mathbb{F}^{< n}[x]$  of degree less than n satisfying (24) and it is given by

<span id="page-62-4"></span>
$$\mathsf{pow}_{\alpha}(x) = \frac{\alpha^n - 1}{n} \cdot \sum_{i=0}^{n-1} \frac{1}{\alpha \cdot \omega^{-i} - 1} \cdot x^i. \tag{25}$$

- <span id="page-62-3"></span>iii) If  $\alpha \in H$ , then  $pow_{\alpha}(x) = x^{j}$ , where j is the unique integer  $0 \leq j < n$  satisfying  $\alpha = \omega^{j}$ . Furthermore, the polynomials satisfying (24) are exactly the polynomials of the form  $t \cdot x^{j}$  with  $t \in \mathbb{F}$ .

$$\frac{\alpha \cdot \mathsf{pow}_{\alpha}(\omega^{-1} \cdot x) - \mathsf{pow}_{\alpha}(x)}{\hat{v}_{H}(x)} = q(x)$$

for some  $q(x) \in \mathbb{F}[x]$ . The numerator has degree at most equal to the denominator (i. e. n-1) and hence q(x) is in fact a constant  $k \in \mathbb{F}$ . Substituting x=1 in the above expression yields

$$k = \frac{\alpha^n - 1}{\hat{v}_H(1)} = \frac{\alpha^n - 1}{n}.$$

$$T_{\alpha} \colon \mathbb{F}^{< n}[x] \xrightarrow{\sim} \mathbb{F}^{< n}[x]$$

$$f(x) \mapsto \alpha \cdot f(\omega^{-1} \cdot x) - f(x)$$

$$\sum_{i=0}^{n-1} c_i \cdot x^i \mapsto \sum_{i=0}^{n-1} (\alpha \cdot \omega^{-i} - 1) \cdot c_i \cdot x^i$$

$$(26)$$

is injective as  $(\alpha \cdot \omega^{-i} - 1)$  does not vanish for any i. If α ∈ H is of the form ω j for some 0 ≤ j < n, we distinguish two cases:

• If f is not of the form t · x j for any t ∈ F, then Tα(f) ̸= Tα(powα(x)) by iii) and the same argument as above yields

$$\Pr_{\gamma \xleftarrow{R} \mathbb{F}} \left( \alpha \cdot f(\omega^{-1} \cdot \gamma) - f(\gamma) = \frac{\alpha^n - 1}{n} \cdot \hat{v}_H(x) \text{ and } f(1) = 1 \right) \le \frac{n - 1}{|\mathbb{F}|}$$

• If f is of the form t·x j for some t ∈ F, then t ̸= 1 (since by assumption f ̸= powα(x) = x j ) and thus

$$\Pr_{\gamma \stackrel{R}{\longleftarrow} \mathbb{F}} \left( \alpha \cdot f(\omega^{-1} \cdot \gamma) - f(\gamma) = \frac{\alpha^n - 1}{n} \cdot \hat{v}_H(x) \text{ and } f(1) = 1 \right) \leq \Pr_{\gamma \stackrel{R}{\longleftarrow} \mathbb{F}} \left( f(1) = 1 \right) = 0.$$

## <span id="page-64-0"></span>Appendix B Compilation with hiding polynomial commitment schemes

In the body of the text, we have restricted our attention to compilation of extended algebraic holographic proofs with polynomial commitments that are not hiding. This appendix briefly illustrates the motivation behind that choice and informally discusses the modifications needed to incorporate hiding PCS. There are multiple reasons one may be interested in allowing hiding polynomial commitment schemes in the eAHP paradigm. For instance, one may want to add a notion of zero-knowledge to eAHPs and the preprocessing arguments compiled from them. As a second, subtler reason, one may want to use the eAHP + PCS compilation process to obtain a preprocessing argument for an existing relation capturing an already-deployed proof system whose instance commitments come from a hiding PCS[59](#page-64-1) (and, in particular, whose SRS includes information for producing hiding commitments).

Suppose given an indexed polynomial-oracle relation (cf. definition [1\)](#page-9-1), where instances can contain oracles to witness polynomials; and a polynomial commitment scheme PC as in [\[Chi+20\]](#page-60-0) section 6.1, where commitments can incorporate hiding randomness. There are at least two natural ways to approach the definition of the compiled relation RPC, by contrast with the non-hiding case (definition [8\)](#page-19-3):

<span id="page-64-3"></span>i) Require that the commitments (ci) n <sup>i</sup>=1 in the instance have the form

$$(c_i)_{i=1}^n = \mathsf{PC.Commit}(\mathsf{ck}, (f_i)_{i=1}^n, (\mathsf{d}_i)_{i=1}^n, (\bot)_{i=1}^n),$$

where ⊥ denotes empty randomness.

<span id="page-64-2"></span>ii) Require that the commitments (ci) n <sup>i</sup>=1 in the instance have the form

$$(c_i)_{i=1}^n = \mathsf{PC.Commit}(\mathsf{ck}, (f_i)_{i=1}^n, (\mathsf{d}_i)_{i=1}^n, (\boldsymbol{\omega_i})_{i=1}^n)$$

for some randomness (ωi) n <sup>i</sup>=1.

Option [ii\)](#page-64-2) results in a largely trivial relation as the hiding property means that commitments carry no information per se. For instance, in the Marlin PCS ([\[Chi+20\]](#page-60-0) appendices B-D), fixing a polynomial f ∈ F[x] and sampling hiding randomness ω R ←− F uniformly at random yields a uniformly random distribution on the resulting commitment (as long as the toxic waste γ from the second portion of the SRS is non-zero). This means that even the definition of plain soundness for the resulting preprocessing argument would need to incorporate a knowledge-soundness-like condition on the witness polynomials f.

We favour option [i\),](#page-64-3) where instance commitments do carry information about witness polynomials by themselves and the unmodified definition of argument soundness remains meaningful. Furthermore, it is this option that captures the VarunaVK relation in practice, where the Varuna (not vkproof) indexer produces (Varuna) verifying keys by committing to proving-key polynomials with empty randomness.

One possible approach is to separate the PCS in instance commitments from that used for compilation of the rest of the eAHP (i. e. for indexed and round polynomials). However, there can

- The definition of a PCS (with hiding randomness) incorporates two[60](#page-65-0) new methods: OpenNH and CheckNH (where NH stands for "non-hiding").
