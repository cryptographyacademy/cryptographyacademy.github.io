## **What a Wonderful World: zkSNARKs in the Algebraic Group Model are Universally Composable**

Gaspard Anthoine1*,*<sup>2</sup> , Dario Fiore<sup>1</sup> , and Mahak Pancholi<sup>1</sup>

> 1 IMDEA Software Institute, Madrid, Spain <sup>2</sup> Universidad Politécnica de Madrid, Madrid, Spain

**Abstract.** Zero-Knowledge Succinct Non-interactive Arguments of Knowledge (zkSNARKs) are important cryptographic primitives critical in many real-world applications. zkSNARKs are not used in isolation but are deployed within a broader context in which other cryptographic protocols may be concurrently executed. Universal-Composability (UC) allows rigorous analysis of cryptographic primitives being used in such arbitrary contexts. A UC analysis is even more desirable for popular, well-audited, and heavily deployed zkSNARKs already being used in practice.

Prior works that study the UC security of existing zkSNARKs (without modifications) are either not modular, hence requiring case-by-case analysis for new proof systems, or have largely focused on zkSNARKs in the Random Oracle Model (ROM). The latter includes zkSNARKs with logarithmic proof sizes compiled from Interactive Oracle Proofs. This state of the art leaves out a large family of very efficient, often constant-size, zkSNARKs that rely on the Algebraic Group Model (AGM) and optionally on the ROM. This includes zkSNARKs compiled from Polynomial Interactive Oracle Proofs, such as Plonk and Marlin, among others.

In this work, we address the UC security for unmodified zkSNARKs that are proven secure in AGM (+ROM). Our approach is modular: we identify simple, and mostly standard properties on the underlying zkSNARK that imply UC security. We observe that checking these properties for existing zkSNARKs is a surprisingly simple task using the rigorous formulation of AGM from Jaeger and Mohan (CRYPTO'24). The simplicity and modularity of our framework makes it easy-to-use for concluding UC security of several zkSNARKs in the same setting. Concretely, using our framework we establish that Plonk and Marlin are UC secure without any overhead.

## **1 Introduction**

Zero-knowledge Succinct Non-interactive Arguments of Knowledge (zkSNARKs) are one of the most useful cryptographic ideas that are being deployed in several real-world applications. Their prevalence necessitates optimal security guarantees. Furthermore, zkSNARKs are not used in isolation but are often composed with other cryptographic primitives within a larger application. In such settings, it is of utter importance to ensure that the interaction of zkSNARKs with concurrently-executed arbitrary protocols or their composition within the context of a bigger protocol do not compromise their security.

The Universal Composability (UC) framework [\[21\]](#page-25-0) addresses exactly this problem. It enables the modular design and analysis of cryptographic protocols and guarantees security even against adversaries that run multiple concurrent sessions and mount arbitrary attacks.

Prior attempts at achieving UC-secure zkSNARKs usually lift or transform an existing zkSNARK to make it secure in the UC framework [\[5,](#page-24-0) [37\]](#page-26-0). These approaches substantially alter the underlying scheme, incurring significant overhead, and do not address many of the existing (unaltered) zkSNARKs commonly used in practice.

This paper focuses on UC security of existing zkSNARKs without modifications. Necessary ingredients for UC security of zkSNARKs are *straight-line* (SLE) and *black-box* extraction (BBE), and *simulation-extractability* (SIMEXT). Informally, a proof system is SLE if a valid witness can be extracted without interacting with the adversary (hence no rewinding); it is BBE if the extractor works without knowing the code of the adversary; and it is SIMEXT if it is *non-malleable*, i.e., an

{1}------------------------------------------------

adversary cannot maul an existing proof to forge a new proof on some statement without knowing the corresponding witness.

Groth [\[42\]](#page-27-0) showed in a modular manner that if a (non-succinct) NIZK in the CRS model satisfies the aforementioned properties then it is UC-secure in the CRS-hybrid model. Succinct proof systems with straight-line black-box extraction, however, cannot exist in the plain CRS-model alone [\[20,](#page-25-1) [39\]](#page-26-1); instead they are extractable in some idealised model. Such models include the random oracle model (ROM), or an idealised group model such as the generic group model (GGM[3](#page-1-0) ) [\[59,](#page-27-1)[61\]](#page-28-0) or the algebraic group model (AGM) [\[34,](#page-26-2)[45\]](#page-27-2). Nonetheless, in such idealised models, several modern zkSNARKs have recently been shown to satisfy SLE, SIMEXT, and sometimes BBE [\[32,](#page-26-3) [33,](#page-26-4) [38,](#page-26-5) [49\]](#page-27-3). This brings up the natural question:

*Following [\[42\]](#page-27-0), can we prove in a modular manner the UC security of existing zkSNARKs proven extractable in an idealised model?*

To further motivate this question, let us briefly summarize the state of the art on SLE of existing zkSNARKs. We can broadly classify most of them in two categories.

**Category 1: zkSNARKs with SLE in the ROM.** This includes purely ROM-based zkSNARKs, such as Killian-Micali [\[48,](#page-27-4) [57\]](#page-27-5), and the ones compiled from Interactive Oracle Proof (IOP) using a Merkle tree [\[10–](#page-25-2)[12\]](#page-25-3). These zkSNARKs achieve logarithmic proof sizes at best.

The zkSNARKs in this category happen to naturally satisfy BBE and SLE via observability in the ROM. Leveraging this observation, two recent works [\[22,](#page-25-4) [26\]](#page-26-6) addressed the UC security of these zkSNARKs, using an hybrid UC model where the ROM is formulated as a global programmable functionality (global-ROM). Concretely, both works show UC security for the Kilian–Micali zk-SNARK [\[56\]](#page-27-6), while [\[26\]](#page-26-6) additionally considers zkSNARKs obtained by compiling an IOP with Merkle-tree commitments via the BCS compiler [\[12\]](#page-25-3).

**Category 2: zkSNARKs with SLE in an idealised group model (+ROM).** This includes constructions such as [\[29,](#page-26-7) [43,](#page-27-7) [51\]](#page-27-8), zkSNARKs obtained from polynomial interactive oracle proofs (PIOPs) using extractable polynomial commitment schemes [\[17,](#page-25-5)[19,](#page-25-6)[27,](#page-26-8)[36,](#page-26-9)[54,54\]](#page-27-9), as well as transparent zkSNARKs relying on discrete-log–type assumptions [\[14,](#page-25-7) [15,](#page-25-8) [60\]](#page-27-10) [4](#page-1-1) . These schemes are also among the most efficient and succinct systems we have today, with many zkSNARKs in this category having constant proof size unlike those in Category 1.

For these zkSNARKs, the essential properties for UC crucially depend on an idealised group model rather than the ROM. Hence, techniques in [\[22,](#page-25-4) [26\]](#page-26-6) are not applicable here.

The recent work of Bobolz et al. [\[13\]](#page-25-9) is the first to address the UC security of a proof system in Category 2. They do so in a hybrid UC model where the GGM is formulated as a global group functionality (global-GGM). Now, via observing group queries, one can recover BBE and SLE as in Category 1. Concretely, they prove that Groth16 [\[43\]](#page-27-7) is UC-secure without any modifications, under this new model.

While [\[13\]](#page-25-9) provides a clear blueprint for establishing UC security in the GGM, it has three major limitations. First, it is a purely GGM-based model while several zkSNARKs additionally require the ROM. Second, it requires the prerequisite security properties of the zkSNARK to be defined and proven in GGM, whereas the common practice is to establish these in the AGM (+ROM). This means that to show UC security for Groth16, the analysis of SIMEXT has to be rewritten in the GGM, and cannot simply invoke the existing theorem proven in the AGM [\[7\]](#page-24-1). And finally, their analysis is not modular. In particular, the framework does not offer a modular way to leverage

## **1.1 Our Results**

Our main result is answering the aforementioned question positively, i.e., zkSNARKs satisfying straight-line simulation-extractability in the AGM+ROM are UC-secure.

1. **UC lifting theorem:** The work of [\[13\]](#page-25-9) introduces a very useful UC framework meant to work well with AGM-based zkSNARKs. However, it is quite non-trivial to use since it essentially requires most proofs to be redone from scratch within their framework. Our main result leverages the observation that AGM-based zkSNARKs are *already* very close to UC: We identify simple properties for zkSNARKs in the stand-alone model formulated in GGM (+ROM) that imply UC security in the presence of a global-GGM +(global-ROM) functionality. Surprisingly, these properties closely mirror standard notions for zkSNARKs. This has the added benefit that potentially one does not need to prove *new* properties to obtain UC security.

<span id="page-2-2"></span>**Theorem 1.** *(Informal.) If a zkSNARK is complete, non-interactive zero-knowledge, and straight-line and black-box simulation-extractable in GGM (+ROM), then it is UC-secure in the presence of global-GGM (+non programmable global-ROM) functionality.*

Given that most analyses exist in AGM (+ROM), the next question is how to lift these analyses to GGM without having to redo them. We do this in two steps:

2. **Lifting guarantees in AGM+ROM to "type-safe" (GGM+ROM):** As a stepping stone toward a general lifting theorem, we first show how to lift such guarantees to an intermediate model, called the *type-safe*[5](#page-2-0) *(GGM+ROM)*. While [\[45\]](#page-27-2) presents a similar lifting result, their framework does not allow algorithms to access a random oracle and is therefore not directly applicable to the zkSNARK constructions considered in this work. We observe that a similar hierarchy to that of [\[45\]](#page-27-2) can be defined even in the case where algorithms require access to ROM. At first sight, it is natural to ask:

*How can a random oracle taking a bitstring as input be used in a restricted model where group elements do not even have a concrete representation?*

To circumvent this limitation, we first introduce a new type-safe random oracle and show that it can be canonically translated into a "classical" random oracle. Leveraging these observations, we show how existing security analyses carried out in the AGM+ROM can be largely reused to establish security in (our variant of) type-safe (GGM+ROM).

<span id="page-2-1"></span><span id="page-2-0"></span><sup>5</sup> A type-safe GGM is closer to Maurer's GGM [\[55\]](#page-27-11) where group elements are stored in a table and group operations can be performed only via their position in the table. As such no concrete group representation is available.

{3}------------------------------------------------

**Theorem 2.** *(Informal.) The security of a primitive proven in the AGM+ROM can be lifted to the type-safe (GGM+ROM) provided that the reduction is (type-safe) generic.*

3. **Type-safe (GGM+ROM) and GGM+ROM equivalence:** The work of [\[62\]](#page-28-1) establishes an equivalence between the type-safe GGM and GGM for single-stage adversaries, and for primitives that can be expressed in the type-safe model. Building on similar observations, we show an analogous equivalence between our variant of the type-safe (GGM+ROM) and GGM+ROM. Combining this equivalence with [Theorem 2](#page-2-1) yields a clear framework for establishing stand-alone security properties in GGM+ROM. Finally, applying our UC lifting theorem [Theorem 1,](#page-2-2) one can conclude UC security.

<span id="page-3-0"></span>**Theorem 3.** *(Informal.) For primitives that exist both in type-safe (GGM+ROM) and GGM+ROM, security against single-stage adversaries is equivalent.*

**How to use our framework.** We apply our framework to show that Plonk [\[36\]](#page-26-9) and Marlin [\[27\]](#page-26-8) are UC-secure in the presence of global-GGM and (non-programmable) global-ROM functionalities. The work of Faonio et al. [\[33\]](#page-26-4) establishes that both Plonk and Marlin, without modifications, are SLE SIMEXT in the (non-programmable) ROM and the AGM. Starting from this result, we first apply our lifting result [\(Theorem 2\)](#page-2-1) to SIMEXT of the same protocols in our variant of the type-safe (GGM+ROM). Using the equivalence result [\(Theorem 3\)](#page-3-0), we then conclude that these zkSNARKs are SLE SIMEXT in GGM+ROM. Completeness and NIZK follow similarly. Since these properties are the prerequisite for our UC lifting theorem [\(Theorem 1\)](#page-2-2), we ultimately obtain UC security for Plonk and Marlin in the presence of global-GGM and (non-programmable) global-ROM functionalities.

## **2 Technical Overview**

## **2.1 UC lifting theorem**

Our goal is to identify properties in the stand-alone setting that are sufficient for proving UC. In this sense, we are closer to [\[26\]](#page-26-6) in our methodology than [\[13,](#page-25-9) [22\]](#page-25-4). While ideally, one would like to model AGM behaviour in UC to be closer to the stand-alone setting, the UC-AGM model [\[1\]](#page-24-2) is not suitable for non-interactive extractable primitives like zkSNARKs (see [Appendix A\)](#page-29-0). The next best option is to model GGM behaviour in UC as done in [\[13\]](#page-25-9).

**Proving UC security via game hops.** We give a short recap on the common approach for showing UC security for zkSNARKs in the literature. UC security is proven by showing indistinguishability between a real-world and an ideal-world experiment. For a zkSNARK *Π*, the real-world experiment is defined as: the honest parties in a session *sid* execute the (prove,verify) protocols specified in *Π* while having access to the global and (if any) local functionalities. While in the ideal-world experiment, no protocol is executed. Instead the honest parties access an ideal NIZK functionality that, at a very high level, behaves as follows: On receiving a prove command from an honest party, run a simulator algorithm on just the statement to generate a proof string (which ensures zero-knowledge). On receiving a verify command from an honest party, run the extractor on the proof string, and check if the output is a valid witness. If it is valid, then output 1, otherwise output 0 (which ensures knowledge-soundness). In addition to this, a simulator algorithm is responsible for simulating the view of the corrupted party in *sid*. This includes simulating interaction with honest parties (as in the real world) and access to the local functionality.

A natural approach towards proving UC-security of a zkSNARK is via doing standard game-hops (some examples where this shows up are [\[26,](#page-26-6) [37,](#page-26-0) [42\]](#page-27-0)) that go as follows:

{4}------------------------------------------------

- **–** Hyb<sup>0</sup> : This is the real-world experiment in UC.
- **–** Hyb<sup>1</sup> : The prove interface is replaced by the zero-knowledge simulator.
- **–** Hyb<sup>2</sup> : The verifier interface is swapped by running the extractor on the proof string to obtain a witness and checking whether the witness is valid or not. This is exactly like the ideal world UC experiment.

To argue indistinguishability between each hop, one would typically invoke an underlying stand-alone property (i.e. zero-knowledge for Hyb<sup>0</sup> -Hyb<sup>1</sup> , and simulation-extractability for Hyb<sup>1</sup> -Hyb<sup>2</sup> ).

Consider the reduction for Hyb<sup>0</sup> -Hyb<sup>1</sup> hop. For simplicity, assume that the parties only have access to a global-ROM (and not global-GGM). Assume that there exists an adversary Z who can successfully distinguish between Hyb<sup>0</sup> *,* Hyb<sup>1</sup> . Then, we build an adversary A given black-box access to Z against the zero-knowledge property. At a high level, A runs Z internally and is responsible for simulating its view. This includes answering prove,verify commands, and simulating accesses global-ROM, while interacting with the challenger in the zero-knowledge game in the presence of a standard ROM. A can simply forward the prove and verify commands to the underlying game and return the responses. To simulate responses to global-ROM given access to standard ROM, A proceeds as follows: forward all *sid* queries to the standard ROM in the underlying game, simulate all *sid*′ ≠ *sid* queries locally, keep a local record of observable queries (this is possible since it sees all queries made by Z), and responds appropriately when asked for the observable set[6](#page-4-0) . Since global-ROM has a very easy domain separation and *sid*′ responses are independent of *sid*, A perfectly simulates Z's view. Hence, any distinguishing advantage of Z implies the same advantage for A in distinguishing real proofs from simulated ones.

**Extending to group-based zkSNARKs.** For group-based zkSNARKs, we not only need to model a global-ROM but also a global-GGM functionality. Global-GGM models Shoup's GGM [\[61\]](#page-28-0) as a global functionality. In Shoup's GGM, the group is modelled as a random mapping of Z*<sup>p</sup>* to bit strings, and group operations can only be performed via queries to the GGM oracle.

The game hops follow similarly as above. However, building a reduction to an underlying property requires care. Since group elements are not independent for each session the domain separation is not as easy as in the case of global-ROM. As a result, in our reduction, we can no longer simulate *sid*′ group queries locally, but all queries regardless of the session must be forwarded to the GGM oracle in the underlying game. Here, an issue arises when trying to simulate the observability interface of the global-GGM functionality. To understand the issue, first consider the mechanism of observability defined in [\[13\]](#page-25-9).

In [\[13\]](#page-25-9), every generator for some session *sid* is associated with a formal variable X*sid*. A group element that is the result of a group operation now has an associated polynomial: for example, a group operation of the form *h* ← *g*<sup>1</sup> + *g*<sup>2</sup> has the polynomial X<sup>1</sup> + X<sup>2</sup> associated with it, where X1*,* X<sup>2</sup> are the formal variables previously defined for *g*1*, g*2, respectively. A group operation is labelled as illegal and hence observable if the polynomial associated with the result happens to have any foreign session variables X*sid*′ or a constant term. In short, operations involving other session's generators become observable.

In our reduction, the group operations performed by the algorithms in the underlying game (i.e., prove and verify interfaces) will not be visible to A. This means that, while simulating observability for global-GGM, A might not be able to classify legal vs illegal queries, or might classify them differently than if it had the full representation. An example of this is: Let *gsid*′ be the generator of *sid*′ . On receiving a prove command from Z, A forwards it to the prove interface of the underlying

<span id="page-4-0"></span><sup>6</sup> Global functionalities often have an Observe interface that allows to observe "illegal" queries that Z might make to force the simulator in *sid* to fail.

{5}------------------------------------------------

game, and receives a proof string (*π*1*, . . . , πn*) in return, where assume that *π*1*, π*<sup>2</sup> are group elements. Later, Z makes a group query of the form *gsid*′ + *a* · *π*<sup>1</sup> + *b* · *π*2. Looking at it naively, A classifies this group operation as illegal since it contains group elements from different sessions (and because it does not know the polynomials associated with *π*1*, π*2). However, with the full polynomial representation of *π*1*, π*<sup>2</sup> perhaps the *sid* terms cancel out, and hence will not be classified as illegal in the hybrid game.

We observe that the (symbolic) group oracle provides a natural mechanism to access the underlying polynomial representation. This means that maintaining partial observability is not a problem for A. It can still determine observability of the future group elements by accessing the underlying polynomial representation via appropriate group queries to the GGM oracle. This allows it to verify if a query cancels out the"foreign" variables associated with other sessions *sid*′ . Concretely, it makes the group queries corresponding to the operation *a* · *π*<sup>1</sup> + *b* · *π*<sup>2</sup> and checks if the result equals to the canonical generator (hence the *sid* terms cancel out). If yes, then classify as legal, and otherwise mark as observable.

A related issue arises when reducing to simulation-extractability in the last hop. In the case of global-ROM alone, the argument is that the extractor in the underlying game has an identical view as that of the extractor in the hybrid (only queries related to *sid*). Hence, they succeed or fail with the same probability. However, in our reduction, the extractor in the game has full view of the queries made to the group oracle (since A forwards everything), while in the hybrid game it only has *sid* related views. We argue that if the generators for each session are chosen independently and if whenever the honest protocol samples a group element it samples it with high min-entropy, then the additional information available in the underlying game is not more useful. Hence, the success probability can now be related identically.

What we have shown above is that the usual game-hop strategy follows through even in the case of global-GGM with mild assumptions on the zkSNARK, and without drastically modifying the game-based properties. The game-based properties must exist in GGM+ROM for our reduction to work. However, most known simulation-extractability analyses exist in the AGM+ROM. To avoid having to redo analysis in Shoup's GGM (as is done in [\[13\]](#page-25-9) for Groth16; simulation-extractability for Groth16 was already known in the AGM in a previous work [\[7\]](#page-24-1)) and avoid subtle security gaps [\[64\]](#page-28-2), we present a framework for lifting proofs in the AGM+ROM to Shoup's GGM+ROM next.

## **2.2 AGM+ROM to GGM+ROM lifting**

**Type-Safe model of computation.** Maurer [\[55\]](#page-27-11) introduced an alternate GGM framework where the group elements are stored in a table and group operations are abstracted as having only access to *handles* representing an index in the table kept by an oracle. Subsequently in a seminal work, Zhandry [\[62\]](#page-28-1) showed that relying on such a table might lead to undesirable behaviours and introduced the notion of Type-Safety using a circuit representation where handles are wires containing values in Z*p*.

The notion of type-safety was further formalized in [\[45\]](#page-27-2) using a typed pseudocode representation of the algorithms. In this framework, each variable in the algorithm is assigned a data type; ⟨bit⟩ for a bit string, and ⟨elem⟩ for a group element. Data types restrict how an algorithm can interact with specific variables. Additionally, they have access to functions that perform group operations by accepting ⟨elem⟩ variables as input, a function *σ* : ⟨elem⟩ → ⟨bits⟩ that outputs a concrete representation of a group element, and *σ* −1 : ⟨bit⟩ → ⟨elem⟩ that outputs a group element given its representation. In "type-safe GGM", algorithms are only allowed to access group operation functions (meaning that they cannot obtain concrete representation for group elements); while in "type-safe 

{6}------------------------------------------------

AGM" they additionally have access to *σ* but not to *σ* −1 (meaning that they cannot generate group elements locally without using functions for group operations).

They use their framework to rigorously formalize common intuitions about the AGM (we will call this the type-safe AGM). Under this formalization, they characterize conditions under which security in type-safe AGM can be lifted to type-safe GGM. The combination of [\[45\]](#page-27-2) and [\[62\]](#page-28-1) already lays the necessary groundwork for links between variants of AGM and GGM. There are a few things missing to be applicable to zkSNARKs, which we discuss next.

**Type-Safe Model with Random Oracle.** Many cryptographic primitives that can be expressed in Shoup's GGM cannot even be expressed in the type-safe GGM [\[41,](#page-26-11) [62\]](#page-28-1) because in the latter model the algorithms do not know group element representation. One type of computation that cannot be expressed is random oracle queries. To make a query to a standard ROM, the zkSNARK algorithm must use the concrete group element representations [\[41\]](#page-26-11). This also holds for the type-safe pseudocode framework of Jaegar and Mohan [\[45\]](#page-27-2).

We resolve this by introducing a type-safe variant of random oracle. Providing a type-safe algorithm with access to a full random oracle that accepts group elements as input may raise concerns. Specifically, *what prevents the algorithm from using the random oracle to derive an encoding for a group element?*

While this is indeed the case if we see the oracle as a random injection *σ*ˆ, we prove in [Section 6.1](#page-21-0) that it is possible to define a similar hierarchy as the one of Jaeger and Mohan [\[45\]](#page-27-2) for algorithms with access to type-safe random oracle. Overall the argument relies on the fact that, as long as the algorithm can't compute the inverse *σ*ˆ −1 , group elements will only come from group queries similarly to the Type-Safe AGM and thus preserves the model's security guarantees.

**Type-Safe Model to Shoup's GGM.** Zhandry [\[62\]](#page-28-1) showed that the two models are not equivalent as was formerly believed. Security in Shoup's GGM is strictly stronger for games where the adversary is split into several stages without sharing full internal state. On the other hand, for "single-stage" adversaries, i.e., adversaries that pass full internal state between stages, and for primitives that can be expressed in the type-safe GGM, the security in both models of GGM is equivalent.

We complete the sequence of implications by showing a similar equivalence between the new type-safe model with access to type-safe random oracle introduced in this work, and Shoup's GGM with access to a standard random oracle. As a result, the type-safe random oracle turns out to be a stepping stone towards proving security in the standard random oracle model, and is never used by the final algorithms.

## **2.3 Plonk and Marlin are UC-secure**

As an application of our framework we show that Plonk and Marlin are UC-secure. The main "non-blackbox" task here is to check that the reductions (in [\[33\]](#page-26-4)) to a computational problem are all type-safe generic. Intuitively since such reductions are written in the algebraic group model, they very likely satisfy the formal requirement as well. Nonetheless, in [Appendix H,](#page-42-0) we reason that this is indeed the case. After this, we can in a black-box manner lift these results all the way up to UC.

## **2.4 Limitations**

An important criticism of the generic group model is that it allows to prove certain (contrived) schemes secure that cannot be securely instantiated under any group in the standard model [\[30\]](#page-26-12). Moreover, security in GGM only holds against generic adversaries. What would be ideal is security against standard adversaries since a known class of sub-exponential time attacks on the discrete-log assumption are via index calculus algorithms which make use of the specific group structure. ## **2.5 Related Works**

We present a more complete survey of related-works in [Appendix A.](#page-29-0)

**UC-secure zkSNARKs in global-ROM.** Ganesh et al. [\[37\]](#page-26-0) were the first to present a UC-secure zkSNARK with constant-sized proofs in the CRS-model. Badertscher et al. [\[5\]](#page-24-0) build upon [\[37\]](#page-26-0) to achieve a UC zkSNARK in the transparent model. Both works are in the non-programmable global-ROM, and present a compiler that can lift an underlying zkSNARK with appropriate properties to a UC-secure one. The works of [\[22,](#page-25-4) [26\]](#page-26-6) study UC security of existing zkSNARKs without modifications. In the programmable global-ROM (programmability seems necessary for such unmodified zkSNARKs), both show that Killian-Micali zkSNARK is UC-secure, while [\[26\]](#page-26-6) additionally shows UC-security for certain IOP-based zkSNARKs obtained via the BCS transform [\[12\]](#page-25-3). Their constructions are secure against adaptive adversaries.

The work of [\[26\]](#page-26-6) is closer to our work in terms of methodology as they too aim to identify stand-alone properties that imply UC security. However, they are only able to realise a weaker NIZK functionality. Informally the weaker functionality is not required to extract if the statement has already been proven in this session. On the other hand, [\[22\]](#page-25-4) models all subprotocols directly in UC, for example, by introducing functionalities for non-interactive (vector) commitment schemes, and building a UC-secure zkSNARK from these.

For simplicity, we only consider static adversaries in this work, leaving the formalisation as an open problem.

**UC-secure zkSNARKs in global-GGM.** To tackle AGM-based zkSNARKs, Bobolz et al. [\[13\]](#page-25-9) present a new UC framework modelling the group as a global generic functionality with restricted observability. Under this new model, they prove Groth16 [\[43\]](#page-27-7) to be UC-secure. Since Groth16 is malleable, it can only realise a weak form of NIZK functionality, while in this paper we aim for the standard NIZK functionality.

**AGM to GGM lifting results.** A "lift" of a security reduction allows one to infer hardness in a target model based on a reduction against adversaries in a source model. First introduced in the original AGM [\[34\]](#page-26-2) to infer security in the GGM from an AGM reduction, a subsequent counter-example [\[64\]](#page-28-2) demonstrated that this implication is not guaranteed. This motivated the work of [\[45\]](#page-27-2) that established a formal framework to derive a hierarchy of idealised models.

The differences compared to G are as follows: Instead of sampling injections,  $\tau_i'$ s are now lazily sampled. Nonetheless, similar to [13], we can argue that the probability of distinguishing is negligible, meaning that, one can always replace G with s-G and expect security to hold (Lemma 1<sup>7</sup>).

## **3.2 Succinct non interactive zero-knowledge proofs**

**Indexed Relation.** An indexed relation R is a set of tuples of the form (i*, x, w*), where i denotes the index, *x* the instance, and *w* the witness. Typically, i describes an arithmetic circuit, *x* is a public wire assignments, *w* denotes the rest of the private inputs, and (i*, x, w*) ∈ R iff the circuit determined by i outputs 1 on inputs (*x, w*).

**(Succinct) non-interactive zero knowledge argument of knowledge.** A zkSNARK for an indexed relation R is a tuple of PPT algorithms *Π* = (*Setup, I, P, V* ) that work as follows. All algorithms have oracle access to RO*,* G,

- **–** pp ← *Gen*(1*<sup>λ</sup>* ): On input security parameter, it outputs a set of public parameters consisting of group generators. We assume that *Gen* will sample group generators by sampling at random from sets *S*1*, S*2, where *S*1*, S*<sup>2</sup> denote the source groups.
- **–** crs ← *Setup*(p): On input p, it outputs the setup string crs.
- **–** (ipk*,* ivk) ← *I*(crs*,* i): On input an index i and crs, it deterministically outputs an index-specific proving key ipk and a verification key ivk.
- **–** *π* ← *P*(ipk*, x, w*): On input ipk, statement *x* and the witness *w*, it outputs a proof string *π*.
- **–** *b* ← *V* (ivk*, x, π*) On input ivk*, x, w* it outputs a decision bit.

We require a zkSNARK to satisfy completeness, non-interactive zero-knowledge (NIZK), and simulation-extractability (SIMEXT). Completeness means that if a proof string is generated by running the prover's algorithm honestly, then the verifier accepts except with negligible probability. NIZK intuitively means that a curious verifier does not learn anything from the proof string apart from the fact that the statement is in the language. Formally, this requires the existence of a simulator algorithm that can simulate a proof string by just knowing the statement and some (witness-independent) trapdoor information. Since we use standard definition for these, we present these definitions in [Appendix B.](#page-29-2) Below we describe the syntactic changes in the NIZK definition relevant to our work.

**NIZK.** In this work, we consider NIZK in the *non-programmable* random oracle model. Hence, we give direct access to RO in both games. In both experiments, the distinguisher A has access to a "prove" and a "verify" interface. As usual, in the "real" game NIZK-0, the prove interface is the honest prover algorithm, while in the "ideal" game NIZK-1 it the simulator's algorithm (*S*<sup>1</sup> is simply a wrapper around *S*). The verify interface just implements the honest verifier algorithm in both games. We include this explicit since it will be needed in the reduction from a UC adversary to a NIZK adversary in later sections. We also do not explicitly write an *S*<sup>0</sup> simulator algorithm that generates the setup string in NIZK-1. Instead we assume that the honest setup algorithm already outputs trapdoors that can be used by the simulator to simulate proofs in NIZK-1.

**Simulation-Extractability.** This is a stronger notion than the more standard knowledge-soundness definition. It essentially guarantees that proofs are *non-malleable*, i.e., no efficient adversary can maul an existing proof observed during the execution and use it to forge a new proof without knowing the corresponding witness. This is crucial in the context of UC, since without non-malleability the environment Z can ask the honest parties/simulator to generate a proof *π* on an arbitrary (*x, w*) pair, and then maul *π* and submit this for verification. In the real world, the verification would succeed, however, in the ideal world, the simulator/extractor will fail to extract. This would allow Z to distinguish between the real and the ideal world experiments.

## 4 Preliminaries: Universal Composability

#### 4.1 Global random oracle

We consider a hybrid model where all parties have access to a global non-programmable random oracle ( $\mathcal{G}_{RO}$ ) from [16,23].  $\mathcal{G}_{RO}$  can be queried by any party and the environment directly using two interfaces: Query and Observe. On Query,  $\mathcal{G}_{RO}$  lazily samples a random value, and otherwise replies by looking up the response in its local table. On Observe, it replies with the set of illegal queries made so far. Each  $\mathcal{G}_{RO}$  query can be parsed as (sid, x). A query (sid, x) is marked as illegal if it is made by a party in a session  $sid' \neq sid$ . Informally, no honest party ever makes such a query, while a corrupt party or the environment may try to make the simulator/extractor in sid fail by making all  $\mathcal{G}_{RO}$  queries outside the session. However, in the latter case, the simulator/extractor will now have access to these queries via Observe. The decsription of the standard  $\mathcal{G}_{RO}$  functionality can be found in Appendix C.

Note that, [26] and [22] require a programmable  $\mathcal{G}_{RO}$  since they consider IOP based zkSNARKs which requires programming for showing zero-knowledge. We only require observable  $\mathcal{G}_{RO}$  as we focus on PIOP based zkSNARKs such as Plonk and Marlin, which are compiled using variants of KZG [46] polynomial commitments. The zero-knowledge simulator in this case simulates via knowing the trapdoor rather than programming.

#### 4.2 Global generic group functionality with restricted observability

We show UC security in the presence of the global generic group functionality (Functionality 5) from [13]. In our proof, in Section 5, we will primarily use the symbolic version of  $\mathcal{G}$ . Hence, here we discuss  $\mathcal{G}$  informally, moving the formal description to Appendix C.

In addition to group operations as in Figure 1,  $\mathcal{G}$  must manage obervability. The observability rules are defined via two rules  $\mathsf{Legal}^{\mathsf{OP}}_{sid}$  and  $\mathsf{Legal}^{\mathsf{PAIR}}_{sid}$ . Each group element seen during the execution is maintained as a formal variable or a polynomial in these variables. Whenever a query is made with a group element g in a session sid that has not been seen before, the functionality registers it by assigning a fresh variable for it (via Touch), and stores it in  $\mathsf{R}_i[g]$ . g will essentially be treated as a generator for session sid. This is used by (honest) protocols to generate random generators for their execution session.

When an  $\operatorname{OP}_{sid}$  query is made, the functionality computes and adds the polynomial corresponding to the result to  $\mathsf{R}_i[h]$  (for example,  $\mathsf{R}_i[h] \cup a_1\mathsf{R}_i[g_1] + a_2\mathsf{R}_i[g_2]$ ). Then, it checks if  $\exists \mathsf{f} \in \mathsf{R}_i[h]$  such that  $\mathsf{f}$  contains any variable from a session  $sid' \neq sid$  or a non-zero constant term (membership in  $\mathsf{Legal}_{sid}^{\mathsf{OP}}$ ). If this is the case, then the query is added to the observable set. This enforces the following behaviour: to not be observable a party in session sid must operate only on the group elements that are derived from the group generators in sid. Group operations with the canonical generator (which results in a constant term) are also considered illegal.

When a PAIR<sub>sid</sub> query is made, the functionality checks the membership of f in Legal<sup>PAIR</sup><sub>sid</sub> where in addition to the conditions in Legal<sup>OP</sup><sub>sid</sub> it also checks that any degree 1 term in f only comes from the target group variables. This rules out pairing operations with the canonical generator. [13] considers these operations as legal. However, the choice is somewhat arbitrary. We decide to add this constraint as it enforces a behaviour that is closer to the expected honest behaviour, i.e., honest parties only operate on group elements that are derived from the group generators in sid.

#### 4.3 Global symbolic group functionality with perfect session separation

As explained in [13], an issue with  $\mathcal{G}$  is that it has imperfect domain separation, i.e., some group elements may belong to two sessions. Ideally, this should amount to an error event in practice. To

resolve this, [13] proposes a purely symbolic version of  $\mathcal{G}$  (denoted by s- $\mathcal{G}$  in this work, Functionality 1), which admits perfect session separation.

The main differences between the two functionalities are in how the map  $\tau$  is defined and in the enforcement of domain separation to define observability. In  $\mathcal{G}$ , the internal maps  $\tau_i$  assigns random group elements to scalar values in  $\mathbb{Z}_p$ , and use a table  $R_i$  to keep track of every possible representation of a group element to determine observability. In contrast, the Symbolic Global Generic Group functionality (s- $\mathcal{G}$ ) modifies the domain of  $\tau_i$  to map formal polynomials in  $\mathbb{Z}_p[\mathsf{X}_{sid},\dots]$  directly to group elements. Since the number of symbolic polynomials is unbounded, it is impossible to take a random injection. Instead,  $\tau_i$  are lazily sampled in  $S_i \setminus \mathsf{img}(\tau_i)$  (i.e. without replacement). Considering polynomial time adversaries, the total amount of queries are  $\mathcal{O}(\mathsf{poly}(\lambda)) \ll |S|$  and thus the functionality never runs out of elements. By assigning unique formal variables to generators, s- $\mathcal{G}$  enforces a different representation for all the elements and therefore allows perfect equality check using the encoding:

$$\tau_i(f(X...)) = \tau_i(f'(X...)) \iff f(X...) = f'(X...)$$

Looking ahead, using a symbolic functionality simplifies the analysis for reducing UC to standalone properties since the above equality check ensures that the adversary built in the reduction behaves identically to s- $\mathcal{G}$  would in the hybrid games.

We use a simplified version of s-G. In particular, we do not require the extra internal interfaces available to the simulation (e.g., ComputeSymbolic, etc), or Laurent polynomials. This is because, we reduce to the underlying stand-alone property in a black-box manner (instead of showing an explicit proof of simulation-extractability as is shown in [13] to argue indistinguishability between hybrids  $H_1$  and  $H_2$  in Claim 5). Because if this, we do not need to track the internal computation of the simulator or it's randomness (the latter requires Laurent polynomials).

It was shown in [13] that replacing  $\mathcal{G}$  with its symbolic counterpart s- $\mathcal{G}$  is indistinguishable to all algorithms making up to Q oracle queries.

## **5.1 Stand-alone protocol to UC-secure protocol**

Let *Gen* be the algortihm producing public group parameters pp given the description of the group, and let *Π* = (*Setup, I, P, V* ) be a non-interactive argument of knowledge in the presence of G+non-programmable ROM. A natural transformation of *Π*, denoted by *Π* **= (Setup***,* **I***,* **P***,* **V)** given below is the corresponding protocol in the UC-framework in the Fsetup, G*,* GRO-hybrid. #### <span id="page-17-0"></span>5.2 Adversary's definition

Before arguing indistinguishability between each hybrid, we informally describe the adversary  $\mathcal{A}^*$  against some property based definition with interfaces (Setup, P, V). Given black-box access to  $\mathcal{Z}$  in the hybrids,  $\mathcal{A}^*$  breaks a property of the underlying zkSNARK. The adversary's algorithm remains the same in each hop, hence we define it only once. The full formal description can be found in Appendix D.

The overall reduction strategy is as follows: Let  $\mathcal{Z}$  be a UC adversary that can successfully distinguish between two consecutive hybrids  $\mathsf{Hyb}_i, \mathsf{Hyb}_{i+1}$ .  $\mathcal{A}^*$  is given black-box access to  $\mathcal{Z}$  and is responsible for generating indistinguishable view for  $\mathcal{Z}$  as in the hybrid games while interacting with the challenger of an underlying property based security notion and with access to a symbolic Shoup's GGM oracle s-G (Figure 4) and a standard random oracle RO.  $\mathcal{A}^*$  will act as a wrapper around  $\mathcal{Z}$  and forward any  $\mathcal{G}_{\mathsf{RO}}$  query with  $\mathit{sid}$  prefix to RO, while locally simulating the responses for  $\mathit{sid}'$  via lazy sampling. When  $\mathcal{Z}$  asks for  $\mathcal{G}_{\mathsf{RO}}.\mathsf{OBSERVE}$ ,  $\mathcal{A}^*$  can correctly respond to this since it sees all queries made by  $\mathcal{Z}$  and hence it can maintain the set of illegal queries. All s- $\mathcal{G}$  queries, regardless of the session will be forwarded to s-G. To be able to respond to  $\mathcal{G}_{\mathsf{RO}}.\mathsf{OBSERVE}$ ,  $\mathcal{A}^*$  can additionally maintain its own version of tables  $\tau$  by registering each new group element with a fresh variable. This exactly matches with the tables maintained by s-G (apart from variables having different representations).

When an INIT<sub>sid</sub> command is received,  $\mathcal{A}^*$  interacts with the challenger (Ch) of underlying game to receive the crs for current execution. Later, when a prove or a verify query is made by  $\mathcal{Z}$ ,  $\mathcal{A}^*$  passes these on to the prove/verify interfaces of the underlying security game. These commands cause an issue with maintenance of the group observability sets. Unlike the real UC experiment,  $\mathcal{A}^*$  only have partial access to the groups representations. More specifically, it doesn't have access to the representation for any internal group element computed by the algorithms of Ch.

However, the reduction relies on the observation that the adversary does not need the full polynomial representation of honest elements to distinguish legal from illegal queries. It only requires the ability to verify if a query cancels out the foreign" variables associated with other sessions sid'. Concretely,  $\mathcal{A}^*$  decomposes operations into components depending on known adversarial variables and unknown honest proof variables. It then performs local checks via group oracle queries to determine which components of the polynomial vanishes, thereby correctly identifying illegal queries. For example, let  $\pi$  be a group element in the output of a Prove query. Since  $\mathcal{A}^*$  does not know how it was computed, it just assigns a fresh variable  $X_{\pi}$  for it. Later, assume that  $\mathcal{Z}$  makes an  $\operatorname{OP}_{sid'}(i,g_1,g_2,a_1,a_2)$  query such that  $g_1$  can be decomposed by  $\mathcal{A}^*$  as a polynomial  $a \cdot \mathsf{X}_{sid} + b \cdot \mathsf{X}_{\pi}$ , and  $g_2$  as  $\mathsf{Y}_{sid'}$ . Let g be the group element corresponding to the variable  $\mathsf{X}_{sid}$ . To correctly decide observability for session sid',  $\mathcal{A}^*$  needs to check that all terms corresponding to sid must be 0. It can do this by checking if  $\mathsf{s}\text{-}\mathsf{G}$ .  $\mathsf{OP}(i,g,\pi,a,b)$  returns a value corresponding to 0 (i.e. the result is equal to the encoding for 0).

{18}------------------------------------------------

Completing the informal description of the adversary: to determine observability,  $\mathcal{A}^*$  decomposes the resultant polynomials exactly as specified in hybrids  $\mathsf{Hyb}_4\mathsf{-Hyb}_8$  but in terms of variables it stores locally. Instead of checking if some specified term of polynomial is non-zero (as is done in each hybrid), it perform group operations corresponding to this term and checks if the result is an encoding of 0.

*Proof.* The difference in  $\mathsf{Hyb}_{9+j+1}$  is that it runs SIMULATE instead of prover's algorithm to generate proof string. The code of  $\mathcal A_{\mathsf{NIZK}}$  is the same that of  $\mathcal A^*$  described in Section 5 and Appendix D.

In the rest of the proof we argue the indistinguishability of the view generated by  $\mathcal{A}_{\mathsf{NIZK}}$ .

Consider the view generated for b = 0 (the same argument holds for b = 1): It is clear from observation that when  $\mathcal{Z}$  does not make any INIT<sub>sid</sub> or PROVE<sub>sid</sub> command then the view generated by  $\mathcal{A}_{\text{NIZK}}$  is identical to the one in  $\text{Hyb}_{9+j}$  (since all group operations would be observed by  $\mathcal{A}_{\text{NIZK}}$ ). Hence, the trace of intermediate group operations is not available to  $\mathcal{A}^*$ . Instead,  $\mathcal{A}^*$  assigns fresh variables for any group element present in the output of (INIT<sub>sid</sub>, PROVE<sub>sid</sub>) queries and adds them to the set  $X_{\pi}$ . On subsequent group queries  $\mathcal{A}^*$  decides whether or not to add the group operation tuple to Ob using  $X_{\pi}$  (instead of the full representation in terms of X that would be available in  $\text{Hyb}_{9+j}$ ). This may possibly lead to misclassification of group operation tuples: tuples marked as illegal in  $\text{Hyb}_{9+j}$  might not be marked as such  $\mathcal{A}^*$ , and vice versa. This may happen only in the two following cases:

Case 1: Collisions with Internal Variables. If  $\mathcal{Z}$  makes a group query containing a group element g such that g has not been seen before by  $\mathcal{A}_{\mathsf{NIZK}}$  but was a part of an intermediate group operation by (Setup, P) computation. In  $\mathsf{Hyb}_{9+j}$ , the group functionality would define the polynomial representation of g with respect to sid variables. However, in the view generated by  $\mathcal{A}_{\mathsf{NIZK}}$ , g would be defined w.r.t the session in which the query first appears. This could be in  $sid' \neq sid$ . Hence, the contents of Ob may now differ which can be observed by  $\mathcal{Z}$  by calling Observe. This case is however already ruled out in  $\mathsf{Hyb}_3$ .

Case 2: Algebraic Dependencies. ## Linear Operations in $sid: h \leftarrow OP_{sid}(i, (g_1, a_1), (g_2, a_2)).$

The bulk of the argument below is that the polynomial representation for h defined by s- $\mathcal{G}$  in  $\mathsf{Hyb}_{9+j}$  and by  $\mathcal{A}_{\mathsf{NIZK}}$  can be split into parts depending only on sid' variables and the rest, where the part depending on sid' is identical for both  $\mathsf{Hyb}_{9+j}$  and  $\mathcal{A}_{\mathsf{NIZK}}$ . Since observability in this case depends solely on checking sid' variables, the behaviour of  $\mathcal{A}_{\mathsf{NIZK}}$  and  $\mathsf{Hyb}_{9+j}$  is identical.

Let  $\hat{X}$  be the set of variables defined by stand-alone symbolic Shoup's GGM functionality s-G on group operation queries made either by (Setup, P) algorithms or sid queries forwarded by  $\mathcal{A}_{NIZK}$ . Let  $\hat{Y}$  be the set of variables defined on group operation queries forwarded by  $\mathcal{A}_{NIZK}$  for session sid'. Clearly,  $X_{sid'} \leftrightarrow \hat{Y}$  (there is a bijection between these two variable sets). Moreover,  $X_{sid} \subseteq \hat{X}$  and each  $X \in X_{\pi}$  can be rewritten as a linear polynomial  $p(\hat{X})$  with zero constant term  $(\deg_{\hat{X}}(p) = 0)$ . Barring observability, this is identical to how these polynomials and variables would be defined even in the UC group functionality s- $\mathcal{G}$ .

Let the polynomials corresponding to elements  $g_1, g_2$  as defined by  $\mathcal{A}_{\mathsf{NIZK}}$  be  $\mathsf{p}_{g_i}(\mathsf{X}_{sid}, \mathsf{X}_\pi, \mathsf{X}_{sid'})$ , where both polynomials are linear. The symbolic representation of the output h is defined as  $a_1\mathsf{p}_{g_1} + a_2\mathsf{p}_{g_2}$ , which is also a linear polynomial and hence can be rewritten as  $\mathsf{f} := \mathsf{f}^{(1)}(\mathsf{X}_{sid}, \mathsf{X}_\pi) + \mathsf{f}^{(2)}(\mathsf{X}_{sid'})$ , where  $\mathsf{f}^{(1)}$  only depends on variables  $\mathsf{X}_{sid}, \mathsf{X}_\pi$ , and  $\mathsf{f}^{(2)}$  contains the rest. Because of the observation above,  $\mathsf{f}^{(1)}(\mathsf{X}_{sid}, \mathsf{X}_\pi)$  can be equivalently written as  $\mathsf{p}(\hat{\mathsf{X}})$  (for some linear polynomial  $\mathsf{p}$  with zero constant term), and  $\mathsf{f}^{(2)}(\mathsf{X}_{sid'})$  can simply be rewritten with variables  $\hat{\mathsf{Y}}$  keeping the coefficients unchanged. The representation of the output h becomes:

$$p_h(\hat{X}, \hat{Y}) = p(\hat{X}) + f^{(2)}(\hat{Y})$$

Note that this is exactly how the polynomial representation for h would be defined in  $\mathsf{Hyb}_{9+j}$ . Since  $\mathcal{A}_{\mathsf{NIZK}}$  maintains the full symbolic representation for all previous queries w.r.t sid', it can compute  $\mathsf{f}^{(2)}$  explicitly and locally verify if it is a zero polynomial or not. Hence, deciding observability for  $\mathsf{OP}_{sid}$  queries is identical in  $\mathsf{Hyb}_{9+j}$  and in the view generated by  $\mathcal{A}_{\mathsf{NIZK}}$ .

## Linear Operations in sid': $h \leftarrow OP_{sid'}(i, (g_1, a_1), (g_2, a_2))$ .

Following the argument for the previous case, polynomial representation for h as defined in  $\mathsf{Hyb}_{9+j}$  would be  $\mathsf{p}_h(\hat{\mathsf{X}},\hat{\mathsf{Y}}) = \mathsf{p}(\hat{\mathsf{X}}) + \mathsf{f}^{(2)}(\hat{\mathsf{Y}})$  while for  $\mathcal{A}_{\mathsf{NIZK}}$  it is  $\mathsf{f}^{(1)}(\mathsf{X}_{sid},\mathsf{X}_\pi) + \mathsf{f}^{(2)}(\mathsf{X}_{sid'})$ . We have  $\mathsf{f}^{(1)}(\mathsf{X}_{sid},\mathsf{X}_\pi) = \mathsf{p}(\hat{\mathsf{X}})$ , and that the map  $\tau$  is defined identically by  $\mathcal{A}_{\mathsf{NIZK}}$  and  $\mathsf{s}\text{-}\mathsf{G}$  (and hence by the group functionality in  $\mathsf{Hyb}_{9+j}$ ). Thus,  $\mathsf{f}^{(1)}(\tau(\mathsf{X}_{sid}),\tau(\mathsf{X}_\pi)) = \tau(0)$  iff  $\mathsf{p}(\hat{\mathsf{X}})$  is a zero polynomial. Hence, deciding observability is again identical for  $\mathcal{A}_{\mathsf{NIZK}}$  and  $\mathsf{Hyb}_{9+j}$ .

$$\mathsf{p}_h(\hat{\mathsf{X}},\hat{\mathsf{Y}}) := \mathsf{p}_1(\hat{\mathsf{X}}) + \mathsf{p}_2(\hat{\mathsf{Y}}) \cdot \mathsf{p}_3(\hat{\mathsf{X}}) + \mathsf{p}_4(\hat{\mathsf{Y}})$$

where after substituting the variables appropriately,  $p_1$  is equivalent to  $f^{(1)}$ ;  $p_2$  to  $f^{(3)}$ ,  $p_3$  to  $f^{(4)} + f^{(5)}$ , and  $p_4$  is equivalent to the term  $f^{(6)}$ .

*Proof.* The difference between the hybrids is that the extractor may fail to output a valid witness in  $\mathsf{Hyb}_{9+\mathsf{s}_1+j+1}$  while the verifier accepts. This we reduce to simulation-extractability of  $\Pi$ . Given  $\mathcal{Z}$  who distinguishes between the two hybrids, we build an adversary  $\mathcal{A}_{\mathsf{SE}}$  whose code is the same as  $\mathcal{A}^*$ . The argument that  $\mathcal{A}$  generates an indistinguishable view is the same as already made in the NIZK case.

If a proof verifies in the UC experiment, it will also verify in the SIMEXT experiment (since all oracle queries are forwarded). All we need to argue is that the view of the extractors  $(E_0, E_1)$  in the SIMEXT experiment is identical to that of the extractors in the UC experiment in hybrid  $Hyb_{9+s_1+j+1}$ . Hence, if it fails to extract a valid witness in the latter, then this fails even in the SIMEXT, breaking the assumption. The difference in the view of the extractors in the two experiments is that in SIMEXT game, the extractors learn the entire group oracle transcript (including sid' queries), while in the hybrid they only learn Ob which is a strict subset. Hence, we must rule out the possibility of valid extraction given this extra information.

The inputs to  $E_0$  is identical in both experiments and so the output  $(i, h_j, B_j)$  outputs are identical. We claim that the outputs of FINDREP $(\cdot, Ob_{sid})$  and FINDREP $(\cdot, \mathsf{tr}_{\mathsf{G}})$  are identical. Given this, the output of  $E_1$  in both experiments will be identical.

Recall how the algorithm FINDREP $(1/2, \cdot)$  works: it parses  $B := (B_1, \ldots, B_n) \in S_i^n$ . Then it initializes a vector Rep as  $Rep[h] \leftarrow 0^n, \forall h \in S_i$ , and for  $j \in [n], Rep[B_j] = (0, \ldots, 1, 0, \ldots, 0)$  where  $Rep[B_j]$  is 1 at position j and 0 otherwise. These steps remain identical for both experiments. The next step is to execute  $\forall ob = (OP, i, g_1, g_2, a_1, a_2, h) \in tr$ , update  $Rep[h] := a_1 Rep[g_1] + a_2 Rep[g_2]$ , where tr is the query tuples taken as input. A difference may now occur if Rep[h] is set differently given access to a larger tr.

Let  $\operatorname{tr}_{\mathsf{G}}$  consist of all group queries ever made in the SIMEXT experiment. This includes all queries by the adversary  $\mathcal{A}_{\mathsf{SE}}$  and the ones internal algorithms of the SIMEXT challenger (by Setup, P, V interfaces). We can partition  $\operatorname{tr}_{\mathsf{G}}$  as  $\operatorname{tr}_{ob}||\operatorname{tr}_{se}||\operatorname{tr}_{\bar{ob}}$ , where  $\operatorname{tr}_{ob}$  denotes queries in Ob (as maintained by  $\mathcal{A}_{\mathsf{SE}}$ ),  $\operatorname{tr}_{se}$  are queries by SIMEXT interfaces, and  $\operatorname{tr}_{\bar{ob}}$  the rest (this only includes legal queries for sid' session maintained by  $\mathcal{A}_{\mathsf{SE}}$ ). Notice that this partitioning is perfect as the symbolic representation of a group element is guaranteed to be unique and that in the hybrid experiment, query tuples given as input to FINDREP will be equal to  $\operatorname{tr}_{ob}||\operatorname{tr}_{se}$ , while in the reduction it will be  $\operatorname{tr}_{ob}||\operatorname{tr}_{se}||\operatorname{tr}_{\bar{ob}}$ . We will proceed by contradiction to show that  $\forall (\operatorname{OP}, i, g_1, g_2, a_1, a_2, h) \in \operatorname{tr}_{\bar{ob}}, \operatorname{Rep}[h] = 0$ . If there exists h such that  $\operatorname{Rep}[h] \neq 0$  then it means that  $\operatorname{Rep}[g_1] \neq 0$  and/or  $\operatorname{Rep}[g_2] \neq 0$ , meaning that  $g_1$  and/or  $g_2$  must be computed via bases elements. Hence, contradicting the fact that h is not observable.

## 6 Inferring GGM security from AGM security

Most security analysis are in FKL-AGM, i.e. in the AGM model defined by Fuchsbauer, Kiltz, and Loss in [34]. Although this model is meant to be close to Shoup's's GGM one cannot always conclude security in the GGM given security in the AGM as shown in [62,64]. However, [45] showed that this link holds between AGM and Maurer's GGM and to formalise it they introduce a pseudocode computation framework. In this section, we extend this framework to include accesses to random oracle, hence making it applicable to zkSNARKs. Then, we complete the picture by lifting the security in this variant of GGM to Shoup's GGM.

#### <span id="page-21-0"></span>6.1 Type Safe Oracle Framework

**Data-types.** Data types restrict how an algorithm can interact with specific variables. We consider two data types  $\langle bit \rangle$  and  $\langle elem \rangle$  that can be seen as containing the discrete logarithm in the group.  $\langle bits \rangle$  represents a sequence of bits of arbitrary size.

**Typed Random Oracle.** A typed random oracle defined in Definition 3 is a natural generalization of the classical ROM. In this model, the oracle can be queried with group elements directly rather than first having to convert them to their representation in bits. This means that a query can be deterministically parsed as a series of group elements and bit strings. While this may seem arbitrary, this assumption already appears implicitly in prior works [9,18,35], where the ROM is defined as  $\mathbb{G} \times \{0,1\}^* \to \{0,1\}^{\lambda}$  (with minor modifications).

<span id="page-21-1"></span>**Definition 3 (Typed Random Oracle).** A typed random oracle is  $\mathsf{RO}_{\mathsf{ts}} : ((\langle \mathit{elem} \rangle e, \langle \mathit{bits} \rangle s)^*) \to \langle \mathit{bits} \rangle r$  that takes a list of tuple  $e \in \mathbb{G} \cup \{\emptyset\}$  and a bit string  $s \in \{0,1\}^* \cup \{\emptyset\}$  and outputs a uniformly sampled bit string r of size n.

It is easy to see that given a typed-random oracle we can generalize the pseudocode framework to algorithms that require access to a random oracle. Intuitively, this does not break the "type-safeness" because the random-oracle itself is typed, and can be directly queried on group elements rather than requiring the algorithm to interpret group elements as bit strings (which would be the case in classical random oracles). Notice that this means that a typed random oracle queried on the bitstring encoding or on an element would yield different output which is normal as they should be seen as two different inputs.

In many cases, typed-random oracles can also easily replace the usual random oracles without affecting the security, or requiring modifications to the algorithms in the idealised settings (as we will see later in the case studies for Plonk and Marlin). A real-world example where this is not the case is ECDSA [58], whose signer algorithm necessarily requires concrete representation of the group elements.

## 7 Implications and Open Questions

Implications of our framework. We sketch in Appendix H how we conclude UC security for two important proof systems – Plonk [36] and Marlin [27].

The result follows from typing the SIMEXT result established by [\[33\]](#page-26-4) in the AGM + nonprogrammable ROM where they show how to infer the aforementioned property for zkSNARKS obtained from a PIOP compiled using KZG [\[46\]](#page-27-12). By analyzing three core theorems from this paper, we identify a sequence of type-safe reductions to the underlying generically hard problem. Then applying type-safe AGM to type-safe GGM lifting theorem [\(Theorem 5\)](#page-23-1), type-safe GGM to shoup's GGM implication [\(Theorem 6\)](#page-23-2), and finally the UC lifting theorem[\(Theorem 4\)](#page-15-1), we conclude UC security for Plonk and Marlin.

**Future Work.** We find a few interesting directions for extensions. First, a natural question is to extend the formalization in this paper to capture security against adaptive adversaries and, since we are in the GGM, achieve unconditional security (as is done in [\[22,](#page-25-4) [26\]](#page-26-6)). Second, we believe our framework can be applied to the class of transparent, discrete-log based proof systems like [\[15,](#page-25-8)[60\]](#page-27-10) in order to show their UC security. Finally, a more interesting, and practically relevant question is to cover the scenario where group-based zkSNARKs can safely reuse parts of the CRS. This question is motivated by practical scenarios where a KZG-based zkSNARK might use a powers-of-tau CRS that is also used by other zkSNARKs or cryptographic primitives (e.g., vector commitments). This scenario escapes our current modeling since it would need to work with a global CRS functionality where, however, the elements cannot belong to any particular session. This extension might require redefining observability for the global-GGM functionality as well as identifying conditions in the zkSNARK design under which such a reuse of the CRS is safe.

#### <span id="page-29-0"></span>A Additional Related Works

UC-AGM. Abdalla et al. [1] present a UC-AGM framework modelling the AGM behaviour in UC. However, this model fails to capture extractable non-interactive primitives such as zkSNARKs. The fundamental issue is that the UC-AGM does not require the adversary to output an explanation alongside the group elements when providing inputs to honest parties (the verifier in this case). Whereas this explanation is needed by the simulator to extract on a verify query. This asymmetry makes UC-AGM unsuitable for zkSNARKs.

UC secure NIZKs. Several non-succinct NIZKs have been proven to be UC secure. One common approach is via compiling a proof system in the stand-alone setting, to a UC one by essentially encrypting the witness as part of the proof. SLE can be obtained by simply decrypting this ciphertext. A series of works follow variants of this recipe [2,3,8,28,50]. Others, compile  $\Sigma$  protocols using Fischlin's transform to achieve SLE [52,53]. The NIZK construction from [28] was shown to be UC secure in , [24]. Finally, the works of [25,42,44] give specific constructions of UC secure NIZKs against various notions of adaptive adversaries.

**Claim 5.** Hyb<sup>2</sup> *and* Hyb<sup>3</sup> *are indistinguishable.*

*Proof.* An additional abort condition is introduced in Hyb<sup>3</sup> compared to Hyb<sup>2</sup> , and otherwise the two hybrids are identical. This event happens when at time *t*, Z makes a group query containing a group element *g* such that *g* is not yet in the view of Z, and *g* has already appeared previously as a part of an honest algorithm execution (as a result of an intermediate group operation). Without having seen *g* before (either as a direct result of querying interfaces in *sid*, or as an output of a group query made by Z), the probability of this event boils down to Z guessing *g* correctly. Assuming that whenever an honest algorithm samples group elements, it does so with high min-entropy, this event happens with negligible probability.

## **F.1 Lifting Result**

We will now present our main lifting result, establishing the relationship between algebraic adversaries and the type-safe framework in the presence of random oracles. Most of the definitions, theorem and proofs remain largely unchanged. The big take-away of this section is that appropriate versions of random oracles can be compatible with type-safe models while preserving security guarantees. This allows to capture SNARKs in this framework.

Advantage Metrics. Let Ch denote a security game in  $\mathsf{SA}^{\mathsf{RO}_{\mathsf{ts}}}[U]$ . Ch is given the security parameter as the input and interacts with an adversarial algorithm  $\mathsf{A} = (\mathsf{A}_1, \dots, \mathsf{A}_k)$ , where each  $\mathsf{A}_i$  passes the entire state to the next. Both Ch and A have access to  $\mathsf{RO}_{\mathsf{ts}}$ . After interaction, Ch outputs a bit b. The probability of b = 1 after the interaction between Ch and A is denoted by  $\mathsf{Adv}_U^{\mathsf{Ch}}(\mathsf{A})$ . The game Ch is said to be generically hard if  $\mathsf{Adv}_U^{\mathsf{Ch}}(\mathsf{A}) \leq \mathsf{negl}(\lambda)$  for all PPT  $A \in \mathsf{TSA}^{\mathsf{RO}_{\mathsf{ts}}}[U]$ .

**Reduction.** Fix U and  $\mathsf{RO}_\mathsf{ts}$ . An algebraic reduction is a map  $R: \mathsf{AA}^{\mathsf{RO}_\mathsf{ts}}[U] \to \mathsf{AA}^{\mathsf{RO}_\mathsf{ts}}[U]$ . It is model preserving if  $\mathsf{A} \in \mathsf{TSA}^{\mathsf{RO}_\mathsf{ts}}[U]$  implies  $R[A] \in \mathsf{TSA}^{\mathsf{RO}_\mathsf{ts}}[U]$ . It is efficiency preserving if  $R[\mathsf{A}]$  is PPT algorithm whenever  $\mathsf{A}$  is.

**AGM-reduction.** Fix U and  $\mathsf{RO}_{\mathsf{ts}}$ . Fix advantage metrics  $\mathsf{A}_U^{met}$  and  $\mathsf{Adv}_U^{met_1}, \ldots, \mathsf{Adv}_U^{met_n}$ , for a constant n. We say algebraic reductions  $R_1, \ldots, R_n$  are  $\Delta_i$ -AGM reductions for these metrics if for all  $\mathsf{A} \in \mathsf{AA}_U^{\mathsf{RO}_{\mathsf{ts}}}$ , we have,

$$\mathsf{Adv}_U^{met}(\mathsf{A}) \leq \Delta_1 \cdot \mathsf{Adv}_U^{met_1}(R_1[\mathsf{A}]) + \ldots \Delta_n \cdot \mathsf{Adv}_U^{met_n}(R_n[\mathsf{A}])$$

**Theorem 8.** Fix universe U and advantage metrics  $Adv_U^{met}$ ,  $Adv_U^{met_1}$ , ...,  $Adv_U^{met_n}$ , where  $Adv_U^{met_1}$ , ...,  $Adv_U^{met_n}$  are generically hard. Let  $R_1, \ldots, R_n$  be  $\Delta_i$ -AGM reductions for these metrics that are model preserving and efficiency preserving. Then  $Adv_U^{met}$  is generically hard.

*Proof.* Let  $A \in \mathsf{TSA}^{\mathsf{RO}_{\mathsf{ts}}}[U]$  be given against security game corresponding to  $\mathsf{Adv}_U^{met}$ . Because all  $R_i$ 's are  $\Delta_i$ -AGM reductions then,

$$\mathsf{Adv}_U^{met}(\mathsf{A}) \leq \Delta_1 \cdot \mathsf{Adv}_U^{met_1}(R_1[\mathsf{A}]) + \dots \Delta_n \cdot \mathsf{Adv}_U^{met_n}(R_n[\mathsf{A}])$$

Since each  $R_i$  is also model preserving and efficiency preserving,  $R_i[A]$  is efficient and in  $\mathsf{TSA}^{\mathsf{RO}_{\mathsf{ts}}}[U]$  whenever A is. Finally, since security games corresponding to  $\mathsf{Adv}_U^{met_1}, \ldots, \mathsf{Adv}_U^{met_n}$  are generically hard, each  $\Delta_i \cdot \mathsf{Adv}_U^{met_i}(R_i[A])$  must be negligible. Since n is a constant  $\mathsf{Adv}_U^{met}(A)$  is upper-bounded by a negligible function.

The difference between strong and weak translation might seems arbitrary as one might thing it is always enought to straightline concatenate the encoding and the string. However using the weak translation might result in behavorial difference between RO and  $RO_{ts}$ . - The prefixing bit of the concatenated string prevents collision of random oracles queries, ie.  $RO(1||s) \neq RO(0||\hat{\sigma}(e))$  even if  $\sigma^{-1}(e) = s$  and simplifies the proofs. It would also be possible to do a straightline concatenation and bound the probability of collision.

We purposefully let the choice of length encoding open as it is not important for the rest of the proofs. One reasonable choice could be to use self-delimiting codes [31] such that one can even encode arbitrary length strings with only a small overhead. If algorithms are limited to polynomial time algorithms one can also use a fixed size encoding as the algorithm can only parse polynomial sized strings. The strong translation would be useful for Theorem 10 to handle canonical translation of non polynomial time algorithm.

<span id="page-38-2"></span>Remark 2. The work of [\[33\]](#page-26-4) proved that KZG polynomial commitment scheme (with the linearization trick) is simulation-extractable in the algebraic group model and non-programmable random oracle model. Starting from this, they show that using KZG for the optimized PIOP compilation results in a simulation-extractable zkSNARK in the same model. As a corollary, they obtain that Plonk and Marlin are SIMEXT in the AGM + non-programmable ROM.

**Concluding UC security.** For us to conclude UC security for Plonk and Marlin given [\[33\]](#page-26-4), all we need to do is type the result. For information theoretic reduction, notice that the algorithm consists of (1) parsing a list of bistrings and derive polynomials, (2) forward these polynomials, therefore it is trivially type safe. For computational reductions, we must do the following:

- **–** Identify the underlying assumptions in the theorem statement. For computational assumptions, show that they are generically hard.
- **–** Identify the sequence of hybrids for which indistinguishability reduces to one of the computational assumption identified before.
- **–** For these hybrids, type the reduction, and check if it is model-preserving (it will usually be efficiency preserving).

For the specific case of [\[33\]](#page-26-4), we will have to consider three main theorems – Theorem 4 (states NIZK and SIMEXT for the compiled zkSNARK), and Theorem 2 (states NIZK and SIMEXT for KZG with linearization trick, i.e., protocol *Πlin*), and Theorem 1 (states SIMEXT for KZG with multiple evaluations, i.e., protocol *Πm*-*eval*).

For Theorem 4, the computational assumptions are (1) Protocol *Πlin* is NIZK and SIMEXT (which actually refers to Theorem 2) and (2) uniqueness of KZG proofs (hard under SDH). We can list the sequence of hybrids where these assumptions are used simply by observation. The hybrids using assumption (1) almost always build the reduction following the same outline: run the main adversary internally, forward queries to oracles of the underlying assumption if needed (eg. simulation queries), run the KZG extractor. This means that as long as the KZG extractor is type-safe (which it is as it performs no group operations), the reduction is model-preserving. Also notice that at some point during the reduction, the adversary has to program the random oracle while the simulator at the end doesn't. In this case programmability is a proof artefact and can be seen as something happening internally in the adversary.

For Theorem 2, the main computational assumption is OMSDH, Affine-MDH, and that protocol *Πm*-*eval* is SIMEXT (which relates to Theorem 1). For the adaptive version of Theorem 1, the computation assumptions are: OMSDH, and affine-MDH. Finally, we remark that assumptions SDH and OMSDH are already shown to be generically hard in [\[33\]](#page-26-4), while affine-MDH reduces to discrete-logarithm. It can be observed that the reductions to these problems are model-preserving since any group element computed in the reduction is done via group operations (and not by bit manipulation). To be explicit, we concretely type one of the main reduction, i.e., to OMSDH (Theorem 1, Lemma 3 in [\[33\]](#page-26-4) ) in [Figure 6.](#page-43-0)
