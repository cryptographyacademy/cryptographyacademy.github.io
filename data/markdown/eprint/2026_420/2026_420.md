# FALCON with message recovery, a specification

#### Abstract

FALCON (Prest et al. NIST submission) is a lattice-based digital signature scheme that is intended to be standardized by NIST under the name FN-DSA. This scheme has smaller signature and public key sizes than the "primary" NIST scheme, ML-DSA, but is more complicated to implement due to floating-point requirements. For this reason, NIST has recommended the scheme be used in special situations where smaller key and signature sizes are particularly important.

There are certain situations, where one can make small modifications to FALCON which results in even shorter outputs, which is particularly interesting for FALCON's intended use cases where smaller outputs are important. In the scenario where one would like to minimize the total public key plus signature length, one could use FALCON in key recovery mode, which is a fairly straight-forward procedure specified in the FAL-CON document.

In the scenario where one would like to minimize the total signature plus message length, one could use the message recovery mode (which is a somewhat less straight-forward modification) as described in (del Pino et al. SCN 2017). The purpose of this note is to fully specify the details of this latter mode for developers wishing to implement it. The savings of using FALCON in message recovery mode is up to 226 bytes as compared to FALCON-512 and up to 434 bytes compared to FALCON-1024 for the same security levels (i.e. NIST levels 1 and 5, respectively). We also sketch how the same technique can be applied to the ring signature version of FALCON from (Gajland et al. Crypto 2024).

The main algorithmic building blocks of FALCON such as key generation and trapdoor sampling, remain exactly the same. The only algorithmic changes are in the hashing and parsing of the messages, randomness, and the hash function.

# 1 FALCON with message recovery

This document proposes a standard way in which one would use the FALCON [4] signature scheme in message recovery mode (first proposed in [1]), called FALCON-MRM. The purpose of this document is to specify the concrete specifications of this mode.

We define R<sup>f</sup> to be the polynomial ring Zq[X]/(f(X)). For both security levels of FALCON and FALCON-MRM, q = 12289. The polynomial f(X) is

A polynomial  $y \in \mathcal{R}_f$  is represented by either a 512 or a 1024 coefficient vector over  $\mathbb{Z}_q$ . If we write y=(a,b) for  $a \in \mathbb{Z}_q^{\lambda}, b \in \mathbb{Z}_q^{512-\lambda}$ , then we mean that y is interpreted as a concatenation of two vectors a and b, where a is a vector consisting of the first  $\lambda$  coefficients of y and b is a vector consisting of the other  $512-\lambda$ .

If the above b is a distribution, then a is chosen according to the distribution b. We will also write [N] to mean the set of natural numbers  $\{1, \ldots, N\}$ .

#### 1.1 Hash Functions

The FALCON-MRM signature uses two hash functions, modeled as random oracles.

• 
$$\mathcal{H}_1: \mathbb{Z}_q^{\gamma} \times \mathbb{Z}_q^{n-\lambda-\gamma} \times \{0,1\}^* \to \mathbb{Z}_q^{\lambda}$$

• 
$$\mathcal{H}_2: \mathbb{Z}_q^{\lambda} \to \mathbb{Z}_q^{n-\lambda}$$

The function  $\mathcal{H}_1$  will be used to hash the salt  $\rho \in \mathbb{Z}_q^{\gamma}$  and the message (consisting of two parts  $M_1 \in \mathbb{Z}_q^{n-\lambda-\gamma}$  and  $M_2 \in \{0,1\}^*$ ) to the domain of  $\mathcal{H}_2$ . Outputs of  $\mathcal{H}_1$  and  $\mathcal{H}_2$ , when concatenated together, will form a vector in  $\mathbb{Z}_q^n$ , whose coefficients represent a polynomial in the FALCON ring  $\mathbb{Z}_q[X]/(X^n+1)$ .

To convert vectors over  $\mathbb{Z}_q$  to bit-strings (for creating inputs to the hash functions), we simply write each element in  $\mathbb{Z}_q$  as a 14-bit string encoded as a 14-bit big-endian integer. The resulting bits are concatenated in index order and packed into bytes; if the length of the resulting bitstring is not a multiple of 8, the last byte is zero-padded in its least-significant bits. Using hash functions  $\mathcal{H}_1$  and  $\mathcal{H}_2$  to hash into vectors over  $\mathbb{Z}_q$ , we use the HashToPoint algorithm in the FALCON specification with input prefixed by an appropriate domain separation tag.

The parameters for the two security levels, corresponding to the security of FALCON-512 and FALCON-1024 are given in the below table. The values of  $\lambda$  and  $\gamma$  are chosen to be the smallest allowable by the proof of Lemma 1 (see Equation (1)).

| Security Level | n    | λ                                             | $\gamma$                                 |
|----------------|------|-----------------------------------------------|------------------------------------------|
| 128            | 512  | $19 \ ( \mathbb{Z}_q^{\lambda}  > 256)$       | $15 ( \mathbb{Z}_q^{\gamma}  > 192)$     |
| 256            | 1024 | $38 \ ( \mathbb{Z}_a^{\hat{\lambda}}  > 512)$ | $24 \ ( \mathbb{Z}_{q}^{\gamma}  > 320)$ |

The message M that is to be hashed will be split into two parts as  $M = (M_1, M_2)$ . The message  $M_1$  will be recovered from the signature, while  $M_2$  is simply sent in the clear. We will assume that  $M_2 \in \{0, 1\}^*$ , while  $M_1$  has already been converted to a vector in  $\mathbb{Z}_q^{n-\lambda-\gamma}$ . In Section 1.4, we give a method of converting a bit-string into a vector over  $\mathbb{Z}_q$  in a nearly optimal way. The message  $M_1$  will always be of the maximal allowed length  $n-\lambda-\gamma \mathbb{Z}_q$  elements, and we leave it to the application to "meaningfully" encode into, and decode from,  $M_1$ .

### 1.2 Signature Scheme

For completeness, we first present the signing and verification algorithms of the original FALCON scheme in Algorithms 1 and 2 before giving the FALCON-MRM algorithms in Algorithms 3 and 4, since we will be proving the security of FALCON-MRM via a reduction from FALCON. The key generation for both version is the same – the public key is a polynomial h and the secret key is a "trapdoor" allowing for pre-image-sampling polynomials  $s_1, s_2$  (with  $||(s_1, s_2)|| < \beta$ ) according to a discrete Gaussian distribution  $D_{\sigma}$  satisfying  $s_1 + hs_2 = t$  for arbitrary  $t \in \mathcal{R}_f$ . FALCON also uses a hash function  $\mathcal{H}: \{0, 1\}^* \to \mathbb{Z}_q^n = \mathcal{R}_f$ .

Unlike FALCON, which uses the same salt size  $|\rho| = 320$  for both security levels, mostly for simplicity, we vary the size of the salt (i.e., the value  $\gamma$ ) in addition to the output size of  $\mathcal{H}_1$  (i.e., the value  $\lambda$ ), both of which have an effect on the security, and compactness, of the schemes. The intuition for why we put  $\rho$  into the part of the message that can be recovered is in the case that if the whole message is smaller than what fits in  $M_1$ , we can use the leftover space to recover  $\rho$ . Since  $\rho$  is always part of the signature, there is no downside in always including it into the "recoverable" part.

Let  $Q_1, Q_2$  be the number of times  $\mathcal{H}_1, \mathcal{H}_2$  are queried, and  $Q_S$  be the number of signature queries. First, let's compute the probability of an abort happening when programming f. The number of times that f (and  $f^{-1}$ ) is set is  $Q_1 + Q_S$ . An abort can happen only in (2), and happens with probability upper bounded by

$$\frac{Q_1 Q_S}{|\mathbb{Z}_q^{\gamma}|} + \frac{Q_S^2}{|\mathbb{Z}_q^{\gamma}|}$$

where the first term is when a collision happens when f was originally programmed in (1) and the second term is the probability that it was programmed in (2). The time it takes to obtain an abort in the first term is  $Q_1 + Q_S$  and the time in the second term is  $Q_S$ . # 1.4 Encoding bit strings into $\mathbb{Z}_q^k$

The encoding and decoding algorithms convert bit strings to vectors over  $\mathbb{Z}_q$ , where each element in  $\mathbb{Z}_q$  is represented by a positive integer between 0 and q-1. For efficient packing, we would like to find values of x, y such that  $q^x$  is only a little bit larger than  $2^y$ , which will allow us to pack y bits into  $\mathbb{Z}_q^x$  with minimal waste. We do this encoding by interpreting the y bits as an integer and then converting it into base q. The parameters x=12, y=163 produce a rather good encoding with x=2, y=27 being only a slightly worse one. Our procedure thus first goes through consecutive blocks of 163 bits, and converts them to vectors in  $\mathbb{Z}_q^{12}$ . Once there are less than 163 bits left, it goes through blocks of 27 bits and converts them to vectors in  $\mathbb{Z}_q^2$ . The algorithms in Section 1.4.1 give pseudo-code for the encoding and decoding procedures.

**Maximum encoded length.** Table 2 shows the maximum bit-string length that can be encoded using the above method into  $\mathbb{Z}_q^{478}$  and  $\mathbb{Z}_q^{962}$ , respectively. Note that in both cases, it is just one bit less than the information-theoretic maximum. Letting B(k) denote the number of bits encodable into  $\mathbb{Z}_q^k$  using our encoding procedure (i.e. 6492 and 13067), the two values of k (478 and 962) satisfy  $B(k) = 163 \cdot \lfloor k/12 \rfloor + 27 \cdot \lfloor (k \mod 12)/2 \rfloor$ . This ensures that encoding consumes exactly k elements with no partial blocks.

**Padding.** The encoding and decoding algorithms presented here operate on bit strings of precisely B(k) bits. Padding and encoding of shorter bit-strings are left to the application developer.

#### 1.4.1 Encoding and Decoding Algorithms

Bit and integer conversions. For a bit string  $b_0b_1\cdots b_{\ell-1}$ , we define

$$\mathsf{BitsToInt}(b_0,\dots,b_{\ell-1}) = \sum_{i=0}^{\ell-1} b_i \cdot 2^{\ell-1-i}$$

(big-endian convention). The inverse  $\mathsf{IntToBits}(x,\ell)$  returns the unique  $\ell$ -bit big-endian representation  $(b_0,\ldots,b_{\ell-1})$  of x when  $0 \le x < 2^{\ell}$ .

## 1.5 A note on Ring Signatures

A ring signature scheme allows any member of an N-member group to produce a signature which can only be traced to the group, but not to the individual user.<sup>4</sup> One motivating application for this construction is the deniability feature in the Signal Protocol; Signal Messenger is interested in using Ring-FALCON-MRM in a future update to the Signal Protocol.

The work of [2] gave a simple transformation of a FALCON signature scheme to a ring-signature. The idea is that if every ring member i ∈ [N] has a FALCON public key h<sup>i</sup> and a trapdoor T<sup>i</sup> allowing him to perform trapdoor sampling (i.e. signing messages using FALCON), then these members can create a signature on a message M with salt ρ as s0, s1, . . . , s<sup>N</sup> satisfying

<sup>4</sup>Unlike a related concept known as "group signatures", a ring signature scheme does not have any central authority, and it's only the individuals themselves who know their secret keys.

{9}------------------------------------------------

$$\|(s_0, s_1, \dots, s_N)\| \le \beta \text{ and } s_0 + \sum_{j \in [N]} h_j s_j = \mathcal{H}(M, \rho).^5$$

The idea is that user i can create the s<sup>j</sup> for j ∈ [N] \ i from the discrete Gaussian distribution Dσ, and then use the FALCON sampler with his trapdoor T<sup>i</sup> to create a "FALCON" signature s0, s<sup>i</sup> satisfying

$$s_0 + h_i s_i = \mathcal{H}(M, \rho) - \sum_{j \in [N], j \neq i} h_j s_j.$$

As in the FALCON signature, one does not need to send a part of the signature – s<sup>0</sup> – in the ring version because it can be recovered from the other si . And as in the message-recovery version of FALCON we can instead choose to send s0, but recover a part of the message M, allowing for the exact same possible savings. We sketch the algorithms for the ring versions and message recovery ring versions of the FALCON signature below.
