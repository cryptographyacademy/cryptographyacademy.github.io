# CatCrypt: From Rust to Cryptographic Security in Lean

Bas Spitters

March 26, 2026

#### **Abstract**

We describe the methodology and scope of CatCrypt, a library for machine-checked cryptographic security proofs in Lean. CatCrypt provides an end-to-end pipeline from Rust reference implementations to security proofs in the computational model in Lean. The translation from Rust to Lean is done using the Hax tool.

CatCrypt covers 172 cryptographic protocols and constructions with machine-checked security theorems in the computational model. Of these, 110 have the full Rust-to-Lean pipeline. All bounds have been systematically cross-referenced against their published sources (IETF RFCs, NIST standards, and academic papers). Some proofs were ported from SSProve (Rocq), EasyCrypt, ProVerif, CryptoVerif and Squirrel; most are independent formalisations with no prior machine-checked treatment. CatCrypt also includes a verified Lean implementation of a substantial part of the hax transpiler pipeline.

This work is an experiment of what can be done by a researcher working with GenAI. Until recently, the formalization of *one* protocol required months of expert effort. In contrast, the whole of CatCrypt was developed in a period of two months. Because it was developed with AI, we develop a new methodology to increase confidence that the specifications are correct. Moreover, we will continue to audit the code in the coming months to gain even more confidence in the *specification* of the results.

We hope this work will facilitate the adoption of formal methods in the development of security-critical software. This is especially urgent due to AI's increased hacking capabilities, the explosion of AI generated software and the ongoing post-quantum transition, which requires the development of new cryptographic protocols and their secure implementation.

## **2 The [Method](#page-2-0)ology Pipeline**

<span id="page-1-0"></span>CatCrypt follows the methodology of Bhargavan et al. [Bha+25]: cryptographic protocol implementations are written in Rust, extracted to Lean via hax [Bha+24], and verified against abstract security specifications in the SSProve framework. The pipeline has the following layers:

- 1. **Standards.** We collect IETF/NIST standards, [reference](#page-12-4) implementations, KAT test vectors and scientific articles with cryptographic proofs.
- 2. **Rust reference implementation.** These are written [in safe, h](#page-12-3)ax compatible, Rust. The Hax library already contains a collection of such reference implementations. Our reference implementations are validated against RFC/NIST test vectors and differential testing against official reference implementations. We use Kani [Van+22] for bounded model checking against official implementions.
- 3. **Hax extraction.** The hax compiler extracts Rust types and functions into Lean, preserving the computational structure. We have ported a substantial par[t of the H](#page-15-2)ax transpiler to Lean and proved it correct (§3.2).
- 4. **Abstract types and dependencies.** A Dependencies typeclass abstracts cryptographic primitives from their implementations. This decouples the security proof from the concrete implementation: the same proof works for any instantiation of the dependencies. This is in line with the usual cryptographi[c m](#page-3-1)odel where one assumes that a primitive has a certain property, like being a PRF. The hax-extracted code is then shown to be an instance of these typeclasses: for example, the extracted HMAC-SHA256 implementation provides a concrete HKDFDependencies instance, and its PRF assumption supplies the *ε*hkdf term in the protocol's security bound.
- 5. **Game hopping and security theorem.** The security proof is then carried out on the lean translation of the rust reference implementation.
- 6. **Concrete primitives via libcrux.** For performance, optionally a bridge layer instantiates the abstract assumptions with optimized primitives from the libcrux verified library [KBF23]: X25519 and P-256 for DDH, HKDF-SHA256 for PRF, AES-128 for PRP, and AES-128-GCM for AEAD. Concrete protocol theorems (HPKE, EDHOC, SPQR) combine these bridges with the abstract security proofs. We have generated Hax valid reference implementations of these libcrux primitives. Validation proceeds in the three ways d[escribed](#page-14-2) in 2.

**Trusted computing base.** The Trusted Computing Base (TCB) consists of four components:

- (i) *Lean kernel.* All proofs are machine-checked by the Lean type checker. The kernel and its dependent type theory are trusted.
- (ii) *Hax extraction* (layer 1*→*2). The hax compiler translates Rust into Lean. This translation is not itself fully verified; a bug in hax could produce Lean code that does not faithfully represent the Rust source. However, we have a verified implementation for a large part of

{2}------------------------------------------------

the hax pipeline in Lean (AST, denotational semantics, a number of compilation phases with feature-erasure proofs, and translation to RawCode), which reduces the trusted surface of the extraction. Moreover, the generated Lean code is again validated against the test vectors, now written directly in Lean. Moreover, it can be inspected by hand (possibly with the help of AI).

- (iii) Cryptographic assumptions. The usual cryptographic hardness assumptions (DDH, PRF, IND-CPA, etc.).
- (iv) *Primitive correctness* (layer 6). The libcrux library consists of formally verified code (from F\* and Vale). What must be trusted is that we have correctly translated its specifications from another proof assistant to Lean. This is partially mitigated by test-vector validation (native\_decide) and differential testing against the original implementations.

## 2.1 Specification confidence

<span id="page-2-0"></span>Lean checks that our proofs are complete, but how do we know we proved the correct theorems? The protocol specifications are grounded in IETF/NIST standards and derived from reference implementations validated against official test vectors, which gives confidence in their correctness. The security proofs are game-based, and many of the games and bounds are published in the literature. We have systematically cross-referenced all 138 security bounds against their published sources (see Section 6).

To further build confidence in our security definitions, we have constructed a web of formally proved implications between alternative formulations of the same security notion. For instance, we prove that IND-CPA, IND\$ (real-or-random), and semantic security are all equivalent (up to small constant factors); that SUF-CMA implies EUF-CMA (and coincides with it for deterministic schemes); that collision resistance implies both target collision resistance and preimage resistance [RS04]; and that IND-CPA combined with INT-CTXT implies IND-CCA2 (the Bellare–Namprempre theorem [BN00]). Each definition has been carefully cross-validated (with AI) against six textbooks: Boneh–Shoup [BS20], Katz–Lindell [KL20], Rosulek [Ros21], Shoup's game-playing framework [Sh004], Goldreich's Foundations of Cryptography [Gol01; Gol04], and Cramer–Damgård–Nielsen [CDN15], as well as the EasyCrypt standard library [Bar+14] and the SSProve Rocq formalization [Has+23]. In particular, over 40 examples have been ported from The Joy of Cryptography [Ros21], covering 12 of its 16 chapters, including PRF→PRG, PRF→MAC, CBC mode, Encrypt-then-MAC, Merkle–Damgård, universal hashing, and Shamir secret sharing. A bug in any single definition would break at least one implication in the web, providing an additional layer of assurance beyond what any individual cross-check could offer.

### 3 Framework Architecture

#### 3.1 Computational Cryptography

We follow the usual monadic encoding of computational cryptography. Variants of which can be found in EasyCrypt [Bar+14], CryptHOL's spmf [BLS20], and SSProve [Has+23].

The base computation type is the *stateful sub-probabilistic monad*:

$$\mathsf{SPComp}\ \alpha := \mathsf{Heap} \to \mathsf{SDistr}\ (\alpha \times \mathsf{Heap})$$

where SDistr  $\alpha := \mathsf{PMF}(\mathsf{Option}\ \alpha)$  is the type of sub-probability distributions (probability mass functions on  $\mathsf{Option}\ \alpha$ ). The heap provides mutable state for oracle implementations; sub-distributions model both probabilistic sampling and failure. In this way, we model an imperative probabilistic language in Lean.

Game-based security is then expressed via *statistical distance*:

$$\mathsf{sdist}(f,g) \; := \; \sup_{D} \bigl| \Pr[D \circ f = 1] - \Pr[D \circ g = 1] \bigr|$$

{3}------------------------------------------------

where *D* ranges over all distinguishers. Like in SSProve, we define packages. SSProve defines a deeply embedded domain-specific language for protocols, and a Dijkstra monad-based program logic. In contrast, we work directly with denotational semantics. The SSP methodology is implemented as a reflexive tactic in Lean.

### **3.2 Verified hax pipeline**

<span id="page-3-1"></span>The hax transpiler [Bha+24] extracts Rust code into Lean, but it is itself an unverified OCaml program: a bug in hax could silently produce Lean code that does not faithfully represent the Rust source. To reduce this trusted surface, we have ported a substantial part of the hax transpiler pipeline to Lean and proved it correct.

We start our for[malisatio](#page-12-3)n with an imperative expression AST (ImpExpr), which we branch of from the hax frontend.ImpExpr models safe Rust: references, mutable assignments, loops, early returns, and the question-mark operator. A fuel-bounded big-step denotational semantics (denote) interprets these expressions in a state monad tracking four outcome types (normal value, early return, break, continue), faithfully modelling Rust's non-local control flow. Four feature-erasure compilation phases are then verified: (1) *dropReferences* eliminates borrows and derefs, exploiting the absence of aliased mutable references in safe Rust; (2) *localMutation* rewrites mutable assignments into functional let-rebinding; (3) *functionalizeLoops* converts for/while loops with break/continue into fold combinators; (4) *cfIntoMonads* translates early returns and question-mark into ControlFlow patterns. Each phase is proved to preserve the denotational semantics and to establish an inductive feature-absence predicate; the composite pipeline\_correct theorem guarantees that the fully transformed program denotes identically to the original.

After the four phases, a translation pass converts the resulting *fully functional* expressions into SSProve's typed RawCode deep embedding. A key theorem establishes that the output contains no oracle calls, ensuring compatibility with the game-based proof infrastructure. This gives a verified path from the Rust-level AST to the SSProve computation monad, reducing the hax TCB to the front-end parser and the phases not yet ported.

### 5 Related Work

Extraction pipeline. Jasmin [Alm+17a] is a framework for developing high-speed, high-assurance cryptographic software: the Jasmin language gives the programmer assembly-level control, while its Rocq-verified compiler guarantees memory safety and constant-time execution. The security proofs are carried out in EasyCrypt. The Last Yard [Has+24] closes the gap between Jasmin implementations and cryptographic security proofs by connecting them to SSProve (Rocq) via hax, establishing end-to-end verification for high-speed cryptography.

Computational proof tools. In addition to the tools for computational cryptography already mentioned, IPDL [Gan+23] provides equational reasoning for concurrent reactive pro-

{9}------------------------------------------------

tocols, and FCF [PM15] formalises game-based proofs in Rocq with a focus on foundational constructions.

**Scope comparison.** We have counted distinct protocols and constructions formalised in each framework (from GitHub repositories and the literature):

| Framework | Proof assistant | Protocols    | Sources                                      |
|-----------|-----------------|--------------|----------------------------------------------|
| EasyCrypt | standalone      | $\sim 74$    | std. lib + Formosa-Crypto + $\sim$ 30 papers |
| CryptHOL  | Isabelle        | $\sim \! 27$ | 4 AFP entries + papers                       |
| SSProve   | Rocq            | $\sim 22$    | examples / + 3 papers                        |
| FCF       | Rocq            | ~8           | examples/ + thesis                           |
| CatCrypt  | Lean            | 172          | this work                                    |

CatCrypt covers all EasyCrypt standard-library protocols and most Formosa-Crypto targets. On a small sample, CatCrypt proofs are approximately 70% shorter than EasyCrypt proofs.

Symbolic methods. A complementary line of work focuses on symbolic protocol verification. ProVerif [Bla01] provides fully automatic verification of authentication and secrecy in the Dolev-Yao model. CryptoVerif [Bla06] provides proofs in the computational model with a high degree of automation. Tamarin [Mei+13; Bas+25] supports both unbounded sessions and has been used to find practical vulnerabilities. DY\* [Bha+21] provides modular symbolic verification in  $F^*$ . Squirrel [Bae+21] implements the CCSA (Computationally Complete Symbolic Attacker) paradigm, where computational soundness holds by construction. Connecting the results of these specialised tools with game-based proofs in a general-purpose proof assistant remains an open challenge.

Autoformalization. Large language models have opened a new avenue for translating informal mathematics into machine-checked proofs. Wu et al. [Wu+22] demonstrated that LLMs can translate competition problems into Isabelle, and Jiang et al. [Jia+23] introduced the Draft-Sketch-Prove pipeline, where an LLM drafts an informal proof, translates it into a formal sketch with sorry holes, and an automated prover fills the gaps. Concurrently with our work, Urban [Urb26] autoformalized 130k lines of Munkres general topology in Mizar. He does not address the specification problem. Moreover, the goal of his project was to formalize a well-organized mathematical textbook. Whereas, our goal is focused on the verification of cryptographic software and protocols, which requires interactive development.

# 6 GenAI Methodology

<span id="page-9-0"></span>CatCrypt was developed with extensive use of GenAI. All Lean code was written by the AI, with human direction, design and review. The development proceeded in phases. First, we ported the SSProve framework from Rocq to Lean, building on Mathlib's existing probability theory and algebraic hierarchy. This required significant human guidance to establish the right design, abstractions and automation. Once the core framework was solid, we ported existing formalised proofs from SSProve (Rocq), EasyCrypt, CryptoVerif and Squirrel. These ported proofs provided enough examples for the AI to learn the patterns of CatCrypt, enabling it to autoformalise new protocols directly from IETF/NIST standards and the scientific literature. Still the whole process required a few months of intense interactive development, during which we gained many new insights, some of which will be published separately.

Our pipeline gives high confidence that the implementations of the protocols are correct. The security bounds in our proofs are checked against published bounds in the literature or against

{10}------------------------------------------------

previous formalisations, where available. The specifications may still contain errors. The author and GenAI found a number of specification errors in the early stages of development. Those we found were usually quickly fixed. The size of the library and the reuse of definitions and assumptions across many protocols provides additional confidence that the specifications are correct. GenAI was also used in the writing process.

The CatCrypt sources are in a private GitHub repository, which we plan to make public as soon as we have reorganized it in a way where we can release individual protocols.

## **8 Conclusion**

CatCrypt demonstrates that machine-checked security proofs for real-world cryptographic protocols can be developed at scale. The combination of Rust extraction via hax, game-based security proofs, and domain-specific automation enables an end-to-end pipeline from implementation to security theorem.

CatCrypt builds on decades of work by the HACS community, especially the open source verified cryptographic protocol theory, which was likely used in the LLM training data, and more directly used for porting purposes.

As argued in [PS24], proposed cryptographic (NIST/IETF) standards should be accompanied by machine-checked proofs of security, preferably in the computational model, and reference implementations should be verified against these proofs. The methodology presented here suggests that this [can b](#page-15-15)e done at scale.
